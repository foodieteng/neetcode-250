題目簡介// THE QUESTION

QUESTION

給整數陣列 nums,只要任一值出現至少兩次就回傳 true,全部相異則 false。 例 [1,2,3,1] → true(1 出現兩次)。

「有沒有重複」是雜湊集合(hash set)的招牌用途:把看過的值丟進集合,下一個要放的時候先問「已經有了嗎」。 這是整個 Arrays & Hashing 章節的核心工具 —— O(1) 均攤的「看過沒」查詢。

2

為什麼用雜湊集合// KNOCKOUT

× REJECT · O(n²)
雙層迴圈兩兩比對
對每個元素掃剩下的找相同 → O(n²)。n 到 10⁵ 時 10¹⁰ 次,直接 TLE。沒用到任何結構。
△ REJECT · O(n log n)
先排序再看相鄰
排序後相同值會靠在一起,掃一遍看有無相鄰相等 → O(n log n)。可行,但破壞原順序、又比雜湊慢。
BOTH BEATEN · ADOPT
● ADOPT · O(n)
雜湊集合一次掃描
unordered_set 記「看過的值」,每個 O(1) 均攤查詢。掃一遍、撞到重複立刻回 → O(n) 時間、O(n) 空間。用空間換時間的典範。
3

核心策略 · 先查再放// CHECK THEN INSERT

  1. 準備一個空的 unordered_set<int> seen。
  2. 掃描每個 nums[i]:先問 seen.count(nums[i]) —— 已經在裡面就 return true(找到重複)。
  3. 否則 insert 進 seen,繼續。全部掃完都沒撞到 → return false。

順序很重要:先「查」再「放」。若先放再查,每個值都會查到自己剛放的,永遠回 true。 「查」看的是之前看過的,「放」是為了之後的比對。

提前結束的價值// EARLY EXIT

找到「一個」重複就夠,不用數幾個

題目只問「有沒有」重複,不問「有幾個」。所以撞到第一個重複就能 return true —— 剩下的完全不用看。 最好情況(前兩個就重複)是 O(1);最壞(全相異或重複在最後)才 O(n)。 「存在性問題」通常能提前結束 —— 一旦找到證據就收工,是很常見的加速。

6

複雜度// COMPLEXITY

T = O(n) 均攤　　S = O(n)

每個元素一次 O(1) 均攤的查 + 插。「均攤」:雜湊偶爾要 rehash(重新配置桶),但攤到每次操作仍是 O(1)。 空間最壞 O(n)(全相異時集合裝下所有值)。想省空間可改排序法 O(n log n) 時間 / O(1) 空間 —— 時間換空間的取捨。

7

一句洞察// INSIGHT

雜湊集合 = O(1) 的「看過沒」。掃一遍,先查再放:在集合裡就是重複、立刻回 true(提前結束)。 用 O(n) 空間把 O(n²) 壓成 O(n)。
