面向要點
工具unordered_set = O(1) 均攤的「看過沒」
流程掃一遍,先 count 再 insert

應用題// APPLY

App 1. LC 219 含重複元素 II:要求兩個相同值的索引距離 ≤ k。集合要怎麼改?存什麼?
App 2. 若問的不是「有沒有重複」而是「哪個值重複 / 重複幾次」,資料結構要換嗎?
App 3. 若值域很小(例如保證 0 ≤ nums[i] < n),能不能不用雜湊、達到 O(n) 時間 O(1) 空間?
// ANSWERS
A1. 因為 count 要查的是「之前看過的值」,而 insert 是把當前值「留給之後比對」。
若先 insert 再 count:當前值 x 剛放進集合,下一行 count(x) 一定查到自己剛放的那個 → 每個元素都回 true,連全相異的陣列也誤判成有重複。
正確順序:先查(集合此刻只含「前面」的值,查到才是真重複)→ 再放(自己納入集合,供後續元素比對)。這個「先讀舊狀態、再寫新狀態」的順序,是所有「邊掃邊建表」型演算法的通用紀律 —— 和 Two Sum 的「先查補數、再存自己」一模一樣。
A2. 這題用 set 就夠 —— 只需要「這個值看過沒」的是非題,不需要附帶資訊。
什麼時候要 map:當你除了「有沒有」還要記額外資料時 —— 例如記次數(unordered_map<int,int>,像 242 Valid Anagram 數字母)、記索引(像 Two Sum 要回傳位置、219 要算距離)。
準則:只問存在性 → set;要附帶值(次數 / 索引 / 其他)→ map。用 map 解這題不會錯,但殺雞用牛刀,面試選最貼切的結構更好。
A3. 因為雜湊表的操作大多數是 O(1),但偶爾會觸發 rehash(元素太多、負載因子超標時,重新配置更大的桶陣列、把所有元素搬過去,那一次是 O(n))。把這些偶發的昂貴操作攤平到每次操作,平均仍是 O(1) —— 這叫均攤(amortized)。
最壞情況:如果所有鍵雜湊到同一個桶(hash collision 極端、或被惡意構造),查詢退化成 O(n),整體變 O(n²)。實務上標準庫的雜湊函式讓這幾乎不發生,面試講「均攤 O(1)、最壞 O(n) 因碰撞」就很完整。「均攤」是誠實描述雜湊表複雜度的關鍵字。
A4. 看重複出現在哪裡:
● 逐一 insert + 提前結束:重複越早出現越快 —— 撞到就回,可能只掃幾個(最好 O(1))。最壞(全相異或重複在最後)才 O(n)。
● 建集合比大小:unordered_set<int> s(nums.begin(), nums.end()) 一定會把整個陣列灌進集合(O(n) 且沒有提前結束),再比 size。程式碼最短,但重複很早出現時白掃了後面全部。
結論:平均 / 重複早出現 → 逐一 insert 更快;兩者最壞都是 O(n)。要效能就用提前結束版,要極簡就用建集合版 —— 這是「可讀性 vs 效能」的小取捨。
A5. 排序 + 看相鄰:sort(nums) 後,相同的值一定相鄰,掃一遍看有沒有 nums[i]==nums[i-1]。時間 O(n log n)(排序主導),額外空間 O(1)(原地排序,不算排序遞迴堆疊的 O(log n))。
代價有二:(1) 時間從 O(n) 變 O(n log n);(2) 破壞原陣列順序 —— 若後續還要用原順序,得先複製一份(那又花 O(n) 空間,失去省空間的意義)。
這正是經典的時間↔空間取捨:雜湊用 O(n) 空間換 O(n) 時間;排序用 O(n log n) 時間換 O(1) 空間。面試能講出「兩種都會、依限制選」就到位了。
App 1. 219 要「兩個相同值的索引距離 ≤ k」。改用 滑動視窗 + 集合:維護一個「最近 k 個元素」的 unordered_set。掃到 i 時:若 nums[i] 已在集合中 → 找到距離 ≤ k 的重複,回 true;否則 insert;並在 i ≥ k 時把 nums[i-k] 移出集合(讓它只保留最近 k 個)。
或用 unordered_map<值, 最後出現的索引>:每次查「這個值上次出現在哪」,若 i - 上次 ≤ k 就 true,否則更新索引。
關鍵差異:217 只需 set(存在性);219 需要「位置資訊」→ 要嘛用 map 存索引、要嘛用固定大小的視窗集合。加了「距離」約束,就得把位置也記進去。
App 2. 要換成 unordered_map<int,int>(值 → 次數)。掃一遍,freq[x]++;想知道哪個值重複,就找 freq[x] > 1 的鍵;想知道重複幾次,直接讀 freq[x]。
差別:217 只問是非(set 足矣、可提前結束);「哪個 / 幾次」是統計問題,需要掃完全部才能給出完整答案(不能提前結束,因為要數到底),而且要 map 存次數。問題從「存在」升級成「計數」,結構就從 set 升級成 map,也失去提前結束。這條升級路徑貫穿整個 Hashing 章節。
App 3. 可以,用原地標記(把陣列自己當雜湊表)。若保證 0 ≤ nums[i] < n:掃每個 x = |nums[i]|,把 nums[x] 標成負數當「看過 x」的記號;若發現 nums[x] 已經是負的 → x 之前出現過 → 重複。O(n) 時間、O(1) 額外空間(借用原陣列的正負號當標記位)。
前提很強:值域必須落在索引範圍內、且能容忍改動(或事後還原)符號。這是「值域即索引」的技巧,和 41 First Missing Positive、448 同源。一般整數(值域無界)做不到,只有「值恰好能當索引」時才成立 —— 這也是為什麼通解還是雜湊。
