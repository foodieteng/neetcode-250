垂直法讓「某個字串到底了」和「字元不一致」用同一個 return 處理 —— 兩者都代表「共同前綴到此為止」。共同前綴的長度,天生被最短字串和第一個分歧欄兩者中較小的那個卡住。

// ANSWERS
A1. 因為共同前綴必須是每一個字串的前綴 —— 當然也包含第一個字串 strs[0]。所以答案一定是 strs[0] 的某個前綴(可能是它全部、也可能更短),拿 strs[0] 當「上限候選」最自然。
只會變短的原因:每多比對一個字串,共同前綴的條件就多一層限制 —— 新的 prefix 必須同時是舊 prefix 和新字串的前綴,是兩者的「交集」。交集只會越取越小或不變,絕不可能變大。所以 prefix 長度單調非遞增,這保證了(1) 演算法會收斂(2) 一旦砍到空字串就可以確定答案是 ""、提前結束。「候選只減不增」是這類『逐步收緊交集』演算法的共同性質。
A2. str.find(prefix) 回傳 prefix 在 str 裡第一次出現的起始位置。
● 回 0:prefix 從 str 的第 0 個字元開始出現 —— 這正是「prefix 是 str 的前綴」的定義。
● 回 大於 0:prefix 出現在 str 中間(例 find("cd") 在 "abcd" 回 2),但不在開頭 → 不是前綴。
● 回 npos:prefix 根本不在 str 裡 → 更不是前綴。
所以只有 == 0 才代表前綴;!= 0(涵蓋「大於 0」和「npos」兩種)就代表「還不是前綴,得繼續砍」。用 find(...) == 0 判前綴是個簡潔慣用法,但要記得它是「首次出現位置」,不是布林 —— 中間出現也會回非零,別誤用成「有沒有出現」。
A3. 因為 std::string::find 對空字串參數有明確規範:str.find("") 回傳 0(空字串被視為在任何字串的位置 0「找得到」,只要起始位置 ≤ 長度)。
所以當 prefix 一路被砍成 "",下一次 while 判斷 str.find("") != 0 → 0 != 0 → 假,迴圈立刻停止。因此不會再進迴圈執行 substr(0, prefix.size()-1);若真的執行了,prefix.size() 是 0,0 - 1 對 size_t 會環繞成巨大值、substr 會拋 out_of_range —— 但 find("")==0 的語意剛好在越界前一步守住。這是個「函式庫的邊界語意恰好幫你收尾」的漂亮巧合,也是為什麼不用手寫 prefix.size() > 0 的額外判斷。
A4. 回 "",而且不用特判。
以 ["","abc"] 為例:prefix 起始 = strs[0] = ""(第一個就是空字串)。第一輪比對 str="":"".find("") == 0 → 不砍;if (prefix.empty()) → 真 → 直接 return ""。
若空字串不是第一個,如 ["abc",""]:prefix="abc",比到 str="" 時,"".find("abc") 找不到回 npos(!=0)→ 一路砍 "ab"→"a"→"",砍到空後 find("")==0 停,empty() 抓到 → 回 ""。
邏輯本身就涵蓋了空字串:任何字串和空字串的共同前綴必為空,而「砍到空」正是演算法對這種情況的自然結果。好的演算法讓極端情況落在一般邏輯裡,不需要一堆 if 特判 —— 這題是個好例子。
A5. 最壞複雜度一樣,都是 O(S)(S = 所有字串總字元數):最壞情況(所有字串完全相同)兩種都得把每個字元比一次。
垂直法明顯更快的情況:共同前綴很短、但字串很長。例如 ["aaaa...(1000個a)", "baaa...", ...] —— 第 0 欄就分歧(a vs b),垂直法比一欄就 return(O(字串數));而水平法可能先拿 prefix="aaa...(1000)" 去 find 第二個長字串,find 本身可能掃很長才發現不匹配、再一個個砍,慢得多。
本質差異:垂直法「一遇到分歧就全域停止」,水平法「把每個字串各自砍到滿意」。當分歧很早出現,垂直法的「早停」省下大量比較。所以垂直法通常是效能更好的選擇,水平法勝在程式碼極短、好記。面試可先寫水平法,再提一句「若要更快、尤其前綴短時,改逐欄掃描能早停」。
App 1. 用 字典樹(Trie)。把所有字串插入 Trie 後:
● 共同前綴 = 從根往下走,只要當前節點只有一個子節點、且不是任何字串的結尾,就繼續往下;一分岔(多個子節點)或碰到某字串結尾就停。走過的路徑就是最長共同前綴。
● 反覆查詢 / 分組:Trie 天生支援「以某前綴開頭的所有字串」查詢 —— 走到該前綴對應的節點,其子樹就是全部答案。10 章 Tries 就是幹這個的。
取捨:單次求 LCP 用掃描法就好(O(S)、零額外結構);但若要大量前綴查詢 / 自動補全 / 分組,建一次 Trie(O(S))、之後每次查詢 O(查詢長度),長期划算。一次性 → 掃描;反覆查 → Trie。
App 2. LCP 有結合律:LCP(a, b, c) = LCP(LCP(a, b), c) —— 一組字串的共同前綴,等於「先求前兩個的共同前綴,再和第三個求…」。因為「共同前綴」就是取交集,而交集運算滿足結合律。
所以可以分治 / 並行:把字串分兩半,各自遞迴求出 LCP(左半) 和 LCP(右半)(可平行),再求這兩個結果的 LCP 合併。分治樹深度 O(log n),每層合併總量 O(S),總計仍 O(S) 但可並行到 O(S/p + ...)。
關鍵洞察:因為 LCP 可結合,合併順序不影響結果 —— 這正是能安全分治 / 並行的數學保證。(水平掃描其實就是「從左到右依序折疊」的特例:LCP(...LCP(LCP(s0,s1),s2)...)。)
App 3. 最偷懶的做法:把每個字串反轉,求共同前綴,再把結果反轉回來 —— 因為「後綴」在反轉後就變成「前綴」。一行前處理 + 一行後處理,完全複用現成的 LCP 邏輯,不寫新程式。
若不想反轉整個字串(省那 O(S) 複製),也可以從尾端往前比對:垂直法改成比較「每個字串的倒數第 i 個字元」(s[s.size()-1-i]),遇分歧或某字串到底就停。邏輯和前綴版對稱,只是索引從尾巴數。
「反轉化歸」是處理後綴 / 對稱問題的通用技巧:能把後綴問題變前綴、把「從右」變「從左」,直接套用已有的解 —— 先想「能不能轉換成已解過的形式」,往往比從頭寫更省事、更不易錯。
