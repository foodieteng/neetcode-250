題目簡介// THE QUESTION

QUESTION

給兩字串 s、t,問 t 是不是 s 的字母異位詞(anagram)—— 用完全相同的字母、相同的次數、只是順序不同重排而成。 例 "anagram" 與 "nagaram" → true。

anagram 的定義換句話說:兩字串的「字元多重集合」相同 —— 也就是每個字元出現的次數都一樣。 順序完全不重要,只看誰出現幾次。這就直接指向「計數」這個工具。

2

核心 · 建表 ++、消耗 --// COUNT THEN CONSUME

用 s 建次數表,用 t 一個個消掉

● 先擋長度:s.size() != t.size() → 字數都不一樣,絕不可能是 anagram,直接 false。
這是免費的快速淘汰,省掉後面白算。

● 建表:掃 s,每個字元 freq[c]++ —— 記下「s 各字元要用幾個」。

● 消耗:掃 t,每個字元 freq[c]--。若某字元 t 有但 s 沒有(鍵不存在),或減到負數(t 用得比 s 多)→ 立刻 false。全部消完不出事 → true。

為什麼「減到負數」就能判偽:freq[c] 記的是「s 還剩幾個 c 可供 t 消耗」。 一旦 t 想消一個 c 卻已經沒庫存(減成 −1),代表 t 的 c 比 s 多 → 次數對不上 → 不是 anagram。

3

長度相同,只要「沒有人變負」就夠// WHY NOT CHECK ALL ZERO

先擋長度後,不用再檢查「是否全部歸零」

既然開頭已保證 s.size() == t.size(),s 的總字元數 = t 的總字元數。若消耗過程中沒有任何字元變負(每個 t 的字元都成功從庫存扣掉),那麼「扣掉的總數 = t 的長度 = s 的長度 = 建表的總數」—— 剛好把每個庫存都扣光,必然全部歸零。所以只要中途沒出現負數,結尾直接回 true,不必再掃一遍確認全零。 長度相等這個前提,幫我們省掉了結尾的檢查。

Follow-up · Unicode 怎麼辦// UNICODE

用 hash map,別用固定 26 陣列

若只有小寫英文,最快是 int cnt[26](下標 c - 'a'),O(1) 空間。但題目 follow-up 問「若含 Unicode 呢?」—— Unicode 有數萬個碼點,[26] 陣列完全不夠。

此時用 unordered_map(你的寫法)—— 不假設字元只有 26 種,任何字元當鍵都行,空間隨實際出現的相異字元成長。 代價是常數比陣列大(雜湊 vs 直接下標)。字元集有界且小 → 陣列;字元集大或未知 → hash map。你的 unordered_map 版本正是為了通用性,順帶答了 follow-up。

6

複雜度// COMPLEXITY

T = O(n)　　S = O(k)(k = 相異字元數)

掃 s 一遍 + 掃 t 一遍 = O(n)。空間是次數表大小:小寫英文 O(26)=O(1);Unicode 則 O(k)(k = 出現的相異字元數)。 對比「排序兩字串再比對」的 O(n log n) —— 計數法更快。

一句洞察// INSIGHT

anagram = 字元次數完全相同。先擋長度,再用 s 建表 ++、t 消耗 --, 出現負數就 false。長度已相等 → 沒人變負就必然全歸零 → true。Unicode 用 map 不用 26 陣列。
