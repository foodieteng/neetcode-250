重點回顧// RECAP

面向要點
定義anagram = 字元多重集合相同 = 每個字元次數一樣
先擋s.size() != t.size() → false
建表掃 s,freq[c]++
消耗掃 t,count==0 || --freq[c]<0 → false
結尾長度已相等 + 沒人變負 → 必然全零 → true
容器小寫 int[26](快);Unicode unordered_map
複雜度O(n) 時間(勝過排序 O(n log n))

觀念題// QUIZ

Q1. 為什麼開頭要先擋 s.size() != t.size()?不擋會漏判哪種情況?
Q2. 消耗階段用「一個 freq 表 ++/--」而非「建兩張表比對」有什麼好處?為什麼減到負數就能判偽?
Q3. 條件 freq.count(c)==0 || --freq[c]<0 裡,count 為什麼要放在 || 的左邊?unordered_map 的 [] 有什麼副作用?
Q4. 為什麼消完 t 後不用再檢查「freq 是否全部歸零」?這靠的是哪個前提?
Q5. follow-up:若字串含 Unicode,你的 unordered_map 版和 int[26] 版分別會怎樣?為什麼 map 是對的?

範例 Trace// 兩種偽的抓法

// 偽 A:s 沒有的字元 → count==0 抓到 s="rat" → freq={r:1, a:1, t:1} t="car": c → freq.count('c') == 0 ← s 根本沒有 c return false(第一個字元就擋掉)
// 偽 B:某字元 t 用超過 → --freq[c]<0 抓到 s="aabb" → freq={a:2, b:2} t="abbb": a → freq[a]:2→1 b → freq[b]:2→1 b → freq[b]:1→0 b → freq[b]:0→ -1 ← b 用超過 return false(第四個字元變負)
兩個守門條件各司其職:count==0 抓「t 有 s 沒有的字元」(質不對); --freq[c]<0 抓「某字元 t 比 s 多」(量不對)。有了長度守門,質、量兩關都過 = anagram。

應用題// APPLY

App 1. LC 49 字母異位詞分組:把一堆字串裡互為 anagram 的分到同一組。要怎麼給每個字串一個「anagram 指紋(key)」?
App 2. 若要判斷的是「忽略大小寫、忽略空格」的 anagram(如句子 anagram),程式要多做什麼?
App 3. 另一種常見寫法是「排序兩字串再比對」(sort(s)==sort(t))。它和計數法各自的優缺點?什麼時候排序法反而方便?
// ANSWERS
A1. 因為 anagram 的必要條件是字數相同 —— 長度都不一樣,連比都不用比。
不擋會出兩種問題:(1) 白算(長度差很多時,建表消耗都是浪費);(2) 更嚴重的是漏判:考慮 s="ab"、t="a"。建表 {a:1,b:1},消耗 t 只有一個 a(freq[a]:1→0),迴圈就結束了,b 那個沒被消掉的庫存永遠沒人檢查 → 誤判 true。
長度守門同時保證了「t 一定會把 s 的庫存全部消耗到」,這正是後面「不用檢查全零」的基礎。長度守門不只是快速淘汰,更是整個邏輯正確的前提。
A2. 一張表 ++/-- 的好處:省一半空間、而且能邊消耗邊提前結束(一撞到偽就回),不用建完兩張表再逐格比對。
為什麼減到負數就判偽:freq[c] 記的是「s 提供了幾個 c、還剩幾個可被 t 消耗」。t 每出現一個 c 就消耗一個庫存(--)。若某次消耗時庫存已經是 0,-- 讓它變 -1 —— 代表 t 的 c 比 s 多,次數對不上。「庫存透支」= 次數不匹配 = 不是 anagram。把「比較兩個次數」轉成「一邊加、一邊減、看會不會透支」,是計數比對的通用簡化。
A3. 因為 unordered_map::operator[] 有個副作用:存取不存在的鍵時,會自動插入一個值為 0 的項。所以若直接寫 --freq[c] 而 c 不在表裡,[] 會先建一個 c:0、再減成 -1。
把 count(c)==0 放左邊,靠 || 的短路:一旦 count 發現鍵不存在(==0 為真),整個條件已定為真、右邊的 --freq[c] 根本不執行 → 不會誤建鍵。若把 count 放右邊或省略,map 就會被塞進一堆值為 -1 的垃圾鍵(雖然這題結果湊巧仍對,但污染了資料結構、也拖慢速度)。順序 = 用短路避開 [] 的建鍵副作用。
A4. 靠開頭的長度守門 s.size() == t.size()。
推理:建表時 ++ 的總次數 = s.size();消耗時 -- 的總次數 = t.size()。既然兩者相等,「加進去的總量」剛好等於「扣掉的總量」。若消耗過程沒有任何字元變負(每次 -- 都合法地從正庫存扣),那麼總扣量 = 總加量 就保證了每個字元的庫存都精確地被扣回 0 —— 不多不少。所以沒人變負 ⟺ 全部歸零,結尾直接 return true,再掃一遍檢查全零是多餘的。
若沒有長度守門,「沒人變負」就不等於「全零」(可能 s 有剩庫存沒被消,如 A1 的例子),那時才需要額外檢查。長度相等這個前提,把「全零檢查」變免費了。
A5. ● unordered_map<char,int> 版:正確。它不假設字元種類,任何字元(含 Unicode 碼位)都能當鍵,空間隨實際出現的相異字元成長。這正是為什麼你的寫法能直接答 follow-up。
● int[26] 版:會壞。c - 'a' 假設字元落在 'a'..'z',Unicode 字元算出來的下標會越界或錯亂(負數或超過 25),讀寫非法記憶體。
為什麼 map 對:計數的本質是「字元 → 次數」的映射,int[26] 是「假設 key 只有 26 種」的特化,map 才是通用形式。值域有界且小 → 陣列(快、省);值域大或未知 → hash map(通用)。更嚴謹地,若是完整 Unicode 還要考慮多位元組編碼(如 UTF-8 要先解碼成碼位再計數),但核心結論不變:用 map 不用固定陣列。
App 1. 49 分組 的關鍵是給每個字串一個「anagram 指紋」—— 互為 anagram 的字串要有相同的指紋。兩種常見指紋:
(1) 排序後的字串:"eat"、"tea"、"ate" 排序都變 "aet" → 當 map<string, vector<string>> 的 key。簡單,O(L log L) 每字串。
(2) 次數簽名:把 26 個字母的次數編成一個 key(如 "a2b0c1..." 或 array<int,26>)→ O(L) 每字串,不用排序。
本質相同:兩種都是「把 anagram 的不變量(排序結果 / 次數向量)抽出來當 key」。242 判「兩個是不是 anagram」= 比較指紋是否相等;49 = 用指紋分組。會了 242 的「次數即身分」,49 只是把它當 hash key。
App 2. 在建表 / 消耗之前先正規化(normalize)每個字元:把大寫轉小寫(tolower)、跳過空格與標點(不計入次數表)。長度守門也要改成「過濾後的有效字元數相等」,不能再直接比原字串長度(因為空格數可能不同)。
做法:各自掃一遍,只把「字母」++/--,非字母略過。核心的「次數比對」邏輯完全不變 —— 只是在餵進計數器之前加一層清洗。「先正規化再比對」是所有『忽略某些差異』類題目的通用前處理(忽略大小寫、空格、順序…都在這一層處理掉)。
App 3. ● 排序法 sort(s)==sort(t):優點是極短、極好記(一行),不用想守門和負數。缺點是 O(n log n)(比計數的 O(n) 慢)、且會改動 / 複製字串。
● 計數法:O(n) 更快、能提前結束,但要處理長度守門、負數、[] 副作用等細節。
什麼時候排序法反而方便:(1) 字元集很大 / 是任意可比較物件(不只 26 字母)時,排序不用煩惱「開多大的計數表 / 用什麼 hash」;(2) 49 分組 需要一個「可當 map key」的指紋,排序後的字串天生就是好 key(次數向量要自己編碼成字串才行)。
準則:單純判兩個是否 anagram、追求速度 → 計數;要一個可比較 / 可當 key 的規範形式 → 排序。兩者都該會,依場景選。
