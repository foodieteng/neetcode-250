完整實作// O(n) TIME · O(1) EXTRA

p1929.cpp C++17 
class Solution {
public:
    vector<int> getConcatenation(vector<int>& nums) {
        int n = nums.size();          // 先凍住原長度!
        for (int i = 0; i < n; i++) {
            nums.push_back(nums[i]);     // 讀前半、寫後半
        }
        return nums;
    }
};

就地把 nums 擴成 2n 再回傳。i < n(凍住的 n)保證只跑 n 次、讀的是原本前半。

小優化 · 先 reserve(2n)// AVOID REALLOC

p1929_reserve.cpp C++17 
    vector<int> getConcatenation(vector<int>& nums) {
        int n = nums.size();
        nums.reserve(2 * n);            // 一次配足,免中途多次擴容
        for (int i = 0; i < n; i++)
            nums.push_back(nums[i]);
        return nums;
    }

不 reserve 也是 O(n)(均攤),但過程中 vector 可能擴容數次(每次搬移舊資料)。 先 reserve(2n) 一次配足容量,後續 push_back 保證不再搬移 —— 常數更小,也讓「讀 nums[i] 的參考不會失效」這件事更穩。

另解 · 開新陣列(更好讀)// FRESH ARRAY

// 不改原陣列,語意最直白 vector<int> ans(2 * n); for (int i = 0; i < n; i++) { ans[i] = nums[i]; // 前半 ans[i + n] = nums[i]; // 後半 } return ans; 直接照題目的 ans[i]、ans[i+n] 定義填,不碰「邊 push 邊看 size」的雷

這版連凍不凍 n 的問題都不存在 —— 因為預先開好 2n 大小、用下標賦值而非 push_back。 面試時這寫法最不容易出錯,也最貼近題目的形式定義。就地版省一點空間,新陣列版更清晰,兩者都對。

邊界與陷阱// PITFALLS

陷阱症狀正解
迴圈用 i < nums.size() push_back 讓 size 一直漲 → 無窮迴圈、記憶體爆掉 先 int n = nums.size(); 凍住,用 i < n
快取參考 int& r = nums[i] push_back 擴容後參考失效 → 讀到懸空值 用索引 nums[i](每次重新定址),或先 reserve
新陣列版寫 ans.push_back 已用 ans(2*n) 開好大小,再 push 會變 4n 用下標賦值 ans[i]=…,不要再 push
n=1 以為要特判 不用,迴圈跑一次,[x] → [x,x] ✓
