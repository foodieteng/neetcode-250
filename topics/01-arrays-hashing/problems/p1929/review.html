重點回顧// RECAP

面向要點
做法把 nums 複製一份接在自己後面 → 長度 2n
關鍵先 int n = nums.size(); 凍住,迴圈用 i < n
雷i < nums.size() → size 隨 push 變大 → 無窮迴圈
安全讀前半、寫後半,兩區不重疊;用索引不快取參考
優化reserve(2n) 免中途擴容
另解開 ans(2n),ans[i]=ans[i+n]=nums[i],最清楚
複雜度O(n) 時間 · O(1) 額外

Q3. 有人寫 int& r = nums[i]; nums.push_back(r); 偶爾出怪值,為什麼?正確要怎麼寫?
Q4. reserve(2n) 有沒有改變時間複雜度?它到底省了什麼?
Q5. 「就地擴 nums」和「另開 ans(2n) 下標填」兩種寫法,各自的優缺點?面試該選哪個?

範例 Trace// [1,3,2,1]

// 凍住 n=4,依序接後面 初始 [1, 3, 2, 1] n = 4 i=0 push nums[0]=1 → [1,3,2,1, 1] i=1 push nums[1]=3 → [1,3,2,1, 1,3] i=2 push nums[2]=2 → [1,3,2,1, 1,3,2] i=3 push nums[3]=1 → [1,3,2,1, 1,3,2,1] return [1,3,2,1,1,3,2,1] 驗證 ans[i]==ans[i+4]==nums[i] ✓
檢查定義:ans[0..3] = [1,3,2,1](= nums)、ans[4..7] = [1,3,2,1](= nums)。 每個 i 都滿足 ans[i] == ans[i+n] == nums[i] —— 這就是「拼接兩份」的形式化定義。

應用題// APPLY

App 1. 若要拼接 k 份(長度 k·n)而非 2 份,程式怎麼改?要注意什麼邊界?
App 2. 「複製一份接在後面」在環形陣列題(如環形子陣列最大和、下一個更大元素 II)裡是常見前處理。為什麼把陣列接兩份能把「環形」變成「線性」?
App 3. 若不用額外空間、也不能改動 nums,只要能用 ans[i] 的形式查詢任意 i ∈ [0, 2n),怎麼做?
// ANSWERS
A1. 因為 push_back 會讓 nums.size() 每次 +1。若迴圈條件是 i < nums.size(),那麼 i 每輪 +1、上界 size 也 +1 —— i 永遠追不上不斷後退的終點,迴圈不會停。vector 一路擴大,最終記憶體耗盡(bad_alloc 或 OOM)。
先 int n = nums.size(); 把長度凍成常數,迴圈就只跑固定 n 次、剛好接上原本那 n 個。「凍住 n」不是風格,是正確性 —— 它切斷了 i 和 size 的無限追逐。這也是「一邊遍歷容器、一邊往同一容器新增」時的通用原則:先固定要遍歷的範圍。
A2. 因為讀的位置 i 恆滿足 i < n —— 永遠落在原本的前 n 個;而 push_back 寫入的是第 n 位及之後。讀區 [0, n) 和寫區 [n, 2n) 完全不重疊,所以新加進去的元素索引都 ≥ n,不可能被 i < n 的迴圈讀到。
換句話說,凍住的 n 同時扮演兩個角色:迴圈上界(只跑 n 次)+ 讀寫分界線(前半唯讀、後半只寫)。這個乾淨的分界,正是「邊讀邊寫同一個 vector」在這裡安全的原因。
A3. 因為 push_back 在容量不足時會重新配置一塊更大的記憶體、把舊資料搬過去、釋放舊記憶體。此時所有指向舊記憶體的參考 / 迭代器 / 指標全部失效。int& r = nums[i]; 快取的是舊位置的參考,一旦這次 push_back 觸發搬遷,r 就變成懸空參考(dangling),讀它是 UB —— 偶爾看起來對(還沒被覆蓋)、偶爾出怪值。
正解:用索引 nums.push_back(nums[i]); —— nums[i] 是在 push_back 的參數求值時當場重新定址的… 嚴格說,參數 nums[i] 在 push_back 之前求值成一個值的拷貝,所以即使接著擴容也沒事。或先 reserve(2n) 保證全程不搬遷。教訓:別跨越可能擴容的操作快取容器內部的參考。
A4. 沒有改變時間複雜度 —— 有沒有 reserve 都是 O(n)。push_back 的擴容是均攤 O(1)(每次容量翻倍,搬移總量是等比級數 ≈ 2n),所以 n 次 push 總共仍是 O(n)。
reserve(2n) 省的是常數因子:不 reserve 時,vector 從 n 擴到 2n 途中可能搬移一兩次(把已有元素複製到新記憶體);reserve 一次配足 2n 容量,後續 push 保證零搬移。對這題(n ≤ 1000)差別微乎其微,但在大資料或效能敏感場景是實打實的優化,也順帶保證了「參考不會因擴容失效」。複雜度不變、常數變小。
A5. ● 就地擴 nums:優點是不用額外容器(直接改輸入再回傳),空間最省。缺點是要記得凍住 n、要注意「邊讀邊寫」和參考失效的細節 —— 出錯機率略高。
● 另開 ans(2n) 下標填:優點是語意最直白(直接照 ans[i]=ans[i+n]=nums[i] 的定義寫)、沒有任何邊界陷阱(預開大小、下標賦值,不碰 push/size/參考)。缺點是多用一份 O(n) 空間、且改動了輸入之外的容器。
面試建議:除非題目明確要求「就地」,優先寫新陣列版 —— 更好讀、更難錯,能清楚展示你懂題目的形式定義。若面試官追問空間優化,再補一句「也可以就地擴 nums、記得先凍住 n」。清楚 > 炫技,尤其在第一題。
App 1. 拼 k 份:外層再包一圈。就地版:int n = nums.size(); for (int t = 1; t < k; t++) for (int i = 0; i < n; i++) nums.push_back(nums[i]); —— 關鍵仍是凍住原始 n,內層永遠只讀原本前 n 個(它們的值不變、位置也在前面),外層跑 k−1 輪各接一份。新陣列版更穩:vector<int> ans(k*n); for i in [0,k*n): ans[i] = nums[i % n]; —— 用 i % n 直接對應到原陣列,一行搞定,連凍不凍都不用想。取模 i % n 是「重複拼接」的通用寫法。
App 2. 因為環形的本質是「走到尾端會繞回開頭」。把陣列接成兩份 [a0..a(n-1), a0..a(n-1)] 之後,任何一段「跨越尾端繞回開頭」的環形子區間,在雙倍陣列裡都變成一段連續的線性子區間(長度不超過 n)。於是「環形問題」就退化成「在雙倍陣列上做線性掃描 / 滑動視窗 / 單調隊列」,不用再特判繞回。
例:下一個更大元素 II 常用「掃 2n 次、索引取 i % n」;環形子陣列最大和也可用雙倍陣列 + 長度 ≤ n 的視窗。「複製接兩份」是把環攤平成線的標準前處理 —— 這題練的正是那個基本動作。
App 3. 根本不用真的建出 2n 的陣列 —— 直接用取模索引:查詢 ans[i] 就回傳 nums[i % n]。因為 ans[i] = nums[i](i < n)、ans[i+n] = nums[i],合起來正是 ans[i] = nums[i % n] 對所有 i ∈ [0, 2n) 成立。
這樣零額外空間、不改 nums,查詢 O(1)。這是「虛擬拼接」的思路:當你只需要隨機讀取拼接結果、而非真的持有它時,用 i % n 映射回原陣列就夠了 —— 和 App 2 的環形處理是同一個 i % n 技巧的兩種用途。能不建就不建,是空間優化的第一直覺。
