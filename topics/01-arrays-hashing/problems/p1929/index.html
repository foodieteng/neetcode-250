題目簡介// THE QUESTION

QUESTION

給長度 n 的陣列 nums,做出長度 2n 的 ans,滿足 ans[i] == nums[i] 且 ans[i+n] == nums[i]。 也就是把 nums 複製一份接在自己後面。例 [1,2,1] → [1,2,1,1,2,1]。

整個 NeetCode 250 的第一題,熱身用。演算法本身一行就懂:掃過原陣列,把每個元素push_back 到尾巴。 但它藏了一個新手最常踩的雷 —— 迴圈邊界。看懂這個雷,比題目本身有價值。

2

核心 · 先「凍住」長度// FREEZE n FIRST

關鍵 · 一定要先存 n,不能邊 push 邊看 size

做法:int n = nums.size(); 先凍住原長度,再 for (i = 0; i < n; i++) nums.push_back(nums[i]);。 只跑 n 次,把原本 n 個元素依序接到後面,長度變 2n。

為什麼不能寫 i < nums.size()?因為每次 push_back 都讓 nums.size() 變大 1 —— i 加 1,size 也加 1,i 永遠追不上上界 → 無窮迴圈(陣列一直長大直到記憶體爆掉)。 先凍住 n,是這題唯一真正要想的事。

3

邊讀 nums[i] 邊 push,為何安全// READ WHILE WRITING

讀的是「前半」,寫的是「後半」,兩不相干

迴圈裡 nums.push_back(nums[i]) 同時讀 nums[i] 又寫入尾端。因為 i < n,讀到的永遠是原本的前 n 個,而寫入的是第 n 位之後 —— 兩個區域不重疊,新加的元素不會被重複讀到。

一個 C++ 細節:push_back 可能觸發重新配置記憶體(擴容),讓舊的迭代器 / 參考失效。這裡安全,是因為我們用的是索引 nums[i](每次重新定址),不是快取的參考。若寫成 int& r = nums[i]; nums.push_back(r); 就可能讀到懸空參考 —— 這是個微妙但真實的坑。

複雜度// COMPLEXITY

T = O(n)　　S = O(1) 額外(輸出不計)

掃一遍、每個 push_back 均攤 O(1) → O(n)。就地改 nums,除了輸出本身沒有額外空間。 想更快可先 nums.reserve(2 * n) 一次配足,避免中途多次擴容(見下一頁)。
