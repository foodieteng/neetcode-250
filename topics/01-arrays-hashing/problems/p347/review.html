重點回顧// RECAP

關鍵一句話
三步計數 → 依頻率分桶 → 由高往低收集 k 個
桶排序前提頻率是 0..n 小整數 → 可當索引
索引即排序「索引 = 頻率」→ 桶天然按頻率有序
桶大小n + 1(頻率最高可到 n)
提早停湊滿 k 立刻停,不掃低頻桶
複雜度桶 O(n) / 堆 O(n log k) / 排序 O(n log n)

觀念題// CONCEPT Q

Q1

桶排序能達到 O(n) 的前提是什麼?這題的哪個量滿足了這個前提?

Q2

為什麼桶陣列大小要 n + 1 而不是 n?什麼輸入會讓 buckets[n] 真的被用到?

Q3

堆版為什麼用「大小 k 的最小堆」而不是「裝全部的最大堆」?兩者複雜度差在哪?

Q4

桶排序 O(n) 明明比堆 O(n log k) 快,為什麼實務上還常看到用堆?

Q5

這題和 169 多數元素、128 最長連續序列 有什麼共通的「用索引取代比較」精神?

範例 Trace// 頻率 = n 的極端例

// nums=[7,7,7,7], k=1 —— 唯一值,頻率剛好 = n n=4 ① counts = {7:4} ② buckets 大小 = 5(索引 0..4) buckets[4] = [7] ← 頻率 4 = n,用到最後一格! ③ i=4: 桶 4=[7] → result=[7] size==k=1 → 停 [7]
這個例子正是「桶大小要 n+1」的原因:當所有元素相同,某個值的頻率剛好等於 n,會落在 buckets[n]。 若桶只開 n 格(索引 0..n−1),buckets[n] 就越界。多開一格,把「頻率 n」也容納進來。

應用題// APPLY

App 1

LC 692 前 K 個高頻單字:同頻率時要按字典序排。桶排序還能用嗎?桶裡的值要怎麼處理才滿足 tie-break?

App 2

Quickselect 版:另一個 O(n) 均攤解。它和桶排序各自的優缺點?什麼時候 quickselect 更合適?

App 3

資料流 top-k:元素一個個進來,隨時要回報當前 top-k。桶排序(需知道 n)還適合嗎?該用什麼結構?

// ANSWERS
A1

桶/計數排序達到線性的前提是:要排序的鍵落在一個已知的小範圍整數區間,才能拿鍵當陣列索引、跳過兩兩比較。這題排序的鍵是頻率,而頻率一定落在 [1, n](甚至可含 0)—— 是個上界為 n 的小整數。所以能開 n+1 個桶,用頻率當索引直接歸位,無需比較。

A2

因為頻率的最大值是 n(當所有元素都相同,某個值出現 n 次)。索引要涵蓋 0..n 共 n+1 個值,所以陣列大小是 n+1。輸入 [7,7,7,7](n=4)就會用到 buckets[4]。開成 n 格(索引 0..n−1)時,buckets[n] 越界崩潰。

A3

用「大小 k 的最小堆」是為了把每次操作壓到 O(log k) 而非 O(log n)。最小堆只保留目前最高頻的 k 個,堆頂是「第 k 高的門檻」;新頻率若大於堆頂就 push 再 pop 掉最小的,堆大小恆 ≤ k。總時間 O(n log k)。若改用「裝全部 n 個的最大堆」再彈 k 次,建堆 + 操作是 O(n log n) —— 反而更慢,失去用堆的意義。要前 k 大,就維持大小 k 的最小堆,是通用套路。

A4

幾個現實理由:(1) 空間——桶排序要開 n+1 個桶,當不同值很少、n 很大時(如 10⁹ 個元素但只有 100 種),桶陣列浪費;堆只需 O(k)。(2) 串流——資料一個個來、n 未知或無限時,無法預先開桶;大小 k 最小堆能邊來邊維護。(3) k 很小——log k 幾乎是常數,堆的簡潔勝過桶的額外記憶體。桶排序在「n 明確、頻率密集」時最優,堆在「串流 / 稀疏 / 小 k」時更實用。

A5

共通精神是「當鍵是小範圍整數,用陣列索引取代比較 / 雜湊」：347 用頻率當索引(桶排序);128 用值的存在性做集合、只從段首擴展;169 / 41 用值本身當位置(投票 / 歸位)。核心都是把「排序 / 查找」問題轉成「陣列定址」問題,用 O(1) 索引換掉 O(log) 比較,達到線性。

App 1

LC 692:桶排序仍可用,但每個桶裡的單字要按字典序排序後再輸出,以滿足「同頻率按字典序」的 tie-break。做法:分桶後,從高頻桶往低掃,對每個桶內的單字 sort(字典序),依序取直到湊滿 k。或直接用自訂比較的堆(頻率為主、字典序為輔)更直觀。純桶排序失去了 tie-break 的天然順序,得在桶內補一次排序。

App 2

Quickselect:對 (值,頻率) 陣列做「第 k 大」的分割,把前 k 高頻的一堆挪到一邊,O(n) 均攤(最壞 O(n²),可隨機化避免)。優點:只需 O(不同值數) 空間,不像桶要開 n+1 格;不需頻率上界。缺點:結果無序(只保證前 k,不保證彼此排序)、最壞退化、實作較繁瑣。桶排序穩定 O(n) 且好寫;quickselect 在不同值多、想省空間、不在意順序時勝出。

App 3

資料流:桶排序需要預知 n(才能開桶)且要看完全部才能分桶,不適合串流。改用雜湊表(值→頻率)+ 大小 k 的最小堆:更新某值頻率後,維護堆內是否該值、是否擠掉堆頂。若要「近似」top-k 且記憶體有限,用 Count-Min Sketch + heap 或 Space-Saving 演算法。核心:串流下無法「事後分桶」,得增量維護一個 top-k 結構。
