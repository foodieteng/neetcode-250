應用題// APPLY

App 1. 很多題(如 Two Sum、Valid Anagram)都用到「值→索引 / 字元→次數」的映射。若那些題的 key 也有界,能不能改用直接定址陣列取代 unordered_map?各有什麼利弊?
App 2. 若要支援 containsKey(key)(明確回傳「有沒有這個 key」),直接定址版怎麼實作?和 get(key) != -1 有何差別?
App 3. 若 key 是字串(不是整數),還能直接定址嗎?雜湊函式要怎麼設計?這牽涉什麼經典技巧?
// ANSWERS
A1. 唯一差別:格子裡存什麼。705 存一個 bit(在/不在),706 存一個完整的 value(int)。連帶地,706 需要一個哨兵(-1)表示「沒有 value」,而 705 用 false 就夠。其餘(key 當索引、O(1)、零碰撞、空間=值域、桶版結構)完全一樣。
「set 是只有 key 的 map」:集合只關心「這個 key 在不在」,而映射關心「這個 key 對到什麼」。你可以把 set 看成「value 退化成一個 bit(存在=true)的 map」——所以 unordered_set<K> 和 unordered_map<K, bool> 本質同構。理解這點,學會其中一個就等於學會另一個,只差「value 欄位多大」。
A2. 因為要有一個值代表「這個 key 沒有映射」,而這個值不能和任何合法 value 撞。題目保證 value ≥ 0,所以 -1 是安全的「空」標記。
用 0 會出事:0 是合法的 value(可以 put(k, 0))。若陣列初始化為 0,那麼「從沒 put 過的 key」和「put(k, 0) 存了真實的 0」都讀到 0 —— get 無法區分「不存在」和「值就是 0」,語意崩壞。例:get(未put的key) 會錯誤回傳 0(像是有映射)。哨兵必須是「不可能的合法值」,-1 在 value ≥ 0 時滿足,0 不滿足。
A3. 靠 0 ≤ value ≤ 10⁶ 這個保證 —— value 恆非負,所以 -1 永遠不是真實 value,能安全代表「空」。
若 value 可以是任意整數(含 -1):單一哨兵失效(put(k, -1) 會和「沒映射」混淆)。解法:另開一個 vector<bool> present 專門記「這個 key 有沒有映射」,value 陣列存任意值。put 設 present[k]=true 並存值;get 先看 present[k] 再決定回值或 -1;remove 設 present[k]=false。把「存在性」和「值」拆成兩個獨立陣列,就不依賴任何哨兵、value 是什麼都不怕。這是哨兵技巧的通用退路:哨兵撞得到合法值時,就多用一個 bit 記存在性。
A4. 直接定址版的 put(k,v) 就是 mp[k]=v —— 不管 k 原本有沒有值,直接覆蓋。「插入新的」和「更新舊的」對陣列賦值來說是同一個操作,不需要區分。
桶版不是:桶裡是一串 (key,value),put 必須先掃桶找有沒有這個 key —— 找到就改那個 pair 的 value(更新)、沒找到才push 一個新 pair(插入)。若不先找就直接 push,同 key 會在桶裡出現兩份,get 可能拿到舊的、remove 也刪不乾淨。差別的根源:直接定址「一個 key 一個固定位置」,天然唯一;桶版「一個桶多個 key 排在一起」,得自己維護唯一性。
A5. 因為每格存的東西大小不同:706 的 vector<int> 每格 4 bytes(一個 int),705 的 vector<bool> 是位元壓縮特化,每格只 1 bit。10⁶ × 4 bytes = 4 MB vs 10⁶ / 8 bytes ≈ 125 KB —— 差 32 倍(4 bytes = 32 bits)。
若 value 只有 0/1:那 map 就退化成 set(value 只需要 1 bit),可以用 vector<bool> 存,回到 125 KB。更一般地,value 的位元數決定每格大小 —— value 範圍 0..10⁶ 需要 20 bits,實務上用 int(32 bit)最方便。若記憶體極度吃緊、value 範圍小,可以用更窄的型別(int16_t 等)或位元打包。直接定址的空間 = 值域 × 每格位元數,兩者都可調。
App 1. 可以,而且常常更快。若 key 有界(如字元 0..25、小整數),用固定陣列取代 unordered_map:Valid Anagram 就常用 int cnt[26] 取代 map<char,int>;計數類、值→索引類只要 key 值域小都適用。
利:陣列沒有雜湊開銷、沒有碰撞、快取友善,常數比 hash map 小很多,且 O(1) 最壞保證(不像 hash 最壞 O(n))。
弊:吃值域大小的空間(key 值域大就浪費)、且 key 必須能直接當索引(整數、有界)。
準則:key 值域小而密 → 直接定址陣列(快又省);key 稀疏 / 無界 / 非整數 → hash map。這正是 705/706 教的核心判斷,直接遷移到所有「需要映射」的題。
App 2. 直接定址版:containsKey(key) 就是 return mp[key] != -1;(哨兵版)或 return present[key];(雙陣列版)。
和 get(key) != -1 的差別:在哨兵版裡兩者等價,因為 -1 唯一代表「不存在」。但在雙陣列版(value 可為 -1)裡,兩者不同:get(key) 可能合法地回 -1(當 put(key, -1) 存過真實的 -1),此時 get != -1 會誤判成不存在,而 containsKey 查 present[key] 才正確回 true。所以有了獨立的存在性查詢後,「查有沒有」和「查值順便判斷」就分開了 —— 這也是為什麼 std::map 提供 count()/find() 而不是只靠回傳值判斷:存在性應該獨立於值來查詢。
App 3. 字串不能直接當陣列索引(它不是整數),所以純直接定址失效 —— 必須設計雜湊函式把字串映射成整數。經典做法是多項式滾動雜湊(polynomial rolling hash):把字串看成一個 base 進位的數,hash = s[0]·b^(n-1) + s[1]·b^(n-2) + … + s[n-1](對一個大質數取模防溢位),b 常取 31 或 131。
再拿這個 hash % B 決定桶,照樣 chaining(同桶用 list<pair<string,value>>,比對時比整個字串防雜湊碰撞)。這就是 unordered_map<string, V> 的原理。
牽涉的技巧:字串雜湊是字串比對(Rabin-Karp)、去重、分組(49 Group Anagrams)的共同基礎。核心思想不變:不能當索引的東西,就先用雜湊函式把它「壓成」一個整數索引 —— 直接定址是「key 本來就是好索引」的幸運特例,雜湊函式則是把任意 key 變成索引的通用橋樑。
