完整實作 · 一遍雜湊// O(n) TIME · O(n) SPACE

p1.cpp C++17 
class Solution {
public:
    vector<int> twoSum(vector<int>& nums, int target) {
        int n = nums.size();
        unordered_map<int, int> seen;      // 值 → 索引

        for (int i = 0; i < n; i++) {
            int comple = target - nums[i];       // 我需要的另一半
            if (seen.count(comple))                // 補數之前出現過?
                return {seen[comple], i};       // 回傳兩個索引

            seen[nums[i]] = i;              // 沒有 → 存自己,留給之後
        }
        return {-1, -1};                       // 題目保證有解,理論上到不了
    }
};

先算補數、先查、再存自己 —— 三步一氣呵成。存的是 值 → 索引(不是索引→值),因為找到補數後要拿它的索引來回傳。

2

關鍵 · 先查再存,避免用同一元素兩次// SAME-ELEMENT

若先存再查,自己會配到自己

考慮 target = 6、掃到 nums[i] = 3。補數 comple = 6 − 3 = 3 —— 剛好等於自己。

● 先查再存(正確):查 seen 有沒有 3。此刻自己還沒存進去,所以只有「之前真的出現過的 3」才會命中。若前面沒有 3,查不到 → 才把自己 3→i 存入。不會拿自己配自己。

● 先存再查(錯):先 seen[3]=i,下一行查 seen[3] —— 一定查到自己剛存的,回傳 {i, i},同一個元素用了兩次,違反題意。
「先查(看之前)再存(留給之後)」這個順序,同時解決了正確性(不自我配對)和一遍完成 —— 和 217 Contains Duplicate 是同一條紀律。

另解 · 兩遍雜湊(先建表再查)// TWO-PASS

// 先把全部存進表,再逐一查補數(要小心自己) unordered_map<int,int> m; for (int i = 0; i < n; i++) m[nums[i]] = i; // 第一遍:全存 for (int i = 0; i < n; i++) { int c = target - nums[i]; // 補數存在,且不是「自己這個索引」(避免用兩次) if (m.count(c) && m[c] != i) return {i, m[c]}; } return {-1,-1};

兩遍版也 O(n),但要多一個 m[c] != i 判斷防止把某元素和自己配(因為第一遍已全存)。 一遍版更簡潔 —— 靠「查的時候自己還沒存」天然避開這問題,不用額外判斷。一遍法通常首選。 (若有重複值,兩遍版的 m[nums[i]]=i 會被後者覆蓋索引,更要小心;一遍版沒這困擾。)

陷阱症狀正解
先存再查 target=6, nums=[3,3] 首個 3 就配到自己 → {0,0} 先查補數,再存自己
存索引→值 查補數時拿不到它的索引,無法回傳位置 存 值 → 索引

擔心重複值 怕 [3,3] 出錯 一遍版沒問題:第二個 3 查到第一個 3 的索引,回 {0,1} ✓

細節走讀// [3,2,4], target=6

// 先查補數,再存自己 i=0 nums[0]=3 comple=6-3=3 seen 空,count(3)=0 → 存 seen={3:0} i=1 nums[1]=2 comple=6-2=4 count(4)=0 → 存 seen={3:0, 2:1} i=2 nums[2]=4 comple=6-4=2 count(2)=1 命中! → return {seen[2]=1, 2} answer = [1, 2] nums[1]+nums[2] = 2+4 = 6 ✓
// 重複值也 OK:[3,3], target=6 i=0 nums[0]=3 comple=3 count(3)=0 → 存 seen={3:0} i=1 nums[1]=3 comple=3 count(3)=1 命中!(這是「第一個」3,索引 0) return {0, 1} —— 第二個 3 配到第一個 3,不是配自己

重複值不會出錯的原因:掃到第二個 3 時,map 裡的 3 是第一個 3 的索引(0)—— 「先查再存」保證 map 此刻只含之前的元素,所以配到的一定是另一個 3,不是自己。
