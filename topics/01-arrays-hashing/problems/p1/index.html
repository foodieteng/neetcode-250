題目簡介// THE QUESTION

QUESTION

給整數陣列 nums 和目標 target,找出兩個相加等於 target 的元素,回傳它們的索引。 保證恰有一組解,同一個元素不能用兩次。例 [2,7,11,15]、target=9 → [0,1](2+7=9)。

整個刷題界的第一題。它把一個看似要「兩兩配對」的問題,用雜湊表壓成一次掃描 —— 這個「邊掃邊查補數」的模式,是後面無數題(3Sum、4Sum、子陣列和…)的基石。

2

核心 · 別找「兩個」,找「另一半」// THE COMPLEMENT

關鍵轉念 · 固定一個,另一個就確定了

暴力法想「哪兩個加起來是 target」,要試 O(n²) 組。但換個角度: 當我站在 nums[i],我需要的「另一半」是唯一確定的 —— 就是 comple = target − nums[i]。

問題就從「找一對」變成「這個特定的值 comple,之前出現過嗎?」—— 而「某個值出現過沒、在哪個索引」正是雜湊表 O(1) 的拿手好戲。 把「配對」轉成「查表」,是這題的全部智慧。

3

為什麼用雜湊表// KNOCKOUT

× REJECT · O(n²)
雙層迴圈試每一對
對每個 i 掃剩下的找 target−nums[i] → O(n²)。沒用到「補數唯一」這個結構。
△ REJECT · 丟索引
排序 + 雙指標
排序後左右夾逼 O(n log n)。但排序打亂了原索引,這題要回傳原始索引,還得額外記錄,麻煩。
BOTH WORSE · ADOPT
● ADOPT · O(n)
補數 + 雜湊表(值→索引)
一次掃描,對每個值查「補數看過沒」,O(1) 均攤。存值→索引,找到直接回兩個索引。O(n) 時間、O(n) 空間。
4

核心策略 · 先查補數,再存自己// CHECK THEN STORE

  1. 準備 unordered_map<值, 索引> seen。
  2. 掃描 nums[i]:算 comple = target − nums[i],查 seen 有沒有 comple —— 有就回傳 {seen[comple], i}。
  3. 否則把 seen[nums[i]] = i 存起來,繼續。

「先查再存」的順序,和 217 一樣是靈魂 —— 查的是「之前看過的值」, 這保證了不會把自己當成另一半(同一元素用兩次)。先存的話,若某個值剛好是 target 的一半(如 target=6, nums[i]=3),會查到自己剛存的、誤配。

為什麼「一遍」就夠// ONE PASS

那一對的「後出現者」一定會找到「先出現者」

解答那一對元素 (a, b),總有一個先被掃到、一個後被掃到。先掃到的那個(比如 a)當時補數 b 還沒進 map,查不到 → 它把自己存起來。等掃到後者 b 時,它的補數正是 a,而 a 已經在 map 裡了 → 命中、回傳。「先存的等後來的來配」 —— 所以一遍掃描保證找得到,不用回頭。

7

複雜度// COMPLEXITY

T = O(n) 均攤　　S = O(n)

每個元素一次 O(1) 均攤的查 + 存。空間最壞 O(n)(解在最後時,前面全存進 map)。 對比暴力 O(n²) —— 用 O(n) 空間換來線性時間,正是 follow-up 要的「比 O(n²) 更好」。

8

一句洞察// INSIGHT

別找「哪兩個」,找「另一半」comple = target − nums[i] —— 它唯一確定,查表 O(1)。 先查補數(看之前的)再存自己(留給之後),一遍掃描搞定。存值→索引才能回傳位置。
