完整實作 · 投票 + 驗證// O(n) / O(1)

p229.cpp C++17 
class Solution {
public:
    vector<int> majorityElement(vector<int>& nums) {
        int c1 = 0, cnt1 = 0, c2 = 0, cnt2 = 0;

        // ── 第一趟:投票,縮出 2 個候選 ──
        for (int num : nums) {
            if (num == c1) cnt1++;              // 命中 c1
            else if (num == c2) cnt2++;         // 命中 c2(先比值)
            else if (cnt1 == 0) { c1 = num; cnt1++; }  // c1 空 → 收編
            else if (cnt2 == 0) { c2 = num; cnt2++; }  // c2 空 → 收編
            else { cnt1--; cnt2--; }          // 第三種值 → 兩邊各減 1
        }

        // ── 第二趟:驗證真實票數 ──
        cnt1 = 0, cnt2 = 0;
        for (int num : nums) {
            if (num == c1) cnt1++;
            else if (num == c2) cnt2++;
        }

        vector<int> ans;
        if (cnt1 > nums.size() / 3) ans.push_back(c1);
        if (cnt2 > nums.size() / 3) ans.push_back(c2);
        return ans;
    }
};
比對優先序不能亂:==c1 和 ==c2 要排在 cnt==0 收編之前。 否則當 cnt2=0 但 c2 的值正好等於 num 時,會誤把它當「新值」重新收編、漏掉這一票。 先比值、再看空槽,才能讓「票數歸 0 但值還在」的候選繼續累積(動畫 i=4 就是這情形)。

初始化的隱藏假設// INIT PITFALL

這份程式把 c1、c2 都初始化成 0。看似兩槽「同值」會出問題,實際上安全 —— 因為 cnt1、cnt2 都從 0 起, 第一個非 0 的 num 會先填 c1;若某個 num 剛好是 0,它命中 c1==0 就直接計數。但有個潛在細節:

// 若輸入含 0,且 0 是候選之一,兩槽初值都 0 會不會撞? c1=0, c2=0, cnt1=0, cnt2=0 num=0 → num==c1(0) → cnt1++ // 命中 c1,不會誤入 c2 // 只要「==c1」排在「==c2」前,0 一律歸 c1,兩槽不會同時宣稱是 0 安全:優先序保證 0 只被 c1 認領;更保險的寫法是把 c2 初始化成一個絕不出現的哨兵

更防禦性的寫法:c2 初始化成 c1 + 1 之類「保證 ≠ c1」的哨兵,或用 long 存一個超出值域的數。 本題因為優先序 ==c1 先判,0/0 初值可行且已驗證正確,但面試時值得口頭說明這個假設。

邊界與陷阱// PITFALLS

陷阱症狀正解
省略驗證 直接回 2 候選 → [1,2,3] 錯回 [1,2](其實沒有眾數) 第二趟真數,篩 > n/3
比對優先序放反 cnt==0 收編排在 ==c2 前 → 漏票 先 ==c1/==c2 再看空槽
減票只減一邊 第三種值只 cnt1-- → 抵銷邏輯錯,候選失準 兩邊都減:cnt1--; cnt2--;
用 >= 而非 > 「超過 n/3」是嚴格大於,>= 會多收 cnt > n/3
答案去重 若 c1==c2(理論上不會發生)可能重複 push 優先序 + 收編邏輯已保證 c1≠c2
