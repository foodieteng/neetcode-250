重點回顧// RECAP

關鍵一句話
前綴定義prefix[i][j] = 到 (i-1,j-1) 的左上矩形總和
建表容斥上 + 左 − 左上 + 自己
查詢容斥大矩形 − 上長條 − 左長條 + 左上角
加回左上角它被兩長條各減一次 = 減兩次,補回一次
0 邊界多開一圈 0 + 索引 +1 → 免邊界特判
複雜度建 O(mn)、查 O(1)、空間 O(mn)

觀念題// CONCEPT Q

Q1

查詢公式為什麼要「加回左上角」?不加會多算還是少算?用容斥解釋。

Q2

建表公式為什麼要「減去左上」?它和查詢公式的「加回左上角」是同一個道理嗎?

Q3

為什麼要把 prefix 開成 (m+1)×(n+1) 並多一圈 0?少了這圈,查貼邊矩形(r1=0)會怎樣?

Q4

「每列各存一維前綴和」也能查,單查 O(r2−r1)。它比二維前綴和差在哪?什麼情況下反而夠用?

Q5

這題和 560 一維前綴和、238 前綴積 的共同思路是什麼?二維多了什麼麻煩?

範例 Trace// 貼上邊查詢 r1=0

// 查 sumRegion(0,1,2,2):貼著上邊(r1=0),看 0 邊界如何救場 sum = prefix[3][3] − prefix[0][3] − prefix[3][1] + prefix[0][1] r1=0 → prefix[0][3] = 0 (上長條是空的) prefix[0][1] = 0 (左上角也退化成空) sum = prefix[3][3] − 0 − prefix[3][1] + 0 = 大矩形 − 左長條 (上長條和左上角都消失) 貼邊時公式自動退化成正確形式,不必特判 r1==0
當查詢矩形頂到上邊界,「要減的上長條」本來就不存在 —— 對應的 prefix[0][…] 剛好是那圈 0。 公式一字不改,邊界情況自動正確。這正是「多開一圈 0」最漂亮的回報:把邊界判斷編碼進資料結構本身。

應用題// APPLY

App 1

LC 308 可變二維(帶 update):若矩陣元素會被修改,前綴和每次更新要 O(mn) 重建太慢。該換什麼結構?(提示:二維樹狀陣列 / BIT)

App 2

子矩形和等於 target 的個數:結合本題的行前綴 + 560 的雜湊計數,如何 O(m²n) 數出和為 target 的子矩形?

App 3

最大子矩形和(2D Kadane):找和最大的子矩形。為什麼要「固定上下邊界、壓成一維再跑 Kadane」?前綴和在其中扮演什麼角色?

// ANSWERS
A1

不加會少算。查詢是「大矩形 − 上長條 − 左長條」,而上長條和左長條重疊在左上角那塊;減兩條時,左上角被減了兩次。容斥要求「重疊部分只該扣一次」,所以要加回一次把多扣的補回。這就是 |A∪B| = |A|+|B|−|A∩B| 的矩形版。

A2

是同一個道理,方向相反。建表時 prefix[i][j] = 「上矩形 prefix[i-1][j]」+「左矩形 prefix[i][j-1]」,這兩塊重疊在左上矩形 prefix[i-1][j-1],被加了兩次,所以要減一次;再補當前格。查詢是「大矩形減兩長條、加回重疊角」,建表是「兩矩形相加、減掉重疊角」—— 都是容斥處理「重疊被算兩次」。

A3

多一圈 0 是為了讓邊界項自然歸零、且避免負索引。查詢公式含 prefix[r1][…]、prefix[…][c1],當 r1=0 就是 prefix[0][…]。開了 0 邊界,這些項 = 0(空長條),公式直接正確。少了它,r1=0 時你得寫 prefix[r1-1] = prefix[-1] → 越界,或改用一堆 if (r1>0) 特判。0 邊界把邊界邏輯藏進資料結構。

A4

「每列一維前綴」單查是 O(r2−r1)(要把 r1..r2 每列的區段和加起來),不是 O(1)。當查詢矩形通常很扁(列數少)、或查詢次數不多、或想省一半預處理程式碼時,它夠用且更好寫。但本題要求嚴格 O(1)、查詢量大(10⁴),二維前綴和才達標。取捨在「查詢的列數」與「O(1) 硬要求」。

A5

共同思路:預處理前綴聚合,把區間查詢降成常數次角落運算 —— 560 前綴和、238 前綴積、304 二維前綴和,都是「花一次預處理,換每次 O(1) 查詢」。二維多的麻煩是「矩形 = 兩維區間的交集」,單靠相減不夠,重疊的左上角會被重複扣 → 必須用容斥加回。維度每升一維,容斥項數翻倍(1D 兩項相減、2D 四項加減、3D 八項)。

App 1

LC 308:元素可改時,靜態前綴和每次 update 要 O(mn) 重建,太慢。改用二維樹狀陣列(2D BIT / Fenwick):update 和區域查詢都 O(log m · log n)。它用「低位技巧」維護部分前綴,支援單點修改 + 前綴矩形查詢,再用同樣的容斥四角組出任意矩形。是「查詢快 vs 支援修改」的經典升級。

App 2

子矩形和 = target 計數:固定上下邊界一對 (top, bottom)(O(m²) 對),把這兩列之間每一行壓成一個「列和」(用行前綴 O(1) 取得)→ 問題退化成一維「和為 target 的子陣列數」,正是 560 的雜湊計數 O(n)。合計 O(m²n)。降維 + 前綴 + 雜湊 的組合拳。

App 3

最大子矩形和:枚舉上下邊界 (top,bottom),把這兩列之間每一行壓成一個「列和」得到一維陣列,對它跑 Kadane(最大子陣列和)。前綴和(或逐步累加)負責 O(1) 算出每一行在 [top,bottom] 的和,讓「壓成一維」變便宜。整體 O(m²n)。核心思想同 App 2:固定一個維度、把二維壓成一維,再套成熟的一維演算法。
