範例 Trace// 碰撞怎麼發生

// 直接定址永不碰撞(每 key 專屬格) add(5) → mp[5]=true add(774) → mp[774]=true ← 5 和 774 各有各的格子,互不影響 contains(5)=true contains(774)=true
// 桶版:5 和 774 撞同桶(B=769,h(774)=774%769=5) add(5) → 桶[5] = [5] add(774) → 桶[5] = [5, 774] ← 碰撞!兩個不同 key 進同一桶 contains(774) → 掃桶[5]: 5≠774, 774==774 ✓ → true 碰撞是雜湊的常態;直接定址用「桶數=值域」把碰撞歸零,代價是空間
直接定址其實是「桶數 = 值域、雜湊函式 = 恆等(h(k)=k)」的雜湊表 —— 每個桶最多裝一個 key,自然不撞。 真正的雜湊表桶數遠小於值域,靠雜湊函式壓縮,代價就是會撞、要 chaining。兩者是同一光譜的兩端:空間 vs 碰撞的取捨。

應用題// APPLY

App 1. LC 706 設計雜湊映射:要存 key→value 而非只有 key。直接定址版要怎麼改?「不存在」怎麼表示?
App 2. 除了 chaining(拉鏈),還有一種碰撞解法叫 open addressing(開放定址)。它怎麼運作?和 chaining 的優缺點?
App 3. 桶版若元素越加越多,某些桶會變長、退化。真實的 unordered_set 怎麼避免?這叫什麼機制?
// ANSWERS
A1. 靠題目給的 0 ≤ key ≤ 10⁶ —— key 的值域有界且不大。當 key 本身就落在一個「開得起的陣列大小」範圍內,就能直接拿 key 當陣列索引,mp[key] 存「在不在」。不需要雜湊函式把 key 壓縮到小範圍(因為它本來就在小範圍),也不會有兩個 key 搶同一格。
沒有這個界就不行:若 key 可以是任意 int(值域 ~4×10⁹),開那麼大的陣列不可能。「值域有界且可承受」是直接定址成立的唯一前提 —— 題目特意設 10⁶ 就是為了讓這條捷徑可行。
A2. 因為每個 key 有自己專屬的一格(mp[key])—— 不同的 key 對到不同的索引,不可能兩個 key 搶同一格,所以永不碰撞。而「把 key 對到索引」這件事,直接定址用的是恆等映射(索引 = key),不需要任何計算,所以不需要雜湊函式。
和真正雜湊表的關係:雜湊表是「桶數 遠小於 值域,用雜湊函式 h(key) 把大值域壓縮到小桶數」——壓縮就會讓不同 key 撞同桶(碰撞),得靠 chaining 化解。直接定址是雜湊表的極端特例:桶數 = 值域、h(k)=k、每桶最多一個。它用「不壓縮」換來「不碰撞」,代價是空間等於整個值域。
A3. 因為值域無界(如 32 位整數 ~4×10⁹ 種),開不出那麼大的陣列(光是 bool 也要幾百 MB~GB)。改用固定大小的桶陣列 + 雜湊函式:選一個可承受的桶數 B(如質數 769),用 h(key) = key % B 把任意 key 壓縮到 0..B-1。
碰撞處理:separate chaining(拉鏈) —— 因為 B 遠小於值域,不同 key 會 % B 後落在同一桶(碰撞)。每個桶掛一條 list,同桶的 key 串起來;add 先掃桶查重再加、contains 掃桶比對、remove 從桶刪。桶內元素少時平均 O(1),全撞同桶時最壞 O(n)。「大值域壓小桶數 → 必碰撞 → chaining」是所有通用雜湊表的核心。
A4. 桶數用質數:因為 key % B 的分佈品質和 B 有關。若 B 是合數(如 1000 = 8×125),而輸入 key 剛好多是某因數的倍數(如都是偶數、或都是 10 的倍數),它們會集中撞進少數幾個桶,其餘桶空著 → 退化嚴重。質數沒有小因數,能讓各種規律的 key 更均勻散開,碰撞更少。
add 先掃桶:集合不允許重複。若不先檢查就 push_back,同一個 key 加兩次,桶裡會有兩份 —— contains 仍對,但 remove 一次只刪一份(或 list::remove 刪光但語意亂),而且浪費空間、拖慢掃桶。所以 add 必須先掃桶確認不存在才加,維持「每個 key 最多一份」的集合語意。
A5. std::vector<bool> 是個特化:它位元壓縮,每個元素只佔 1 bit。所以 10⁶ 個 ≈ 10⁶ / 8 bytes ≈ 125 KB,很省。
對比:
● vector<int>(如 706 存 value):每格 4 bytes → 10⁶ × 4 = 4 MB,大 32 倍。
● vector<char/bool 非特化>:每格 1 byte → 1 MB。
● std::set:紅黑樹,每個元素含指標開銷,只存實際加入的(用量小時省),但每次操作 O(log n)、且無法「key 當索引」的 O(1)。
直接定址用 vector<bool> 的 125 KB 換全 O(1),在 key ≤ 10⁶ 且要極速時是划算的。若 key 稀疏(值域大、實際只用幾個),set/雜湊反而省空間 —— 又回到空間 vs 速度的取捨。
App 1. 706 存 key→value。直接定址版把 vector<bool> 換成 vector<int>(存 value):put(k,v) → mp[k]=v、get(k) → mp[k]、remove(k) → mp[k]=-1。
「不存在」怎麼表示:用一個哨兵值 —— 題目保證 value ≥ 0,所以用 -1 表示「這個 key 沒有映射」。初始化整個陣列為 -1,get 讀到 -1 就回「不存在」。
但要小心:哨兵值必須落在合法 value 範圍外。若 value 可能是 -1,就不能用 -1 當哨兵(會和真實值混淆)—— 得改用「另開一個 vector<bool> present 記錄存在性」,或用更大的型別/特殊值。706 因保證 value ≥ 0 才能安全用 -1,這是它和 705 唯一的實質差別(bool → int + 哨兵)。
App 2. open addressing(開放定址):碰撞時不另開 list,而是在同一個陣列裡找下一個空位放。最簡單的是線性探測(linear probing):h(key) 那格滿了,就試 h+1, h+2, …(環繞)直到找到空位;查詢時同樣沿著探測序列找,遇到空格才判定不存在。
vs chaining:
● open addressing 省指標開銷(全在一塊連續陣列,快取友善),但刪除麻煩(不能直接清空,會截斷探測鏈,要用「墓碑 tombstone」標記),且負載因子高時劇烈退化(探測鏈變長)。
● chaining 刪除簡單、負載因子容忍度高,但有指標 / list 節點開銷、快取較差。
兩者是雜湊表解碰撞的兩大流派;std::unordered_set 標準要求 chaining(可穩定迭代),而許多高效能雜湊表(如 Google 的 flat_hash)用 open addressing 換速度。
App 3. 靠動態擴容 + rehash(重新雜湊)。雜湊表監控負載因子 load factor = 元素數 / 桶數;當它超過門檻(如 1.0),就把桶數加倍(或找下一個質數)、重新配置,並把所有既有元素用新桶數 % B_new 重新分配一遍(rehash)。這樣桶數跟著元素數成長,每桶平均長度維持在常數 → 平均操作維持 O(1)。
代價:rehash 那一次是 O(n)(要搬全部元素),但因為容量每次翻倍,搬移總量是等比級數,攤到每次操作仍是均攤 O(1)(所以雜湊表複雜度總說「均攤 O(1)」,見 217)。
本題的直接定址表不需要 rehash —— 因為桶數(= 值域)一開始就固定到頂,永遠夠用、永不碰撞。這正是它「用空間換掉所有動態複雜度」的體現:沒有擴容、沒有 rehash、沒有負載因子,只是很吃記憶體。
