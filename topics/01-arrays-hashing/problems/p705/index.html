題目簡介// THE QUESTION

QUESTION

不用任何內建雜湊表,自己實作 MyHashSet:add(key) 加入、remove(key) 移除、contains(key) 查詢是否存在。 關鍵約束:0 ≤ key ≤ 10⁶。

這題考「雜湊表的原理」。但這裡有個題目特意給的捷徑:key 的範圍有界(0..10⁶)—— 當 key 本身就能當索引,連雜湊函式都不用,直接開一個布林陣列就是最簡單的「集合」。

2

核心 · key 就是索引// DIRECT ADDRESS

直接定址表 · 一個 bool 陣列搞定

開一個大小 10⁶+1 的 vector<bool> mp,mp[k] = 「k 在不在集合裡」。三個操作全是一次陣列存取:

　add(k)　　→　mp[k] = true
　remove(k)　→　mp[k] = false
　contains(k) →　return mp[k]

因為每個 key 有專屬格子,永遠不會碰撞(collision)、不需要雜湊函式。這是雜湊表的退化特例 —— 當 key 剛好能直接當索引,「雜湊」這一步就省了。

4

取捨 · 快,但很浪費空間// TRADE-OFF

空間 = key 上限,不看實際用量

直接定址表所有操作 O(1)、零碰撞,非常快。但陣列得開到 key 的最大值 10⁶ —— 即使題目最多只呼叫 10⁴ 次(頂多裝一萬個元素),還是照開一百萬格。這是用大量空間換免雜湊的極致。 能這樣做,完全靠 key ≤ 10⁶ 這個界;若 key 是任意 32 位整數,開 2³² 大小的陣列不可能。「值域有界」是直接定址成立的唯一前提。

5

key 無界時 · 雜湊函式 + 桶// REAL HASHING

面試想聽的「通用」設計

若 key 無界,不能開超大陣列 → 開一個固定大小(如質數 769 個)的桶陣列 buckets,用雜湊函式 h(key) = key % 769 決定放哪個桶。

不同 key 可能雜湊到同一個桶(碰撞)→ 每個桶掛一條鏈結串列 / vector(chaining),同桶的 key 串在一起。add 先查該桶有沒有再加、contains 掃該桶、remove 從桶裡刪。平均 O(1)(桶內元素少),最壞 O(n)(全撞同桶)。這才是 unordered_set 內部的真面目;直接定址表是它「key 能當索引」時的簡化版。

6

複雜度// COMPLEXITY

直接定址:T = O(1)／op　　S = O(值域)=10⁶

每個操作一次陣列存取 O(1)。空間固定 10⁶(不隨用量變)。雜湊 + 桶版:時間平均 O(1)／最壞 O(n),空間 O(桶數 + 元素數) —— 更省空間、可處理無界 key。

7

一句洞察// INSIGHT

key 有界 → key 直接當索引,一個 bool 陣列就是集合,O(1)、零碰撞、免雜湊。 代價是空間 = 值域。key 無界時才需要雜湊函式 + 桶(chaining)處理碰撞。
