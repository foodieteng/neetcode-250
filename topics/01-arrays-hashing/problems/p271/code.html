完整實作 · 長度前綴// len@payload

p271.cpp C++17 
class Codec {
public:
    // 編碼:每段 = 長度 + '@' + 內容
    string encode(vector<string>& strs) {
        string enc = "";
        for (auto str : strs)
            enc = enc + to_string(str.size()) + "@" + str;
        return enc;
    }

    // 解碼:讀長度 → 按長度精準切
    vector<string> decode(string s) {
        int n = s.size(), i = 0;
        vector<string> dec;

        while (i < n) {
            int j = s.find('@', i);          // 長度和內容的界
            int len = stoi(s.substr(i, j - i));  // i..j-1 是純數字
            dec.push_back(s.substr(j + 1, len)); // 往後讀剛好 len 個
            i = j + 1 + len;               // 跳過整段內容
        }
        return dec;
    }
};
解碼的迴圈不變量:每輪開頭 i 都指向「一段的長度數字」。因為長度是純數字,find('@', i) 找到的 一定是本段的分幀符,不會撞到內容裡的 '@'。 substr(j+1, len) 按長度抓內容,再 i = j+1+len 一步跳過整段 —— 內容從不參與掃描,任意字元安全。

對照:分隔符法為什麼崩// WHY DELIMITER FAILS

// 用 '#' 當分隔符 join,遇到內容含 '#' 就出事 encode(["a#b", "c"]) → "a#b#c" decode("a#b#c") 按 '#' 切 → ["a", "b", "c"] ✗ 錯! 原本是 ["a#b", "c"] // 長度前綴法: encode(["a#b", "c"]) → "3@a#b1@c" decode 讀 len=3 → "a#b"(含 # 也照抓);讀 len=1 → "c" ✓ 分隔符和資料同一層 → 撞車;長度前綴把「多長」和「內容」分層 → 不撞

分隔符法的根本問題:讓內容(可能含分隔符)去決定界線。長度前綴則是先聲明長度、再按長度切, 內容永遠只是「被數過去的 len 個位元組」,不參與任何界線判斷。跳脫法(escape)雖能救分隔符法,但規則繁瑣易錯,遠不如長度前綴乾淨。

陷阱症狀正解
用分隔符 join 內容含分隔符 → 拆錯段 長度前綴,按長度切

substr 越界 len 大於剩餘長度 → 讀爆(此題保證正確編碼故安全) 信任自產編碼;跨系統要驗長度

i 更新錯 忘了 +1(跳過 '@')或漏加 len i = j + 1 + len

細節走讀// WALKTHROUGH

// decode("3@cat3@n@p2@hi")—— 第 2 段含 '@' i=0 : find('@',0)=1 len=stoi("3")=3 substr(2,3)="cat" i=2+3=5 i=5 : find('@',5)=6 len=stoi("3")=3 substr(7,3)="n@p" i=7+3=10 ↑ 內容 "n@p" 的 '@'(index 8)被當資料,i 直接跳到 10 i=10: find('@',10)=11 len=stoi("2")=2 substr(12,2)="hi" i=12+2=14 i=14 = n → 結束 ["cat", "n@p", "hi"]
第 2 段是關鍵:find('@', 5) 從 index 5 找,回傳 6(長度後的分幀符),而非 index 8(內容裡的 '@')。 因為 index 5 起是純數字 "3",第一個 '@' 必在數字之後。抓完 "n@p" 就 i=10 跳過整段,index 8 的 '@' 從頭到尾沒被當界線看待。
