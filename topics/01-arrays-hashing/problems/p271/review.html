重點回顧// RECAP

關鍵一句話
編碼格式每段 len + '@' + payload 接起來
解碼讀長度 → substr(j+1, len) 按長度切
分幀符用非數字(如 '@'),隔開長度與內容
不變量i 永遠指向一段的長度數字開頭
任意字元按長度讀,內容不參與掃描 → 安全
vs 分隔符分隔符讓內容參與界線判斷 → 撞車
複雜度encode / decode 皆 O(L)

觀念題// CONCEPT Q

Q1

為什麼「按長度讀」能支援任意字元,而「用分隔符切」不行?本質差別在哪?

Q2

decode 用 find('@', i) 找分幀符。為什麼保證找到的是分幀符、不是內容裡的 '@'?

Q3

分幀符能不能改用數字(如 '5')?為什麼一定要用非數字?

Q4

空字串 "" 怎麼被編碼與還原?會不會出問題?

Q5

連分幀符 '@' 都可能出現在內容裡,為什麼用它當分幀符仍然安全?這和「用逗號當分隔符」有何不同?

範例 Trace// 含空字串 ["", "a@b", ""]

// 空字串 + 含 '@' 的內容,雙重考驗 encode(["", "a@b", ""]) → "0@" + "3@a@b" + "0@" = "0@3@a@b0@" decode("0@3@a@b0@"): i=0 : find('@',0)=1 len=0 substr(2,0)="" i=2 → "" i=2 : find('@',2)=3 len=3 substr(4,3)="a@b" i=7 → "a@b" i=7 : find('@',7)=8 len=0 substr(9,0)="" i=9 → "" i=9 = n → 結束 ["", "a@b", ""]
len=0 時 substr(j+1, 0) 天然回空字串,空字串不必特判。 含 '@' 的 "a@b" 靠 len=3 整段抓下,內部 '@' 全是資料。空字串和含分幀符的內容,兩個最容易崩的邊界,長度前綴都自動正確。

應用題// APPLY

App 3

巢狀結構(list of lists of strings):如果要編碼「字串陣列的陣列」,長度前綴怎麼遞迴套用?為什麼它天然支援巢狀?

// ANSWERS
A1

本質差別是「內容有沒有參與界線判斷」。分隔符法靠「掃到分隔符就切」—— 內容一旦含分隔符,就會被誤當界線。長度前綴法先聲明長度、再按長度數過去,內容只是「被數過的 len 個位元組」,完全不參與界線決策。把「結構(多長)」和「資料(是什麼)」分層,資料愛是什麼字元都不影響結構,所以任意字元安全。

A2

因為迴圈不變量:每輪開頭 i 都指向「一段的長度數字第一位」。長度是純數字(不含 '@'),所以從 i 往後第一個 '@' 必定是「數字結束、內容開始」的那個分幀符。內容裡的 '@' 位置都 ≥ j+1,在讀完長度、按 len 抓完內容後就被 i = j+1+len 一步跳過,find 根本沒機會掃到它。

A3

不能用數字。分幀符的職責是「標記長度數字的結尾」,find 靠它知道長度到哪為止。若分幀符也是數字,就和長度本身混在一起 —— "12@" 你分不清是「長度 12」還是「長度 1、分幀符 2」。所以分幀符必須是長度表示法(數字)裡不會出現的字元。'@'、'#'、任何非數字都行。

A4

"" 編成 "0@"(長度 0 + 分幀符 + 空內容)。解碼時 len=0,substr(j+1, 0) 回空字串,i = j+1+0 跳到下一段。完全不必特判空字串 —— 長度 0 就是「讀 0 個字元」,自然正確。這是長度前綴優雅的地方:空字串、含分幀符的字串,都落在同一套規則裡。

A5

關鍵是 '@' 在這裡只負責「分開長度和內容」,不負責「分開兩段」。段與段的界是靠長度切的,不是靠 '@'。所以內容裡的 '@' 不會被誤認 —— 它出現在「已經按長度框好的內容區」內,decode 根本不會對它呼叫 find。而「用逗號當分隔符」是讓逗號直接決定段界,內容裡的逗號就會冒充段界。同樣是特殊字元,職責不同:一個分「欄位」、一個分「記錄」,前者安全、後者危險。

App 1

固定寬度:每段前面用固定 4 位元組(big-endian)存長度,不需分幀符。優點:decode 更快(直接讀 4 位元組轉 int,免 find 掃描)、格式更規整、無需「非數字分幀符」的假設。缺點:短字串也要付 4 位元組長度(空間略浪費),且長度上限被寬度綁死(4 位元組 ≤ 4GB)。這正是網路協定常用的定長長度前綴。文字版的 len@ 好讀好 debug,定長版更緊湊快速。

App 2

LC 297:同樣是「把結構壓成字串、再無歧義還原」。樹用前序遍歷 + 空節點標記(如 #)自帶界線:每個節點後面接它的左右子樹,null 用哨兵標出,decode 時前序重建就知道每棵子樹在哪結束。共通思路:編碼時把「還原所需的結構資訊」一起寫進去 —— 271 寫長度、297 寫節點順序 + null 標記,都讓 decode 能唯一地切分。

App 3

巢狀:長度前綴天然遞迴。編碼「字串陣列的陣列」時,把每個內層陣列先用本題的 encode 編成一個字串,再把這些字串當成「外層的元素」再套一層長度前綴。decode 時外層按長度切出每個內層編碼串,對每串再跑一次 decode。因為長度前綴把「一個元素」框成獨立單位,元素內部是什麼(包括另一層編碼)都不影響外層界線 —— 這種「可組合性」正是長度前綴分幀的威力。
