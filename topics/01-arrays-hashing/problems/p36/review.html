重點回顧// RECAP

關鍵一句話
三規則列、行、宮各自不重複 → 三組獨立條件
資料結構row[9]、col[9]、box[9] 三組集合
宮號公式boxId = r/3*3 + c/3(整數除法,0..8)
流程每格:算宮號 → 查三組 → 無重複才三組都插
短路|| 任一集合已有即 false
bitmask27 個 int 位元運算取代集合,更省更快
複雜度固定盤 O(1);一般 O(N²)

觀念題// CONCEPT Q

Q1

推導 boxId = r/3*3 + c/3:為什麼是這個式子?它和「二維座標壓成一維」的通式什麼關係?

Q2

為什麼要「先查三組、後插入」?先插後查會發生什麼?

Q3

為什麼判斷宮不能只靠 (r,c) 座標本身,而要壓成 boxId?兩個座標差很遠的格子怎麼會同宮?

Q4

bitmask 版把集合換成 int。row[r] & bit 和 row[r] |= bit 分別對應集合的哪個操作?為什麼這裡能用位元?

Q5

本題只驗證有效性。LC 37 數獨求解需要「填空並保持有效」,會怎麼複用這裡的檢查邏輯?

範例 Trace// boxId 對照表

// 九個格子各自的 boxId(驗證公式) (r,c) r/3 c/3 boxId = r/3*3 + c/3 (0,0) 0 0 0 ← 左上宮 (2,2) 0 0 0 ← 同左上宮(和 (0,0) 同宮) (0,8) 0 2 2 ← 右上宮 (4,4) 1 1 4 ← 正中宮 (8,0) 2 0 6 ← 左下宮 (8,8) 2 2 8 ← 右下宮 九個宮各拿到唯一的 0..8
把宮想成一個 3×3 的「宮陣」:r/3 是宮的列座標、c/3 是宮的行座標。 boxId = 宮列 × 3 + 宮行 就是把這個小小 3×3 陣列逐列攤平成 0..8 —— 和把二維陣列存成一維陣列的索引公式一模一樣。

應用題// APPLY

App 1

一般化 N²×N² 數獨(宮是 N×N):boxId 公式怎麼改?這裡的 3 對應什麼?

App 2

單一雜湊集合版:能不能只用一個 set,把 row/col/box 的資訊編碼進 key(如 "r3=5"、"c5=5"、"b0=5")?這樣寫的優缺點?

App 3

LC 37 Sudoku Solver:要真的把數獨填滿。回溯時每填一個候選數都要檢查有效性 —— 用本題的三組集合當「即時可用性」結構,填/退時如何 O(1) 更新?

// ANSWERS
A1

9×9 盤切成 3×3 個宮,宮本身排成 3 列 3 行。r/3(整數除)把行 0–8 壓成宮的列座標 0–2;c/3 把列 0–8 壓成宮的行座標 0–2。要把二維宮座標 (宮列, 宮行) 攤平成一維編號,就用通式 宮列 × 每列宮數 + 宮行 = r/3 × 3 + c/3。這和「把 a[i][j] 存進一維陣列時算 i × 寬 + j」是同一個公式。

A2

必須先查後插。若先把 digit 插進三組再查,查詢一定會查到剛剛插入的自己,於是每個格子都被誤判成「重複」,整個盤永遠回 false。順序是「這個數字之前出現過嗎?」—— 所以要在插入當前數字之前查。

A3

因為「同宮」是由 (r/3, c/3) 決定,而不是 (r,c) 相近。(0,0) 和 (2,2) 座標差 2,但 r/3、c/3 都是 0 → 同屬左上宮。若只看 (r,c) 無法把它們歸到一起。boxId 把「屬於同一宮」的所有格子映到同一個編號,那個宮的集合才能匯集整宮數字、抓到跨格重複。

A4

row[r] & bit = 集合的「contains」(第 d 位是否為 1);row[r] |= bit = 集合的「insert」(把第 d 位設為 1)。能用位元是因為值域小且固定:數字只有 1–9,用一個 int 的 9 個低位就能表示「哪些數字在集合裡」。集合退化成一個整數,查/插各一條位元指令,零雜湊、零配置。

A5

LC 37 用回溯填空:在空格嘗試每個候選數字 1–9,只有「該數字不在 row[r]、col[c]、box[boxId]」時才合法放入,遞迴下一格;回溯時再移除。這裡的三組集合(或 bitmask)正是判斷候選是否合法的 O(1) 結構 —— 36 驗證的邏輯直接變成 37 的「剪枝條件」。學會 36 就打好了 37 的地基。

App 1

宮是 N×N、整盤 N²×N² 時,boxId = (r / N) * N + (c / N)。原本的 3 就是宮邊長 N(標準數獨 N=3)。集合數量從 9 變成 N² 組。公式的骨架不變:用宮邊長做整除得宮座標,再攤平。

App 2

可以。把三種歸屬編碼進不同前綴的字串 key,全丟進一個 unordered_set<string>:插入 "r"+r+d、"c"+c+d、"b"+boxId+d,插入前檢查是否已存在。優點:只用一個容器、統一邏輯;缺點:每格要組三個字串,字串雜湊開銷比三個小集合大、也吃更多記憶體。可讀性 vs 效能的取捨,面試講清楚即可;實務上三組小集合或 bitmask 更快。

App 3

LC 37:維持三組集合(或 bitmask)當「即時可用性」狀態。填格 (r,c)=d 時:row[r].insert(d)、col[c].insert(d)、box[boxId].insert(d),全 O(1);回溯移除就 erase(bitmask 版用 ^= bit 清位),也 O(1)。嘗試候選前先用 count / & bit 檢查衝突 → O(1) 剪枝。整個回溯的「合法性檢查 + 狀態更新」都建立在 36 這套三組結構上,只是多了填/退的對稱操作。
