重點回顧// RECAP

關鍵一句話
資料結構unordered_set:去重 + O(1) 查存在
起點法只從「x−1 不在集合」的最左端往右數
守衛if (count(s−1)) continue; —— O(n) 的靈魂
攤還論證每數被檢視一次 + 被某起點走一次 = O(n)

觀念題// CONCEPT Q

Q1

為什麼「for 裡有 while」卻是 O(n) 而非 O(n²)?攤還論證的核心是什麼?

Q2

拿掉 if (count(s−1)) continue; 這行,程式還對嗎?哪種輸入會讓它退化成 O(n²)?

Q3

為什麼判斷起點用「x−1 不在集合」而不是「x+1 不在集合」?反過來(從右端往左數)可以嗎?

Q4

為什麼遍歷 seen 而不是原 nums?對重複值多的輸入差在哪?

Q5

題目沒要求 O(n) 的話,排序法 O(n log n) 有什麼優點?什麼時候反而更好?

範例 Trace// 攤還:每數只被走一次

// seen={1,2,3,4,6,7,8},標出「誰在 while 裡走過誰」 起點 1 → while 走過 1,2,3,4 (curL=4) 起點 6 → while 走過 6,7,8 (curL=3) 非起點 2,3,4 → continue(不進 while) 非起點 7,8 → continue(不進 while) while 總步數 = 4 + 3 = 7 = 集合大小 外層 for 檢視 = 7 次 合計 ≈ 2×7 → O(n) 最長 = 4
把每個數塗成「它屬於哪段」:1,2,3,4 屬第一段、6,7,8 屬第二段。 每段的 while 只由該段起點跑一次,走過該段所有成員。所以 while 全部步數加起來 = 每段長度和 = n。 這就是為何巢狀迴圈的總工作量是線性 —— 攤還到每個數頭上只有常數次。

應用題// APPLY

App 1

回傳序列本身:不只長度,還要回傳最長那段的所有數。怎麼在起點法上加最少的東西?

App 2

Union-Find 版:也能用並查集把「相鄰數」union 起來,最後找最大連通塊。這比集合起點法好還是差?複雜度和常數各如何?

App 3

LC 298 二元樹最長連續序列:值連續但要沿樹的父子路徑。這裡「連續」的鄰接關係從「數值 ±1」變成「樹邊 + 值 +1」,DFS 怎麼帶狀態?和本題共通的抽象是什麼?

// ANSWERS
A1

核心是攤還:守衛讓只有「段的最左端」進入 while,而每個數屬於恰好一段,只會被那段起點在 while 裡走過一次。所以所有 while 的步數加總 = 各段長度和 = n,加上外層 n 次檢視,共 O(n)。「單一 while 可能很長」不代表總量平方 —— 要看全程加總。

A2

還是對的(答案正確),但會慢。輸入 [1,2,3,…,n]:少了守衛,每個數都會往右數整段 —— 1 數 n 個、2 數 n−1 個…總和 O(n²)。守衛把「非起點也往右數」的重複工作剪掉,是正確性以外、專為複雜度而存在的一行。

A3

兩者對稱,都可行 —— 重點是「只從段的其中一端起算,保證每段只數一次」。用 x−1 不在 → x 是最左端、往右數;也可以用 x+1 不在 → x 是最右端、往左數。只要固定挑一端當起點即可,不能兩端都當起點(那會數兩次、破壞攤還)。習慣上用左端 + 往右,讀起來自然。

A4

遍歷 seen 讓每個 distinct 值只被檢視一次。若遍歷 nums,重複值(如三個 2)會被各檢視一次;雖然它們都不是起點、會被 continue 擋掉不影響正確性,但「檢視次數」變成 O(nums 長度) 而非 O(distinct 數),讓攤還論證不再乾淨(極端重複時多做無用功)。遍歷 set 最穩。

A5

排序法 O(n log n) 的優點:空間可到 O(1)(就地排序,不需額外雜湊表),而且沒有雜湊碰撞的最壞退化風險(unordered_set 理論最壞 O(n) 單次查詢)。當記憶體吃緊、或要穩定最壞複雜度保證、或資料本來就接近有序時,排序法反而更實在。集合法勝在達到 O(n) 平均時間。

App 1

在更新 maxL 時,順便記下該段的起點值和長度(bestStart、bestLen)。最後從 bestStart 連續輸出 bestLen 個數即可重建序列,O(bestLen) 額外、O(1) 額外空間。不必存下每段,只留最佳的起點+長度。

App 2

Union-Find:把每個數和 x+1(若存在)union,最後統計最大集合大小。正確,但常數更大(路徑壓縮 + 按秩合併是 O(α(n)) 近常數,實務仍比純雜湊查慢),且要多一個 value→id 的映射。集合起點法更簡潔、常數更小,是本題首選;Union-Find 是「連通分量」思維的通用武器,殺雞用牛刀但概念可遷移。

App 3

LC 298:鄰接從「值差 1」改為「父子邊 且 child = parent+1」。DFS 回傳「以當前節點為起點、向下的最長連續長度」:若某子節點值 = 當前值+1,則 1 + dfs(child),否則該分支重置為 1;全程更新全域最大。共通抽象:定義好「連續的鄰接關係」,再找這個關係下的最長鏈 —— 128 的鄰接是 ±1,298 的鄰接是樹邊 +1。換了圖,套路不變。
