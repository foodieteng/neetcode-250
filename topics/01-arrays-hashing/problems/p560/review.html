重點回顧// RECAP

關鍵一句話
核心恆等式sum(i..j) = prefix[j] − prefix[i−1],要 = k ⇔ prefix[i−1] = prefix[j] − k
雜湊表存什麼前綴和 → 出現次數(不是存在性)
種子freq[0]=1,空前綴,救「從頭開始」的子陣列
順序先查 cnt += freq[prefix−k] 再記 freq[prefix]++
加多少加 freq[prefix−k] 這個「次數」,不是加 1
為何不用滑窗有負數 → 和非單調 → 視窗收縮邏輯失效
複雜度O(n) 時間、O(n) 空間

範例 Trace// [1,2,3], k=3

// 種子救「從頭開始」的子陣列 [1,2] freq={0:1} prefix=0 cnt=0 num=1: prefix=1 need=1-3=-2 freq[-2]?無 cnt+=0=0 freq={0:1,1:1} num=2: prefix=3 need=3-3=0 freq[0]=1 cnt+=1=1 freq={0:1,1:1,3:1} → [1,2] num=3: prefix=6 need=6-3=3 freq[3]=1 cnt+=1=2 freq={...,3:1,6:1} → [3] 2
[1,2] 之所以被算到,全靠 need=0 命中種子的 freq[0]=1 —— 那個 0 代表「第一個元素之前的空前綴」。 沒有種子,這條「頂到左邊界」的子陣列就永遠配不到對,答案會少 1。

應用題// APPLY

App 1

LC 974 Subarray Sums Divisible by K:數「和能被 k 整除」的子陣列。前綴和的雜湊 key 該存什麼?負數取模要怎麼修正?

App 2

LC 523 Continuous Subarray Sum:問是否存在長度 ≥ 2、和為 k 倍數的子陣列。雜湊表存「餘數 → 最早出現的 index」而非次數,為什麼?

App 3

LC 930 Binary Subarrays With Sum:0/1 陣列數和為 goal 的子陣列。除了前綴和雜湊,還能用「恰好 = goal」= 「≤ goal」−「≤ goal−1」的滑窗兩次差分,為什麼這裡滑窗又能用了?

// ANSWERS
A1

滑動視窗要求「擴張右界 → 視窗和單調變化」,才能靠「和太大就收左界」來搜尋。有負數時,右移可能讓和變小,於是「和超過 k 就收縮」不再成立 —— 可能收過頭、也可能該收不收。前綴和 + 雜湊完全不依賴單調性,對負數天然成立。

A2

freq[0]=1 代表「還沒加任何元素時,前綴和 = 0 已出現一次」。它讓「從 index 0 開始」的子陣列能配對:當某個 prefix[j] = k,need = k−k = 0 正好命中種子。拿掉它,[1,2,3],k=3 會把 [1,2](prefix=3, need=0)漏掉,答案變 1(只剩 [3] 靠 need=3 配到)。

A3

同一個前綴和值可能出現多次(有負數或 0 時),每一次都是一個能配對的左端點。[1,-1,1,-1],k=0:走到最後 prefix=0,而 0 之前出現過 2 次 → 一次配出 2 條子陣列。若寫 cnt+=1 只會得 3,漏掉一條。必須加「那個前綴和累積的次數」。

A4

對調成「先 freq[prefix]++ 再查」,會把當前這一步的自己也算成一個可配對的左端點 → 多算「長度 0 的空子陣列」。k=0 最致命:每個位置 need=prefix 都會命中剛登記的自己,答案暴增 n。順序必須是先查後記。

A5

LC 325 求最大長度:雜湊表改存「前綴和 → 最早出現的 index」(只記第一次,越早 index 越能拉長)。掃到 j 時查 prefix−k 是否出現過,若有,用 j − firstIndex[prefix−k] 更新答案最大值。因為要「最長」,所以每個前綴和只保留最早的位置,且不覆蓋。這是「計數」→「記位置」的變形。

App 1

LC 974:key 存前綴和對 k 的餘數。兩個前綴和同餘 ⇔ 它們的差能被 k 整除。C++ 的 % 對負數會給負餘數,要修正成 ((prefix % k) + k) % k 落到 [0,k)。一樣種子 freq[0]=1、cnt += freq[r] 加次數。

App 2

LC 523:問「存不存在」且要長度 ≥ 2,所以不需次數、要位置。存「餘數 → 最早 index」;掃到 j 時若同餘數出現過且 j − firstIndex ≥ 2 就回 true。存最早 index 才能讓長度盡量夠 2;若存最新 index 會漏掉本可成立的較長區間。

App 3

LC 930:0/1 陣列所有元素 ≥ 0,前綴和單調不減 —— 單調性回來了,所以滑窗又能用。用「和 ≤ goal 的子陣列數」減「和 ≤ goal−1 的」= 恰好 = goal 的數(atMost(goal) − atMost(goal−1))。當然前綴和 + 雜湊也照樣可解,兩條路都行。關鍵:非負 → 單調 → 滑窗可用,這正是 560 有負數時不能用滑窗的反面。
