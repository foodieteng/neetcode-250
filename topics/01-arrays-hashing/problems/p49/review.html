關鍵一句話
核心等價anagram ⇔ 排序後字串相同
標準鍵sorted(w)(或 26 字母計數)當雜湊 key

觀念題// CONCEPT Q

Q1

為什麼「排序後字串」能唯一代表一個 anagram 群?兩個非 anagram 的字有沒有可能排序後相同?

Q2

計數鍵版為什麼一定要在數字間加分隔符 '#'?不加會撞在哪?

Q3

排序鍵 O(N·L log L) vs 計數鍵 O(N·L),什麼時候差距才明顯?本題 L ≤ 100 值得為此改用計數鍵嗎?

Q4

若字元不只小寫英文,而是任意 Unicode,計數鍵的 26 陣列還能用嗎?哪種鍵更通用?

Q5

這題和 242 驗證 anagram 的關係是什麼?兩者共用的核心觀念是哪一個?

範例 Trace// 計數鍵 vs 排序鍵

// "eat" 兩種鍵的算法 排序鍵: "eat" → sort → "aet" 計數鍵: e=1,a=1,t=1,其餘 0 key = "#1#0#0#0#1#0...#1...#0" (a 位 1、e 位 1、t 位 1) "tea"、"ate" 算出的兩種鍵都和 "eat" 相同 → 同桶 "tan" 的 t,a,n 計數不同 → 不同桶 兩種鍵都把 {eat,tea,ate} 綁在一起、把 tan 分開
不管用排序還是計數,本質都是「把字正規化成一個與字母順序無關的指紋」。 排序是「重排字母」,計數是「數字母」—— 殊途同歸,都讓 anagram 塌縮成同一個 key。

應用題// APPLY

App 1

質數乘積鍵:給每個字母一個質數(a=2,b=3,c=5…),把字的字母對應質數相乘當 key。為什麼這也是合法的 anagram 指紋?它有什麼隱患?

App 2

LC 249 Group Shifted Strings:把「位移後相同」的字串分組(如 abc、bcd)。指紋該怎麼設計?(提示:相鄰字母差的序列)

App 3

找出現次數最多的 anagram 群:不只要分組,還要回傳最大的那組。在本解法上加什麼最省?

// ANSWERS
A1

因為排序只重排字母、不改變字母的多重集合(哪些字母各幾個)。兩個字是 anagram 的定義就是「字母多重集合相同」,而排序把同一個多重集合映到唯一的字串(字母遞增排列)。所以 anagram ⇔ 排序後相同,是雙向的:不可能有兩個非 anagram 的字排序後相同(那會迫使它們字母集合相同,矛盾)。

A2

不加分隔符,不同計數會拼出同一串。例:count=[1,11,0,…](a 出現 1 次、b 出現 11 次)拼成 "111…";而 count=[11,1,0,…] 也拼成 "111…" —— 兩個不同的字母分佈撞成同鍵,把不是 anagram 的字錯併一組。加 '#' 變成 "#1#11#0" vs "#11#1#0",就能區分。分隔符讓每個計數欄位邊界清楚。

A3

差距在 log L 因子,只有字串很長(L 上千、上萬)時才顯著。本題 L ≤ 100,log L ≈ 7,而排序常數小、程式短,實務上排序鍵完全夠快也更好寫。不值得為了理論上的 log L 改用更囉嗦的計數鍵 —— 除非題目明確 L 很大或卡常。面試講清楚兩者取捨即可。

A4

26 固定陣列失效(字元集不再是小寫英文)。這時排序鍵更通用 —— 對任何可比較的字元序列都能排序成指紋。若要計數,得改用 map<char,int> 或 unordered_map<char32_t,int> 動態計數再序列化成 key,但寫起來比排序麻煩。所以字元集不確定時,首選排序鍵。

A5

242 是「判斷兩個字是不是 anagram」,49 是「把一堆字按 anagram 分組」。兩者共用同一個核心:anagram 的標準指紋(排序字串 / 字母計數)。242 是比對兩個指紋是否相等;49 是拿指紋當雜湊 key 做分組。學會指紋這個觀念,兩題一起通。

App 1

質數乘積:由算術基本定理,每個正整數有唯一質因數分解,所以「字母→質數→乘積」把每個字母多重集合映到唯一乘積 → 是合法指紋(anagram 乘積相同)。隱患是溢位:長字串的乘積會爆 long long(如 100 個字母的乘積遠超 2⁶³)。要用大數或取模(取模又可能碰撞)。所以它是漂亮的理論解,實務上排序 / 計數鍵更穩。

App 2

LC 249:指紋改成「相鄰字母的差序列」(對 26 取模處理繞回)。例 abc 差 [1,1]、bcd 差 [1,1] → 同組;az 差 [25]、ba 差 [(0-1+26)%26=25] → 同組。把差序列轉字串當 key,一樣雜湊分桶。核心不變:設計一個「對目標等價關係不變」的指紋。

App 3

分桶時順便追蹤「目前最大桶」:每次 push_back 後比較該桶大小,更新一個 best 指標 / key。或分完後掃一遍 map 取 max_element by size。前者省一趟、O(1) 額外;後者更直觀 O(桶數)。都建立在同一個分桶結果上,不必重算指紋。
