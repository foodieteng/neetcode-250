範例 Trace// words = [wrt, wrf, er, ett, rftt]

// COLLECT + BUILD EDGES 出現的字母:w r t f e wrt vs wrf → 第 2 位 t≠f → 邊 t→f wrf vs er → 第 0 位 w≠e → 邊 w→e er vs ett → 第 1 位 r≠t → 邊 r→t ett vs rftt→ 第 0 位 e≠r → 邊 e→r 圖:w→e→r→t→f(一條鏈)
// A · DFS 三色後序 從某白點開始 DFS,沿 w→e→r→t→f 遞迴到底 f 無出邊 → 標黑、res="f" 回溯:t→res="ft", r→"ftr", e→"ftre", w→"ftrew" reverse("ftrew") = "wertf"
// B · Kahn 消入度 入度:w0 e1 r1 t1 f1 → 佇列=[w] 取 w → res="w",e 入度→0 入隊 取 e → res="we",r→0 入隊 取 r → res="wer",t→0；取 t → "wert",f→0；取 f → "wertf" res.size()=5 = 字母數 ✓
// RESULT // OUTPUT: "wertf"(兩種寫法此例都得同一鏈序)
這個例子是一條單鏈 w→e→r→t→f,拓樸序唯一,所以 DFS 與 Kahn 都給 "wertf"。 若圖存在多個入度 0 的起點,兩法可能給出不同但都合法的順序。

應用題// APPLY

App 1. 若題目保證輸入一定有解(無環、無前綴矛盾),程式可以安全省掉哪些檢查?省了之後風險在哪?
App 2. 若要回傳字典序最小的合法字母順序,兩種寫法哪個比較好改?怎麼改?
App 3. 若字母不是 26 個小寫,而是上萬種 Unicode 字元,對資料結構與複雜度分析有什麼影響?
// ANSWERS
A1. 字典序比較是「逐位比,遇到第一個不同就分勝負,之後不看」。所以相鄰兩字 w1, w2 只保證在第一個不同位 w1[j] < w2[j];再往後的字元屬於「已經分完勝負」的部分,兩者誰大誰小完全不受字母表限制,拿來加邊就是假關係。因此找到第一個不同就加邊並 break。
A2. 「前綴矛盾」= 前面完全相同,但前一個字比後一個長,例如 ["abc","ab"]。任何字母表下,一個字的前綴(ab)都必須排在它自己(abc)前面,不可能反過來。所以這種輸入本身違反字典序、無解,直接回 ""。程式用 j == minL && w1.size() > w2.size() 偵測。
A3. 三色:白(0)未訪、灰(1)正在遞迴堆疊中、黑(2)子樹已全部完成。DFS 在一個節點的所有後代都完成後才把自己加入 res(後序),得到的是「每個點都排在其後代之後」——也就是逆拓樸序,所以最後 reverse 轉回正序。判環:若沿途撞到灰色節點,代表回到還在堆疊中的祖先 = 回邊 = 有環。
A4. 同一條有向邊 c1→c2 可能被多對相鄰單字重複推出。若每遇到就 indegree[c2]++,但 graph 是 set 只會存一次,indegree 就會比實際入邊數多算。結果 c2 的入度永遠減不到 0,進不了佇列,最後輸出長度不足 → 誤判有環。所以要 if(!graph[c1].count(c2)) 確認是新邊,才同時 insert + indegree++。(DFS 版用 set 天生免疫。)
A5. 沒有任何邊的字母(只出現、不與別人比出順序)仍是字母表的一員,必須出現在答案裡。所以要先掃過所有字元把每個字母都放進 graph / indegree(入度 0)。漏了它,答案就會少字母;在 Kahn 版還會讓 res.size() != indegree.size() 的判斷失準。
App 1. 可省掉判環(DFS 撞灰 / Kahn 長度檢查)與前綴矛盾兩個回 "" 的分支,直接輸出拓樸序。風險:一旦「保證有解」的假設被打破(髒資料),程式會回傳不完整或錯誤的順序而不自知,除錯困難。實務上通常仍保留檢查。
App 2. Kahn 版好改:把普通佇列換成最小堆(priority_queue / set),每次取出目前入度 0 中字典序最小的字母即可,得到唯一的「字典序最小拓樸序」。DFS 逆後序不容易直接保證全域字典序最小,通常改用 Kahn + 小根堆。
App 3. 用 unordered_map 存圖與入度,本來就不綁定 26 個字母,換成大字集只是 V 變大。複雜度仍是 O(C + V + E),只是不能再把 V「當常數 26」忽略;若原本用固定大小 int[26] 陣列則要改回雜湊表。空間也隨相異字元數與邊數線性成長。
