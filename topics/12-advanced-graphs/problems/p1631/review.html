範例 Trace · A Union-Find// [[1,2,2],[3,8,2],[5,3,5]] → 2

// GRID (值 = 高度) · src=(0,0) dst=(2,2) 1 2 2 3 8 2 5 3 5 邊權 = 相鄰兩格 |高度差|。照 |Δ| 由小到大加入、union。
// KRUSKAL 加邊(依 |Δ|) |Δ|=0 (0,1)-(0,2)、(0,2)-(1,2) → 一座島 |Δ|=1 (0,0)-(0,1) → src 併入該島 |Δ|=2 (0,0)-(1,0)、(1,0)-(2,0)、(2,0)-(2,1)、(2,1)-(2,2) 加到 (2,1)-(2,2) 時 dst 併入 → src↔dst 連通!
// RESULT 最佳路徑沿邊界:1→2→2→(2)→3→5 … 最大 |Δ| = 2 中心高度 8 的格(|Δ| 到鄰居 = 5、6)完全沒用到 // OUTPUT: 2
答案 2 = 這條低起伏路徑上最陡的一步。高聳的中心格(8)被繞開 —— 這就是「最小化最陡瓶頸」: 只要那條最平緩的通道跨得過去即可。

應用題// APPLY

App 1. 若允許八方向(含斜向)移動,三種寫法各要改哪裡?複雜度會變嗎?
App 2. 若同一個 grid 要回答很多次「effort ≤ t 能否到達」的單一查詢,哪種前處理最划算?
App 3. 若不只要 effort 值,還要輸出最佳路徑本身,三種寫法哪個最好加、怎麼加?
// ANSWERS
A1. 因為「體力」的定義是「路上最累的一步」—— 只要有一步高度差很大,那條路的 effort 就被它決定,跟其他步走幾步、差多少無關。所以要最小化的是路徑最大邊權(瓶頸),不是總和。這類「最小化最大值」和一般「最短路徑和」是不同的目標函數。
A2. 邊照 |Δ| 由小到大加入,等於「門檻逐步放寬」。當加入某條權重 w 的邊後 src 與 dst 首次連通,代表存在一條全程邊權 ≤ w 的路;而在加入它之前不連通,代表沒有全程 < w 的路。所以 w 正是能連通的最小瓶頸 = 答案。這就是 Kruskal「加邊到連通」。
A3. 因為路徑成本是「沿途最大邊權」,不是總和。到鄰居的 effort = max(到我這格的 effort, 我與鄰居的 |Δ|) —— 走過去只會被「更陡的那一步」卡住。Dijkstra 的貪心(每次取當前 effort 最小的點)在 max 這個單調運算下依然成立,所以彈出終點時它的 effort 就是最小值。用 + 會變成求邊權和,答案錯。
A4. 可達性對門檻 mid 單調:mid 越大能跨的邊越多,可達的格子只增不減。所以「mid 能到終點」若成立,則所有更大的門檻也成立 → false…false true…true,可二分。上界設為全網格的最大 |Δ|:再大也不會有邊被擋,一定可達;下界 0(相鄰全等高時 effort=0)。
A5. 兩題都是最小化路徑瓶頸(minimax path),解法三招完全一樣(Union-Find / max-Dijkstra / 二分)。差別只在成本放哪:778 的成本是格子本身的高度(node 權),1631 的成本是相鄰兩格的高度差(edge 權)。778 想成「淹格」、1631 想成「跨邊」,骨架相同。
App 1. 只需把 dirs 從 4 方向改成 8 方向(加四個斜向),邊變多但三種寫法主體都不變。複雜度量級不變(每格鄰居 4→8 只是常數),仍是 O(mn·log)。
App 2. 用寫法 A 的離線做法:把邊依 |Δ| 排序、逐步 union,記錄「src 與 dst 在門檻多少時連通」(即答案 ans)。之後每個「effort ≤ t 能否到」查詢只要比較 t ≥ ans,O(1) 回答,不必每次重跑 BFS。若查詢是任意兩點,可在 union 過程建 Kruskal 重構樹,查詢兩點 LCA 上的最大邊權。
App 3. 寫法 B(Dijkstra)最好加:鬆弛成功更新 dist[nr][nc] 時,同時記 prev[nr][nc] = (r,c);最後從終點沿 prev 回溯到起點、反轉即得路徑。Union-Find / 二分只給 effort 值,要回溯路徑得額外再跑一次 BFS(只走 |Δ| ≤ ans 的邊)並記父節點。
