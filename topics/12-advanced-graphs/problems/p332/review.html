重點回顧// RECAP

面向要點
問題本質每條邊走一次 = 歐拉路徑,起點 JFK,取字典序最小
演算法Hierholzer:貪心走最小邊 + 邊走邊刪
資料結構multiset:自動字典序 + O(log) 取最小/刪除
關鍵手法死路節點後序 push,最後 reverse
為何有效死路必是尾端;繞路反轉後接回中間
字典序最小每步走「當前最小」的邊即可保證

觀念題// QUIZ

Q1. 為什麼不能邊走邊按順序輸出機場?會出什麼問題?
Q2. 為什麼用 multiset 而不是 vector 存每個機場的出邊?
Q3. 為什麼是「後序 push + 最後 reverse」?死路節點在其中扮演什麼角色?
Q4. 動畫例子裡 KUL 字母最小、又最先被走到,為什麼答案裡它排最後?
Q5. 這題和「拓樸排序 / 一般 DFS 走點」的本質差別在哪?
3

範例 Trace// [[JFK,KUL],[JFK,NRT],[NRT,JFK]]

// BUILD GRAPH (multiset 自動排序) graph[JFK] = { KUL, NRT } (K < N) graph[NRT] = { JFK } routes = []
// DFS(JFK) JFK: 取最小 KUL,刪 JFK→KUL,dfs(KUL) KUL: 無出邊 → push KUL routes=[KUL] JFK: 取 NRT,刪 JFK→NRT,dfs(NRT) NRT: 取 JFK,刪 NRT→JFK,dfs(JFK) JFK: 無出邊 → push JFK routes=[KUL,JFK] NRT: 無出邊 → push NRT routes=[KUL,JFK,NRT] JFK: 無出邊 → push JFK routes=[KUL,JFK,NRT,JFK]
// REVERSE routes = [KUL, JFK, NRT, JFK] reverse(routes) = [JFK, NRT, JFK, KUL] // OUTPUT: [JFK, NRT, JFK, KUL]
KUL 是字母最小、也最先被走到,但它沒有出邊(死路),所以最早被 push、 反轉後落到最後。這就是「順序輸出會錯、後序+反轉才對」的具體證據。

應用題// APPLY

App 1. 若改成要找歐拉迴路(走完所有邊並回到起點),存在性的判斷條件是什麼(有向圖)?
App 2. 遞迴深度可能到 E,擔心爆棧時,如何把這段 DFS 改寫成顯式 stack 的迭代版?整體邏輯有何不同?
App 3. 若題目不保證一定有解,如何偵測「無法用完所有機票」的情況?
// ANSWERS
A1. 因為貪心走「當前最小」可能提早走進死路,把還沒用的機票困在別的分支。例如 [[JFK,KUL],[JFK,NRT],[NRT,JFK]]:若順序輸出,從 JFK 先走最小的 KUL,KUL 沒有出邊就卡住了,但 NRT 那條環還沒走。正確做法是把死路 KUL 先擱著(push),回頭走完 NRT 的環,最後反轉 —— 死路自動被排到尾端。
A2. 每個機場的出邊需要兩個操作:取字典序最小與取完刪掉。multiset 保持排序,*begin() 取最小、erase(begin()) 刪除都是 O(log)。若用 vector,取最小要 O(size) 掃描、刪除要搬移元素,整體退化到 O(E²);還得自己維護有序。
A3. 一個節點只有在它所有出邊都走完之後才被 push(後序)。最深的死路最先 push —— 它是整條行程的最後一站;起點 JFK 最後 push —— 它是第一站。所以 routes 恰好是逆行程,reverse 一次就還原成正確順序。死路節點是這個機制的關鍵:它保證「走不下去的地方」落在尾端。
A4. 因為 KUL 沒有出邊,是死路。DFS 一走到它就卡住、立刻 push,成為 routes 的第一個元素。而 routes 是逆行程,反轉後第一個變最後一個。所以「字母最小、最早走到」與「答案裡的位置」無關 —— 決定位置的是它何時走投無路。
A5. 拓樸排序 / 一般 DFS 是走每個節點一次;這題是走每條邊一次(歐拉路徑),同一個機場可以重複造訪。所以狀態不是「節點是否訪問過」,而是「邊是否用過」—— 因此要邊走邊刪邊,而不是標記節點 visited。
App 1. 有向圖存在歐拉迴路的條件:① 每個節點入度 = 出度;② 所有有邊的節點屬於同一個(強)連通分量。歐拉路徑(不需回起點)則放寬為:至多一個節點出度−入度=1(起點)、至多一個入度−出度=1(終點),其餘入=出,且連通。
App 2. 用顯式 stack<string>:把 JFK 推入;當 stack.top() 還有出邊時,取最小、刪邊、把該鄰居推入 stack;當 top 無出邊時,把它 pop 並 append 到 routes。最後 reverse。邏輯與遞迴相同(stack 模擬呼叫堆疊),但不受遞迴深度限制。
App 3. 跑完之後檢查 routes.size() == tickets.size() + 1。歐拉路徑用掉 E 條邊會經過 E+1 個節點;若因為卡住而少於這個數,代表沒能用完所有機票(無解)。也可在建圖後先用 App 1 的入/出度 + 連通性條件做前置判斷。
