完整實作// SOLUTION

class Solution {
private:
    unordered_map<string, multiset<string>> graph;   // 出邊,天生字典序
    vector<string> routes;                        // 後序堆疊(逆行程)

public:
    vector<string> findItinerary(vector<vector<string>>& tickets) {
        for (auto& t : tickets) {
            graph[t[0]].insert(t[1]);
        }

        dfs("JFK");
        reverse(routes.begin(), routes.end());       // 逆行程 → 正行程
        return routes;
    }

    void dfs(string u) {
        while (graph.count(u) && !graph[u].empty()) {
            auto v = *graph[u].begin();          // 最小的出邊
            graph[u].erase(graph[u].begin());   // 用掉這張票
            dfs(v);
        }
        routes.push_back(u);                       // 無出邊 = 死路 → 後序 push
    }
};

複雜度分析// COMPLEXITY

步驟成本說明
建圖 insertO(E log E)每張票插入 multiset
DFS 消邊O(E log E)每條邊走一次 + O(log E) 刪除
reverseO(E)翻轉 routes
T = O(E log E)　　S = O(E)

E = 機票數(≤ 300)。log 來自 multiset 的排序與刪除。

邊界與陷阱// PITFALLS

陷阱後果正解
邊走邊順序輸出 提早卡死、行程不完整 必須後序 push + 反轉,死路才不會困住未用的票
用 vector 存出邊 + 每次 min O(E²) 或刪除麻煩 multiset:自動排序 + O(log) 取最小 / 刪除
走過的邊沒刪除 無限迴圈 / 重用票 取出 begin() 後立刻 erase
忘了最後 reverse 得到逆序行程 routes 是後序(逆行程),務必反轉
存取 graph[u] 前沒確認存在 意外插入空節點 先 graph.count(u) 再取(或無妨,但要清楚語意)

細節走讀// WALKTHROUGH

// MULTISET: 為什麼用它 每個機場的出邊要:① 依字典序取最小 ② 取完後刪掉 multiset 兩件事都是 O(log E): *graph[u].begin() 取最小 graph[u].erase(begin()) 刪掉那一張票 用 vector 則取最小要 O(size)、刪除要搬移,整體退化 O(E²)
// DFS: while 消邊 + 後序 push while(還有出邊){ 取最小 v;刪邊;dfs(v); } ↳ 一路往最小走,直到 u 沒有出邊 routes.push_back(u) ← 只有「卡住」時才執行 ↳ 死路節點最早被 push,是行程的尾端
// REVERSE: 為什麼逆序 後序:節點在「它所有出邊都走完」後才 push 第一個 push 的是最深的死路 = 行程最後一站 最後 push 的是 JFK = 行程第一站 所以 routes 是逆行程 → reverse 還原成正確順序
PREV 概念與動畫 NEXT 複習與 Trace
// REMEMBER
multiset 走最小 + 邊走邊刪;死路後序 push,最後 reverse。
