乘積最大子陣列

重點回顧// RECAP

面向要點
狀態maxP/minP = 以 i 結尾的最大 / 最小乘積
遇負swap(maxP, minP)(負數翻轉大小)
轉移maxP=max(maxP×num, num);minP 同理取 min
「重新開頭」num 那項:前面乘積不划算時斷開
答案ans=max(ans, maxP),初始 nums[0]
複雜度O(n) 時間、O(1) 空間

觀念題// QUIZ

Q1. 為什麼乘積版比「最大子陣列和(Kadane)」多需要記一個 minP?加法版為什麼不用?
Q2. swap(maxP, minP) 為什麼要放在乘之前?放在乘之後會錯在哪?
Q3. 轉移式裡 max(maxP × num, num) 的第二項 num(從自己重新開頭)在什麼情況下會勝出?舉個含 0 或大負數的例子。
Q4. 為什麼 ans 要初始成 nums[0] 而不是 0 或 INT_MIN?哪種輸入會踩雷?
Q5. 前後綴乘積解法為什麼「掃前綴 + 掃後綴」就一定涵蓋最大值?0 在裡面扮演什麼角色?

範例 Trace// nums=[2,3,-2,4]

// max/min 版(答案應為 6) i=0: maxP=minP=ans=2 i=1(3): maxP=max(2·3,3)=6 ; minP=min(2·3,3)=3 ; ans=6 i=2(-2): 負→swap→maxP=3,minP=6 maxP=max(3·-2,-2)=-2 ; minP=min(6·-2,-2)=-12 ; ans=6 i=3(4): maxP=max(-2·4,4)=4 ; minP=min(-12·4,4)=-48 ; ans=6 return ans = 6(來自子陣列 [2,3])
這個例子的答案 6 來自負數前面的 [2,3] —— 遇到 -2 後 maxP 掉到負值、乘積被「打斷」,但 ans 已經把 6 記下來。 這說明:ans 記的是全程看過的最大 maxP,不是最後一步的 maxP;答案可能出現在中途任何位置。

應用題// APPLY

App 1. 53 最大子陣列和(Kadane):把這題的「乘」換成「加」,程式要怎麼簡化?為什麼加法版不需要 minP 和 swap?
App 2. 若把限制改成「子陣列乘積最小」,程式要改哪裡?為什麼還是要同時記 max 與 min?
App 3. 若允許陣列含 0,且要回報最大乘積子陣列的起訖位置(不只值),max/min 版要額外記什麼?為什麼 0 讓這件事變棘手?
// ANSWERS
A1. 因為乘法會被負數翻轉:一個很小的負數(絕對值大的最小乘積)乘上另一個負數,會變成很大的正數。所以「以 i 結尾的最大乘積」的來源之一,是「以 i−1 結尾的最小乘積 × 負的 nums[i]」—— 必須記 minP 才接得住。加法版(Kadane)沒有這種翻轉:加一個數不會讓「最小和」突然變成「最大和」,大小關係穩定,所以只記最大就夠。
A2. 因為交換的語意是「在乘以這個負數之前,先把大小關係翻轉過來」。swap 後,原本的 minP 坐到 maxP 的位置,接著 maxP = max(maxP×num, num) 就會用「舊 minP × 負數」當候選 —— 正好是我們要的翻轉。若把 swap 放在乘之後,maxP/minP 已經是用錯誤基底算出來的髒值,交換它們毫無意義,答案會錯。順序:先 swap、再乘、再更新 ans。
A3. 當「前面的乘積 maxP 乘上 num 反而比 num 本身還小」時,num 那項勝出,代表從這裡重新開一段。典型:前一步 maxP=0(碰到 0)或前一步 maxP 是個很負的數。例 [-8, 5] 在 i=1:max(-8×5, 5)=max(-40,5)=5,選了 num=5、丟掉前面的 -8。這就是子陣列「可以不從頭開始」的體現。
A4. 因為答案可能是負數(全負或只有一個負元素時)。若 ans=0,像 [-2] 會回傳 0(錯,應為 −2);若 ans=INT_MIN 雖不會被錯誤蓋掉,但沒必要且易混。用 nums[0] 當起點最穩:它保證是一個合法的子陣列乘積(單一元素),後續只會用更大的取代它。踩雷輸入:單一負數、全負陣列,以及含 0 但最優仍是負數的情況。
A5. 關鍵事實:一段不含 0 的子陣列,其乘積的正負只由負數個數決定。若整段乘積為正,它就是候選最大;若為負,想得到最大正乘積,必須去掉一端的一個負數 —— 而「去掉左端後剩的」正是某個後綴、「去掉右端後剩的」正是某個前綴。所以掃一次前綴、一次後綴,必有一個方向湊到最大。0 的角色是分段器:乘到 0 會歸零,程式把 prefix/suffix 重設為 1,等於在 0 處把陣列切開、各段獨立處理,自然避免「跨越 0」的無效乘積。
App 1. 加法版(53)只需一個變數:cur = max(cur + num, num); ans = max(ans, cur)。不需要 minP 也不需要 swap,因為加法沒有「翻轉」—— 把一個數加到「最小和」上,它仍是相對小的,不會突然變成最大和。乘積版之所以複雜,全是因為負數的乘法翻轉;拿掉乘法這個特性,就退化成乾淨的 Kadane。
App 2. 求「乘積最小」時,把回報從 ans=max(ans, maxP) 改成 ans=min(ans, minP)(追蹤 minP 的最小值)。仍要同時記 max 與 min:因為負數一樣會翻轉 —— 想得到最小(最負)乘積,可能來自「最大正乘積 × 負數」。對稱地,maxP 是 minP 遇負時的潛在來源。所以無論求最大或最小,兩個都得維護,只差最後回報哪一個。
App 3. 要回報起訖,得在更新時記錄區間端點:當 maxP 取 num(重新開頭)時,把「當前段起點」設為 i;當 ans 被 maxP 刷新時,記下 (起點, i)。麻煩在 swap 與翻轉會讓「最大乘積對應的實際區間」在 maxP/minP 之間跳動,得同時追蹤兩者各自的起點;而 0 會強制斷段、重設起點。相比只求值,追蹤區間要多維護數個指標且邏輯更細 —— 這也是為什麼多數版本只回傳乘積值。
