範例 Trace// nums=[2,1,3], target=4

// 金額外層填表(答案應為 7) dp[0]=1 dp[1]= dp[0](num1) = 1 dp[2]= dp[1](num1)+dp[0](num2) = 2 dp[3]= dp[2](num1)+dp[1](num2)+dp[0](num3) = 4 dp[4]= dp[3](num1)+dp[2](num2)+dp[1](num3) = 4+2+1 = 7 return dp[4] = 7(和 nums 順序無關,結果一樣)
注意:把 nums 從 [1,2,3] 換成 [2,1,3],答案不變(都是 7) —— 因為內層 num 只是「加總的順序」,而加法可交換。 真正決定「排列 vs 組合」的是金額 vs 物品誰在外層,不是 nums 內部的排序。

// ANSWERS
A1. 金額外層時,對每個金額 i 都重新枚舉所有 num,等於問「最後一個數是誰」—— 湊 3 的方案裡,「最後放 2、前面湊 1」給出 (1,2),「最後放 1、前面湊 2」給出 (2,1),兩者在 dp[3] 被分別加到 → 排列。物品外層時,把一種硬幣「整批」用完才換下一種,枚舉的是「用幾個 1、幾個 2」的無序組合,(1,2) 和 (2,1) 是同一種、只算一次 → 組合。外層放金額 = 允許每個位置自由選 = 有順序 = 排列。
A2. 因為排列數長很快,中途 dp[i] 可能超過 int 上限 2³¹−1。而 C++ 對有號整數溢位的規定是 Undefined Behavior(未定義行為) —— 不是「保證變負」,而是「編譯器可以假設它不會發生」,結果可能是負值、可能被最佳化成任意行為、甚至讓周邊程式碼一起壞掉。所以不能靠「它會 wrap 成負」來寫程式,必須換型別從根本避免。
A3. unsigned 的算術是模 2³²:溢位時有定義地環繞(超過 2³²−1 就從 0 重來),這是標準保證、不是 UB。最後仍正確,是因為整個 DP 只有加法,而模運算對加法封閉:(a+b) mod m = ((a mod m)+(b mod m)) mod m。所以每一格 dp[i] 都被正確算成「真值 mod 2³²」,就算中途某格真值爆過 2³² 而環繞,它帶給後面的仍是正確的 mod 值,dp[target] 因此永遠 = 真值 mod 2³²。當真值 < 2³²(題目保證 < 2³¹),這個 mod 值就等於真值本身。
A4. long long 有 64 位,對本題(答案 fits 32-bit、中途值遠不到 2⁶³)根本不會溢位,不需要任何 mod 推理,程式意圖一目了然,面試最不容易出錯或被追問。unsigned int 的前提比較細:①只做加法(或其它 mod 封閉運算)②最終答案 < 2³²(這裡還要 < 2³¹ 才能 cast 回 int 不變負)。省一半記憶體、是競賽習慣,但要能講清楚 mod 語意。穩健用 long long,懂原理用 unsigned。
A5. 差在聚合運算對順序的敏感度。322 求「最少枚數」用 min,而 min 只在意最小值、和「用什麼順序枚舉、算到排列還是組合」無關 —— 不管金額外層還是硬幣外層,最終每個 dp[i] 都收斂到同一個最小值,所以順序免疫。本題求「方法數」用加法計數,而「數幾種」天生要區分「算不算順序」:金額外層把不同順序分開加(排列)、物品外層把同組合只加一次(組合),兩者數出不同的量。求極值(min/max)對迴圈順序免疫;做計數(+)則順序決定排列還是組合。
App 1. 把迴圈內外對調:硬幣(物品)放外層、金額放內層,轉移仍是 dp[i] += dp[i−coin],base dp[0]=1。這樣每種硬幣「整批」被考慮一次,枚舉的是無序組合(先決定用幾個 coin₁、再 coin₂…),(1,2) 與 (2,1) 只在同一個外層階段被算一次 → 組合。轉移式一字不改,只換外層 —— 這正是 377 ↔ 518 的唯一差別,也是「計數 DP 迴圈順序」最經典的一對。
App 2. 聚合從「累加」換成「取最小 + 1」:dp[i] = min over num ( dp[i−num] + 1 ),base dp[0]=0、其餘 ∞。這就變成 322。此時迴圈順序不再重要:因為 min 對順序免疫(見 A5),金額外層或物品外層都得到同樣的最少枚數。所以「排列 vs 組合」只在計數時才是問題,求極值時消失。
App 3. 若 nums 含負數,這個「金額由小到大填」的 DP 會失效:負數能讓總和來回增減,可能構造出無窮多排列(例如 [1,−1] 湊 0 可以無限加 1,−1,1,−1…),答案不再有限、DP 也沒有「由小到大」的良好拓撲序。LeetCode 的 follow-up 正是問這個:「若允許負數,還需要什麼額外限制?」 —— 通常要限制每個數字的使用次數上限(或序列長度上限),否則問題無界。這提醒我們這套 DP 隱含假設「每個 num > 0,所以金額嚴格變大、能拓撲排序」。
