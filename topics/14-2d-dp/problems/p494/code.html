完整實作// O(n·P) TIME · O(P) SPACE

p494.cpp C++17 
class Solution {
public:
    int findTargetSumWays(vector<int>& nums, int target) {
        int total = accumulate(nums.begin(), nums.end(), 0);
        int sum = target + total;

        // 守門 1:目標超出 [−total, total] → 不可能
        // 守門 2:sum 是奇數 → P 不是整數 → 不可能
        if (total < abs(target) || sum & 1) return 0;

        int plus = sum / 2;            // P = (target + total) / 2,常數

        vector<unsigned int> dp(plus + 1, 0);
        dp[0] = 1;                  // 空集合:湊出和 0 有 1 種

        for (int num : nums) {
            for (int i = plus; i >= num; i--) {   // 由大到小:每個數字只用一次
                if (dp[i - num] != 0) {            // 純加速,可省
                    dp[i] += dp[i - num];        // 計數:用 + 不是 OR
                }
            }
        }

        return (int)dp[plus];
    }
};

骨架和 416 一模一樣 —— 唯一的差別是 dp[i] = dp[i] || dp[i−num] 換成 dp[i] += dp[i−num]。 「能不能」用 OR、「幾種」用 +,其餘(內層由大到小、dp[0] 當種子)完全共用。

這題的 unsigned:可留,但不是必要// vs. 377

你在 377 組合總和 IV 用 unsigned int 是不得不 —— 那題的真實答案會超過 INT_MAX,靠 unsigned 的 mod 2³² 環繞行為讓中間值不觸發 UB。 這題不一樣,值得分清楚:

上界 · 為什麼 494 永遠不會溢位

答案 = 符合條件的 ± 組合數,而總共才 2ⁿ 種組合。題目給 n ≤ 20,所以:

　答案 ≤ 2²⁰ = 1,048,576
　INT_MAX = 2,147,483,647
　→ 只用掉 0.05% 的範圍

中間過程也不會超過(每格的 dp[i] 都是某種子集數,同樣 ≤ 2²⁰)。 所以 int 就夠,unsigned int 只是多的保險 —— 留著沒錯,但別以為是它救了你。 這兩題的差別是「真實答案的上界」,不是「用哪個型別比較安全」。

377 組合總和 IV494 目標和(本題)
真實答案上界可達 ~3.8×10⁹(超過 INT_MAX)2²⁰ ≈ 10⁶
int 會怎樣有號溢位 = UB,答案變負數完全沒事
unsigned 角色必要 —— 定義良好的 mod 2³²,湊出題目要的截斷值可有可無
教訓先估答案的上界,再選型別 —— 不是反過來
unsigned 的溢位是定義良好的 mod 2³²;int 的溢位是 UB。 377 需要前者、494 兩者都不會碰到。面試講得出「答案上界是 2ⁿ ≤ 2²⁰,所以 int 就夠」,比默默寫 unsigned 有說服力得多。

步驟成本說明
accumulateO(n)一次掃過
兩道守門O(1)提早擋掉不可能的輸入
開 dpO(P)P ≤ (1000+1000)/2 = 1000

偽多項式:成本跟 P 的數值成正比,不是跟它的位元數。 若題目改成 nums[i] ≤ 10⁹、n 仍 ≤ 20,背包就不可行了(P 太大開不了表), 那時候該換 meet-in-the-middle(兩半各 2¹⁰ 再合併)。

邊界與陷阱// PITFALLS

陷阱症狀正解
沒擋 total < |target| P 算出負數 → vector 開負長度,直接爆 先判 total < abs(target) → 回 0
沒擋奇偶 sum/2 無聲捨去 → 算出錯的 P,回錯答案。[2,2]、target=1 應為 0 sum & 1 → 回 0
內層由小到大 同一個數字被重複使用 → 變完全背包,答案偏大 必須 for (i = plus; i >= num; i--)
nums 含 0 以為要特判 —— 其實不用。每個 0 讓答案 ×2(+0/−0 是兩個不同算式) dp[i] += dp[i−0] 自動 ×2,程式碼原封不動
忘了 dp[0]=1 整張表全 0,答案永遠 0 空集合湊出和 0 有 1 種,是計數的種子
誤用 OR 抄 416 抄過頭,得到「能不能」而不是「幾種」 計數要 +=

細節走讀// WALKTHROUGH

// GUARD: 運算子優先順序沒問題 total < abs(target) || sum & 1 C++ 優先順序: < (關係) > & (位元) > || (邏輯) 實際解析成: (total < abs(target)) || ((sum) & 1) ← 正是想要的 而且短路求值幫了忙: 第一項成立就 return,不會進到第二項 → 到 sum & 1 時保證 sum ≥ 0,不會對負數做位元判斷
// ZEROS: 為什麼 nums 含 0 不用特判 num = 0 時,內層 for (i = plus; i >= 0; i--): dp[i] += dp[i - 0] dp[i] += dp[i] ← 讀自己! dp[i] *= 2 語意剛好對:一個 0 可以掛 +0 或 −0,是「兩個不同算式、同一個和」 → 每個 0 讓所有方法數 ×2 nums=[1,0], target=1: total=1, sum=2, plus=1 放 1 → dp=[1,1] 放 0 → dp=[2,2] ← 每格 ×2 return dp[1] = 2 ✓ ("+1+0" 和 "+1-0") 註:這裡「由大到小」不是關鍵 —— num=0 的每格只讀自己,順序無關。
// MAIN: [1,1,1,1,1], target=3 —— 每列都是二項式係數 total = 5, sum = 3+5 = 8(偶數 ✓), plus = 4 和: 0 1 2 3 4 初始 [ 1, 0, 0, 0, 0 ] 放第 1 個 1 [ 1, 1, 0, 0, 0 ] = C(1,s) 放第 2 個 1 [ 1, 2, 1, 0, 0 ] = C(2,s) 放第 3 個 1 [ 1, 3, 3, 1, 0 ] = C(3,s) 放第 4 個 1 [ 1, 4, 6, 4, 1 ] = C(4,s) 放第 5 個 1 [ 1, 5, 10, 10, 5 ] = C(5,s) return dp[4] = 5 ← C(5,4):5 個 1 挑 4 個給正號,剩 1 個給負號 → 4−1 = 3 ✓

為什麼是巴斯卡三角:dp[i] += dp[i−1] 正是 C(k,s) = C(k−1,s) + C(k−1,s−1) —— 「這個 1 不選」+「這個 1 選」。全 1 的輸入讓背包退化成組合數,是驗算時很好用的一組測資。
