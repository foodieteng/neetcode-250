範例 Trace// nums=[1,2,3], target=0

// 改寫 total = 1+2+3 = 6 sum = target + total = 0 + 6 = 6 (偶數 ✓) plus = 6 / 2 = 3 → 數「和 = 3 的子集」有幾個
// 填表(內層由大到小) 和: 0 1 2 3 初始 [ 1, 0, 0, 0 ] 放 1 [ 1, 1, 0, 0 ] i=3..1:dp[1] += dp[0] 放 2 [ 1, 1, 1, 1 ] i=3..2:dp[3] += dp[1], dp[2] += dp[0] 放 3 [ 1, 1, 1, 2 ] i=3: dp[3] += dp[0] return dp[3] = 2
// 驗算:列出全部 2³ = 8 種符號 +1+2+3 = 6 -1+2+3 = 4 +1+2-3 = 0 ✓ -1+2-3 = -2 +1-2+3 = 2 -1-2+3 = 0 ✓ +1-2-3 = -4 -1-2-3 = -6 = 0 的有 2 種 ✓ 對應子集 {1,2}(和=3)和 {3}(和=3)—— 正是 dp[3] = 2
注意「子集」和「算式」是一一對應的:選進正號堆的子集 {1,2} ↔ 算式 +1+2−3。 所以數子集就是數算式 —— 這就是為什麼改寫之後不會多算也不會少算。

App 1. 若 nums[i] 可以到 10⁹(n 仍 ≤ 20),背包還可行嗎?該換什麼解法?複雜度多少?
App 2. 若把「有幾種」改成「存不存在一種」,程式要改哪一個字?若改成「列出所有算式」呢?
App 3. 1049 石頭重量 II 用同一個 ± 改寫,為什麼它的 DP 卻不是計數?兩題在「要什麼」上差在哪?
// ANSWERS
A1. 設正號堆的和 P、負號堆的和 N。算式的值是 P − N = target,而所有數字的總和 P + N = total。兩式相加消掉 N:2P = target + total ⇒ P = (target+total)/2。關鍵:target 和 total 都是輸入給定的,所以 P 是一個算得出來的常數,不是要搜尋的變數。這讓問題從「枚舉 2ⁿ 種符號配置」變成「數出和 = 某個固定值的子集個數」—— 一個標準 0/1 背包,O(n·P) 就解決。整題的價值全在這一步。
A2. 守門 1 total < |target|:全正最多 total、全負最少 −total,目標超出這區間就不可能。危險的是 target 為負的那側 —— 實測 nums=[1]、target=−5:sum = −4、P = −2,vector(plus+1) 等於開一個長度 −1 的 vector,直接爆(size_t 環繞成天文數字 → bad_alloc)。
(反過來 nums=[1]、target=5 不擋剛好沒事:P=3 是正的,dp[3] 湊不到而回 0 —— 但那是運氣,不是保證。)
守門 2 sum & 1:P 是若干整數的和,必須是整數;target+total 為奇數代表湊不出來。不擋的話整數除法無聲捨去。實測 nums=[2,2]、target=1:±2±2 恆為偶數,正確答案是 0,但不擋會算成 sum=5、P=2,回傳 2 —— 不當機、不報錯,就是錯。這比守門 1 的崩潰更可怕。
順序不能對調:靠 || 的短路,先判 total < |target| 才保證到達 sum & 1 時 sum ≥ 0。對負數做 & 1 在二補數下雖仍能測奇偶,但那是依賴表示法的脆弱寫法,不該用。
A3. 會變成完全背包(每個數字可用無限次),答案偏大。由小到大時,dp[i−num] 可能已經在本輪被更新過,代表「這個數字已經用過一次」的狀態,再加一次就等於重複使用同一個數字。
nums=[1,1]、target=0:total=2、P=1,正確答案 2(+1−1、−1+1)。由大到小:放第一個 1 → dp=[1,1];放第二個 1 → dp[1] += dp[0] → dp=[1,2] ✓。由小到大:放第一個 1 時 dp[1] += dp[0] → dp=[1,1](這裡還看不出來,因為 P 太小);把 P 放大就會看到同一個數字被反覆疊加。口訣:0/1 背包由大到小(每個一次)、完全背包由小到大(可重複)。
A4. 因為 dp[i] += dp[i−0] 就是 dp[i] += dp[i],也就是 dp[i] *= 2 —— 每個 0 讓所有方法數翻倍,而這正是正確語意:一個 0 可以掛 +0 或 −0,是兩個不同的算式卻給出同一個和。例 nums=[1,0]、target=1:答案是 2(+1+0 和 +1−0),程式跑出 dp=[2,2] → dp[1]=2 ✓。
附帶一提:num=0 時每一格只讀自己,所以「由大到小」在這裡不是關鍵,順序無所謂 —— 這是 0 的特例,但剛好不影響正確性。結論:0 是這題最常被過度擔心的地方,實際上原本的程式碼一個字都不用改。
A5. 不必要,但無害。估上界:答案是「符合條件的 ± 組合數」,而總共只有 2ⁿ 種組合,題目給 n ≤ 20 ⇒ 答案 ≤ 2²⁰ = 1,048,576,只有 INT_MAX 的 0.05%;中間每格 dp[i] 也是某種子集數,同樣受 2²⁰ 限制。所以 int 完全夠用。
377 則相反:nums=[1,2,3]、target=37 的真實答案是 3,831,006,429 > INT_MAX。用 int 是有號溢位 = UB(實測跑出 −463960867);unsigned int 的溢位則是定義良好的 mod 2³²,剛好給出題目要的截斷值。
方法論:先用組合論估「真實答案的上界」,再決定型別 —— 不是反過來憑感覺加 unsigned。面試講得出 2ⁿ ≤ 2²⁰ 這句,比默默寫 unsigned 有說服力得多。
App 1. 不可行 —— 背包是偽多項式,成本 O(n·P) 跟 P 的數值成正比。nums[i] ≤ 10⁹ 時 P 可達 10¹⁰,表根本開不出來。
此時該換 meet-in-the-middle:把 20 個數字切成兩半各 10 個,各自枚舉 2¹⁰ = 1024 種帶號和,左半存進 hash map;再掃右半的每個和 r,查左半有幾個 target − r,累加。時間 O(2^(n/2)) ≈ 2048 次 + 查表,空間 O(2^(n/2))。判斷準則:n 小但數值大 → meet-in-the-middle;n 大但數值小 → 背包。
App 2. 「存不存在」:把 dp[i] += dp[i−num] 換成 dp[i] = dp[i] || dp[i−num](型別改 bool),回 dp[P] —— 這就變回 416 了。也可以偷懶:直接判 dp[P] != 0,計數非零即存在。
「列出所有算式」:那就不能只留一維。要嘛保留二維表 dp[k][i](前 k 個數字湊出 i 的方法數)再從 (n, P) 回溯每個決策,要嘛直接 DFS 枚舉 —— 因為輸出量本身就是 O(答案數 × n),最壞 2ⁿ,沒有比指數更好的做法(輸出下界擺在那)。這是「計數比列舉容易」的典型例子。
App 3. 兩題的改寫完全相同(± 號 = 分兩堆 P/N),差別在要什麼:
● 494:目標 target 給定 ⇒ P = (target+total)/2 直接算得出來,剩下的問題是「和 = P 的子集有幾個」→ 計數 DP(+=),答案讀 dp[P]。
● 1049:沒有給定目標,要最小化 |P − N| = |total − 2P| ⇒ P 是要搜尋的變數,問題是「哪些和可達、最接近 total/2」→ 可行性 DP(OR),答案是 total − 2·best。
一句話:P 已知 → 數方法(+);P 未知要最佳化 → 找可達的極值(OR)。同一個改寫,聚合方式跟著「問題要什麼」走 —— 這正是背包家族最值得記住的分辨點。
