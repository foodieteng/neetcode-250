重點回顧// RECAP

面向要點
狀態hold(持有)· sold(剛賣,明天不能買)· rest(空手可買)
為何拆兩個「空手」兩者接下來能做的事不同 → 不能合併
冷凍期在哪hold[i] = max(hold[i−1], rest[i−1] − p[i]) —— 買進碰不到 sold
sold 沒有 max它的定義就是「今天賣了」,不賣屬於別的狀態
答案max(sold[n−1], rest[n−1]) —— 不含 hold
壓縮只讀 i−1 → 三個變數,O(1);三行一律用 prev_*

觀念題// QUIZ

Q1. 為什麼「空手」要拆成 sold 和 rest 兩個狀態?兩個狀態什麼時候該合併、什麼時候不能?
Q2. 冷凍期具體體現在哪一行、哪一個字?把那個字改掉會變成哪一題?[1,2,3,0,2] 的答案會變多少?
Q3. sold[i] = hold[i−1] + p[i] 為什麼沒有 max?其他兩條都有,這條憑什麼不用?
Q4. 最後為什麼是 max(sold, rest) 而不含 hold?
Q5. O(1) 版裡 hold = max(prev_hold, reset − prices[i]) 用的是 reset 不是 prev_reset。這樣對嗎?為什麼?該不該改?

範例 Trace// [1,2,3,0,2]

// 完整表(實跑輸出) day 0 1 2 3 4 price 1 2 3 0 2 hold -1 -1 -1 1 1 sold 0 1 2 -1 3 rest 0 0 1 2 2 answer = max(sold[4], rest[4]) = max(3, 2) = 3
// 逐格對照策略 sold[1] = hold[0] + 2 = -1 + 2 = 1 第 0 天買(1)、第 1 天賣(2) → 賺 1 rest[2] = max(rest[1], sold[1]) = max(0, 1) = 1 第 2 天冷凍期,獲利 1 轉進 rest hold[3] = max(hold[2], rest[2] - 0) = max(-1, 1) = 1 第 3 天用「冷凍完的錢」買價 0 的股票 sold[4] = hold[3] + 2 = 1 + 2 = 3 第 4 天賣(2) → 總獲利 3 DP 表和最佳策略逐格吻合
注意 sold[3] = −1 是個爛狀態(第 3 天用價 0 賣掉手上的股票 = 白虧), 但它不會害到答案 —— max 自然會把它淘汰。 DP 不需要你避開爛選擇,只需要你把選擇列全。

應用題// APPLY

App 1. 若冷凍期變成 k 天,狀態機要怎麼改?空間變多少?
App 2. 若每筆交易要收 手續費 fee(= LC 714,無冷凍期),狀態機長什麼樣?幾個狀態就夠?
App 3. 若限制最多交易 2 次(LC 123),為什麼不能只靠 hold/sold/rest?要加什麼?
// ANSWERS
A1. 因為「空手」這個描述資訊不足:明天能不能買,取決於「是不是今天才賣的」。sold(今天剛賣)明天不能買、rest(空手且非剛賣)明天可以買 —— 兩者現金可能一樣,但可選動作不同。
通則:兩個處境能不能合併成同一個狀態,看的不是「現在的值像不像」,而是「接下來能做的事一不一樣」。一樣 → 可合併;不一樣 → 必須拆開。冷凍期的全部資訊,就是靠這個拆分承載的 —— 沒有任何一行程式碼在「數冷凍幾天」,它被編碼進狀態本身。這是狀態機 DP 最核心的設計直覺。
A2. 在 hold 那一行,那個字是 rest:
hold[i] = max(hold[i−1], rest[i−1] − p[i])
買進的資金只能來自 rest,碰不到 sold。昨天剛賣的錢必須先花一天走到 rest 才能再買 —— 那一天就是冷凍期。
把它放寬成 max(rest[i−1], sold[i−1]) − p[i],冷凍期就消失,題目退化成 LC 122(無限次交易、無冷凍)。實測:放寬版和 122 的標準解(累加所有正價差)在 505 筆隨機測資上完全吻合,0 筆不同。[1,2,3,0,2] 從 3 變成 4;[6,1,3,2,4,7] 從 6 變成 7。一個字,換一道題。
A3. 因為 sold[i] 的定義就是「第 i 天賣掉了」。既然賣了,就只有一種來路:昨天必然持有(hold[i−1]),今天以 p[i] 出手。沒有第二個選項可以 max。
「今天不賣」這件事不屬於 sold —— 它屬於 hold(繼續抱)。這正是狀態機的乾淨之處:每個狀態只負責自己那件事,選擇由狀態之間的 max 完成,不是在狀態內部。
如果你在這裡寫了 max(sold[i−1], hold[i−1] + p[i]),語意就被污染成「今天賣或以前賣過」,和 rest 的職責重疊,狀態機就爛掉了。
A4. 因為 hold[n−1] 代表「最後一天還抱著股票」——那筆錢沒有兌現,而且 hold 是扣掉買入成本後的現金。抱著不賣沒有任何好處:同樣的局面,賣掉只會更多(或至少不會更少)。
形式上:sold[n−1] = hold[n−2] + p[n−1] ≥ hold[n−2](價格非負),而 hold[n−1] ≥ hold[n−2] 也成立,但 hold[n−1] 手上的股票沒算進現金 —— 它永遠不是最大值。
寫 max(hold, sold, rest) 不會錯(只是多餘),但能講出「為什麼 hold 不可能贏」比較加分 —— 那代表你知道每個狀態的值到底是什麼單位。
A5. 對,但理由很脆弱,建議改掉。
它正確,純粹因為 reset 在第 3 行才被賦值 —— 第 1 行讀到的 reset 剛好還是昨天的值,等於 prev_reset。所以結果對(實測 711 筆全過)。但這是「順序剛好」,不是「邏輯保證」。
實測地雷:把 reset = max(prev_reset, prev_sold) 那行搬到最前面(三行都用 prev_* 快照,看起來順序無所謂),hold 就會讀到新的 reset,等於允許「昨天賣、今天買」→ 冷凍期消失。[1,2,3,0,2] 從 3 變 4、[6,1,3,2,4,7] 從 6 變 7、[1,4,2,8] 從 7 變 9。
修法:第 1 行寫 prev_reset。你既然特地抓了三個快照,它們的意義就是「這一輪只准讀昨天」—— 抓了就全部用,不要留一個漏網之魚。這樣三行順序才真的無所謂,日後自己或別人重排也不會踩雷。「碰巧對」和「保證對」在面試裡是兩件事,後者才叫寫得好。
App 1. 冷凍 k 天 = 「賣出後要等 k 天才能買」。
做法一(直觀):把 rest 拆成 k 個階段 rest_1 … rest_k(冷凍第 1 天、第 2 天…),依序流動,最後一個才接回 hold。狀態數 O(k),空間 O(k)。
做法二(更簡潔):直接回到陣列版,把轉移改成 hold[i] = max(hold[i−1], rest[i−k−1] − p[i]),也就是「往回看 k+1 天」。這時不能只留三個變數 —— 必須保留最近 k+1 天的歷史,空間 O(k)(或用環形緩衝)。
重點:本題能壓到 O(1) 的唯一原因就是 k=1(只依賴 i−1)。k 一變大,O(1) 就沒了 —— 這也提醒我們:空間壓縮是依賴結構的結果,不是隨手可套的技巧。
App 2. LC 714 沒有冷凍期,所以 sold 和 rest 可以合併(依 A1 的準則:兩者明天都能買,能做的事一樣)→ 只要兩個狀態:
hold = max(hold, cash − p[i])
cash = max(cash, hold + p[i] − fee)
答案 cash。手續費在賣出時扣一次即可(買賣各扣一半也行,只要別重複算)。
這題正好反過來印證 309 為什麼需要三個:少了冷凍期這個限制,那個區分就不再有資訊價值,狀態立刻塌成兩個。狀態的數量,是被「規則」逼出來的,不是憑感覺選的。
App 3. 因為 hold/sold/rest 不記得你交易過幾次 —— 「最多 2 次」是個資源限制,必須進到狀態裡,否則 DP 無從知道還剩幾次額度。
做法:加一維交易次數 → dp[i][j][holding](第 i 天、已用 j 次、是否持有),j ∈ {0,1,2}。或展開成 5 個經典狀態:buy1 / sell1 / buy2 / sell2(+ 初始),每天依序更新。時間 O(n·k)、空間 O(k),k=2 時就是 O(n)/O(1)。
再一般化就是 LC 188(最多 k 次)。
整個股票家族的統一觀:全部都是同一台狀態機,差別只在「規則往狀態裡加了什麼」—— 121 = 只能 1 次 · 122 = 無限次 · 123/188 = 加「次數」維 · 309 = 加「冷凍」狀態 · 714 = 加手續費(不加狀態)。先問「規則需要記住什麼」,狀態就設計出來了。
