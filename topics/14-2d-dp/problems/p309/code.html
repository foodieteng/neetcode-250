買賣股票含冷凍期

寫法 A · 三條陣列(先寫這個最好懂)// O(n) SPACE

p309.cpp C++17 
class Solution {
public:
    int maxProfit(vector<int>& prices) {
        int n = prices.size();

        vector<int> hold(n), sold(n), rest(n);
        hold[0] = -prices[0];          // 第一天就買:現金 = −p[0]
        sold[0] = 0;
        rest[0] = 0;

        for (int i = 1; i < n; ++i) {
            hold[i] = max(hold[i - 1], rest[i - 1] - prices[i]);  // 買進只能來自 rest = 冷凍期
            sold[i] = hold[i - 1] + prices[i];                // 賣掉:昨天必然持有
            rest[i] = max(rest[i - 1], sold[i - 1]);          // 昨天賣的,今天冷凍完
        }

        return max(sold[n - 1], rest[n - 1]);   // 不看 hold:抱著不賣不會最佳
    }
};

三條陣列全部只讀 i−1 —— 沒有任何跨天的依賴。這也是下面能壓成 O(1) 的原因。

寫法 B · 三個變數(follow-up)// O(1) SPACE

p309_o1.cpp C++17 
class Solution {
public:
    int maxProfit(vector<int>& prices) {
        int hold = -1 * prices[0];
        int sold = 0;
        int reset = 0;
        for (int i = 1; i < prices.size(); i++) {
            int prev_hold = hold;          // 先把昨天的三個值凍住
            int prev_sold = sold;
            int prev_reset = reset;

            hold = max(prev_hold, reset - prices[i]);
            sold = prev_hold + prices[i];
            reset = max(prev_reset, prev_sold);
        }

        return max(sold, reset);
    }
};

因為三個狀態都只讀昨天,陣列整條都是浪費 —— 三個變數就夠,O(1) 空間。 先把 prev_* 全部抓好再更新,是正確且推薦的寫法(下面說明為什麼)。

3

寫法 B 藏了一顆地雷// STATEMENT ORDER

你的 B 版是對的(實測 711 筆全過)。但有一行值得盯著看:

// 注意這裡混用了 reset 和 prev_reset hold = max(prev_hold, reset - prices[i]); ← 用的是 reset,不是 prev_reset sold = prev_hold + prices[i]; ← 用 prev_hold(hold 上一行已被覆蓋) reset = max(prev_reset, prev_sold); ← reset 在這裡才被改

它之所以正確,純粹是因為 reset 在第 3 行才被賦值 —— 第 1 行讀到的 reset 還是昨天的值,剛好等於 prev_reset。結果對,但理由是「順序剛好」,不是「邏輯保證」。

實測 · 把三行「重新排序」就會壞掉

三行都用 prev_* 快照,看起來順序無所謂。但只要把 reset 那行搬到最前面:

　reset = max(prev_reset, prev_sold);　← 搬上來
　hold  = max(prev_hold, reset - prices[i]);　← 現在讀到新的 reset
　sold  = prev_hold + prices[i];

hold 就變成能從「昨天剛賣的錢」買進 —— 冷凍期直接消失。實測:

　[1,2,3,0,2]　原版 3　→　重排後 4　← 這是 LC 122 的答案
　[6,1,3,2,4,7]　原版 6　→　重排後 7
　[1,4,2,8]　原版 7　→　重排後 9

修法:第 1 行寫成 prev_reset。那三個 prev_* 快照的意義就是「這一輪只准讀昨天」—— 既然抓了,就全部都用,不要有一個漏網。這樣三行的順序才真的無所謂,別人重排也不會踩雷。

和 518 的迴圈交換是同一種病:一個看起來無害的順序調整, 安靜地把題目換掉(3 → 4 = 有冷凍期 → 沒冷凍期)。 DP 最危險的 bug 不是會崩的那種,是跑得出漂亮答案、但答的是別題那種。

複雜度分析// COMPLEXITY

寫法TimeSpace備註
A · 三陣列O(n)O(n)好讀、好 debug,可印出整張表
B · 三變數O(n)O(1)follow-up 要的;每天 3 次 max
T = O(n)　　S = O(1)(寫法 B)

n ≤ 5000,兩種都輕鬆過。能壓成 O(1) 的唯一理由是「每個狀態只依賴 i−1」—— 如果冷凍期變成 k 天,就得留 k 天的歷史,空間變 O(k)。

邊界與陷阱// PITFALLS

陷阱症狀正解
hold 從 sold 買 冷凍期消失,答案偏大。[1,2,3,0,2] → 4 而非 3(= LC 122,實測 505 筆完全吻合) hold[i] 只能讀 rest[i−1]
重排 B 版三行 同上 —— 因為第 1 行讀的是 reset 不是 prev_reset 三行一律用 prev_*
答案含 hold max(hold,sold,rest) 多此一舉 —— hold 扣了成本,不可能最大 max(sold[n−1], rest[n−1])
sold 寫成 max(...) 誤以為「今天可以選擇不賣」。但 sold 的定義就是「今天賣了」,不賣屬於別的狀態 sold[i] = hold[i−1] + p[i],沒有 max
n = 1 以為要特判 不用。迴圈不跑,回 max(sold[0], rest[0]) = 0 ✓
n = 0 prices[0] 越界。但題目保證 n ≥ 1 面試時口頭提一句就好,不必真的寫
