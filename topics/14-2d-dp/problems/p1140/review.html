重點回顧// RECAP

面向要點
狀態dp[i][m] = 我從 piles[i..]、參數 M=m 能拿的最多石頭
後綴和suffixSum[i] = piles[i..] 總和,O(1) 取「剩下總共」
恆等式我的 = suffixSum[i] − dp[i+x][max(m,x)]
basei + 2m ≥ n → 全拿 → suffixSum[i]
general枚舉 x=1..2m,取 max
參數更新對手接手 max(m, x);多拿養大對手
複雜度O(n³) 時間 / O(n²) 空間

觀念題// QUIZ

Q1. 解釋 suffixSum[i] − dp[i+x][max(m,x)] 為什麼就是我拿到的石頭。「我拿的 x 堆」跑哪去了?
Q2. 為什麼 base(i + 2m ≥ n)是必要的?少了它會怎樣?它同時扮演什麼角色?
Q3. 對手接手的參數為什麼是 max(m, x) 而不是 x?若只寫 x 會低估還是高估對手?
Q4. 為什麼不能貪心「每次拿到當下最多石頭」?用 dp[0][1] 拿 1 堆得 10、拿 2 堆得 9 說明。
Q5. 這題和 877 都用「我的 = 總 − 對手」的減號。兩題的狀態維度差在哪?為什麼 1140 多一維、877 不用?

範例 Trace// [1,2,3,4,5,100] → 104

// 直覺:那個 100 是關鍵,誰拿到誰贏 piles = [ 1, 2, 3, 4, 5, 100 ] suffixSum = [115,114,112,109,105,100 ] 答案 dp[0][1] = 104 Alice 最優策略不是一開始就衝 —— 她先只拿 1 堆(piles[0]=1), 把對手困在「拿不到 100 又不能給 Alice 機會」的局面, 最終 Alice 能收下 100 + 4 = 104,Bob 只拿到 11。 直接搶前面小堆會把 M 養大、讓對手提前搆到 100 → 反而更差
「先忍一手」是這類題的常見最優:少拿讓對手的 M 保持小,把大堆留到自己能穩穩收下的時機。 DP 不需要「看懂」這個策略 —— 它只是枚舉每個 x、算對手最優、取讓自己最大的,最優策略自動浮現在 max 裡。

應用題// APPLY

App 1. 若要回傳的是「Alice 能不能贏」(超過一半)而非「拿多少」,程式怎麼改?會不會有平手問題?
App 2. 這份 O(n³) 的 x 迴圈能不能優化掉?想想 dp[i][m] 隨 m 增大的單調性。
App 3. 若規則改成「每回合固定拿 1 或 2 堆」(M 不變),狀態能不能降回一維?這會退化成哪一類經典題?
// ANSWERS
A1. 把石頭分三段:①我拿的前 x 堆 ②對手接下來拿的 ③我在對手之後又拿回的。從第 i 堆起的所有石頭 = suffixSum[i],最終不是①③(我的)就是②(對手的) —— 沒有第四種歸屬。所以「我的(①+③)= 總 suffixSum[i] − 對手的②」。
對手的② 正是「對手在 piles[i+x..] 當先手能拿的最多」= dp[i+x][max(m,x)]。所以 我的 = suffixSum[i] − dp[i+x][max(m,x)]。
「我拿的 x 堆」沒有消失,它被吸收進 suffixSum[i] 了 —— 因為 suffixSum[i] 本來就包含那 x 堆。這個減法的美妙之處:不用分開記「我已拿多少」,一個後綴和 + 一個減號就把①②③全算清。
A2. base 是 x 能「一路拿到最後一堆」的情形(2m ≥ n − i)。此時最優顯然是全拿,答案是確定值 suffixSum[i],不用再枚舉。
少了它會出兩個問題:(1) general 的迴圈 x=1..2m 會讓 i+x 一路加到超過 n,存取 dp[i+x][…] 越界;(2) 遞迴 / 填表沒有終止的確定值,邏輯不收斂。
所以 base 一石二鳥:既是「顯然全拿」的捷徑,也是遞推的終止條件(邊界)。任何博弈 / 區間 DP 都要先把 base 講清楚 —— 它是整個遞推站得住的地基。你標注「base + general」抓的正是這個重點。
A3. 因為規則是「拿完 X 堆後,M 更新為 max(M, X)」—— M 只會變大或不變,不會變小。我這步拿了 x 堆,對手接手時的參數是「舊 M 和我剛拿的 x 取大」= max(m, x)。
只寫 x 會怎樣:當 x < m 時(我拿得比當前上限少),max(m,x)=m,但你寫成 x 就低估了對手的參數 → 以為對手下一步能拿的堆數比實際少 → 低估對手拿的 → suffixSum[i] − dp[...] 反而高估我的 → 答案偏大、錯。
關鍵直覺:M 是「歷史最大拿取」的單調累積,永不回頭。這也是為什麼狀態要帶 m 這一維 —— 它記錄了這個會影響未來的歷史。
A4. 因為拿的堆數 x 有副作用:把對手的參數推到 max(m, x),對手下一步能拿的上限 2·max(m,x) 跟著變大。眼前多吃,可能讓對手一口氣掃走後面的大堆。
dp[0][1]:拿 1 堆 [2] → 對手 dp[1][1]=16 → 我 26−16=10;拿 2 堆 [2,7] → 對手 dp[2][2]=17(參數被我推到 2,對手能拿更多)→ 我 26−17=9。拿少的反而多 1 分。
貪心「拿當下最多石頭」看不到這個副作用 —— 它以為多拿 7 是賺,實際上是把武器交給對手。只有 DP 把「我這步得分 − 對手因此變強後拿走的」一起算,才找得到真正最優。有副作用的選擇,貪心必敗,要 DP。
A5. 兩題都用「我的 = 總 − 對手的」減號,但狀態維度不同:
● 877 的狀態是 dp[i][j] —— 一個區間 [i,j]。因為兩端都能拿、拿一堆,「還能做什麼」完全由剩哪段決定,不需要額外參數。
● 1140 的狀態是 dp[i][m] —— 一個起點 i + 參數 m。因為只從左端拿(剩下的永遠是一個後綴,用單一 i 就能表示,不用 j),但「這步能拿幾堆」由歷史 M 決定,而 M 會變 → 必須進狀態。
對比很漂亮:877 兩端取用 → 要記「兩個邊界」[i,j];1140 單端取用 → 只要記「一個邊界」i,但規則帶了會變的參數 M → 要記 m。狀態該帶什麼,取決於「未來還能做什麼」由哪些量決定。減號框架不變,維度隨規則走。
App 1. 改動很小:算出 alice = dp[0][1] 後,判 alice * 2 > total(total = suffixSum[0])即 Alice 拿超過一半 → 贏。
平手:若 total 是偶數,理論上可能 alice * 2 == total(剛好一半)。要看題目定義「平手算誰贏」決定用 > 還是 >= —— 和 877 / 486 的 > vs >= 是同一個判斷。原題 1140 只問「Alice 最多拿多少」,回傳 dp[0][1] 這個數,根本不用碰勝負判定,是最乾淨的形式。
App 2. 可以優化掉 x 迴圈,把 O(n³) 降到 O(n²)。觀察:dp[i][m] 對固定 i 隨 m 單調不減(能拿的上限變大,絕不會更差)。而 general 的轉移 max over x 中,dp[i+x][max(m,x)] 這一項可以拆成「x ≤ m(參數仍是 m)」和「x > m」兩段,配合後綴 min/max 預處理把內層 x 迴圈省掉。
實務上 n ≤ 100,O(n³) = 10⁶ 早就夠快,面試講得出這個單調性和優化方向即可,不必真寫 —— 展示你看到了狀態的結構,而不是只會套三層迴圈。
App 3. 若 M 固定(每回合就是拿 1 或 2 堆,不會變),那 m 這一維消失,狀態降回一維 dp[i] = 「我從 piles[i..] 能拿的最多」,轉移 dp[i] = max_{x=1,2}(suffixSum[i] − dp[i+x])。
這就退化成經典的「拿 1 或 2 步的博弈 / 樓梯」型一維 DP(和「石子合併」「跳台階博弈」同構)。
啟示:1140 的難點全在那個會變的 M —— 是它逼出第二維、逼出 max(m,x) 的參數更新、逼出「多拿養大對手」的權衡。規則裡任何「會隨遊戲改變、又影響未來」的量,都得進狀態。拿掉它,題目立刻塌回一維。
