範例 Trace// coins=[1,2], amount=4 —— 最小可辨例

// A · 硬幣外層(正解) 金額: 0 1 2 3 4 初始 [ 1, 0, 0, 0, 0 ] coin=1 (i=1→4) [ 1, 1, 1, 1, 1 ] coin=2 (i=2→4) [ 1, 1, 2, 2, 3 ] return dp[4] = 3 → {1,1,1,1} {1,1,2} {2,2}
// B · 金額外層(交換後) 金額: 0 1 2 3 4 [ 1, 1, 2, 3, 5 ] dp[2] = dp[1] + dp[0] = 2 ← 1+1、2 dp[3] = dp[2] + dp[1] = 3 ← 1+1+1、1+2、2+1 (1+2 和 2+1 重複!) dp[4] = dp[3] + dp[2] = 5 ← 1+1+1+1、1+1+2、1+2+1、2+1+1、2+2 return dp[4] = 5 → 多了 1+2+1、2+1+1(都和 1+1+2 是同一個組合)
3 vs 5 —— 這是最小的能一眼看穿的例子。 寫完 code 之後拿 coins=[1,2]、amount=4 跑一次:得 3 就是 518,得 5 就是 377。 比背口訣可靠得多。

應用題// APPLY

App 1. 若每種硬幣最多只能用 3 枚(有界背包),程式要怎麼改?有沒有比「把每種硬幣複製 3 份」更好的做法?
App 2. 若要輸出其中一種組合(不只是數量),要改什麼?若要輸出全部組合呢?複雜度會變怎樣?
App 3. 面試官說「amount 可以到 10⁹,但 coins.length ≤ 3」。背包還能用嗎?該往哪個方向想?
// ANSWERS
A1. 答案會從 4 變成 9(coins=[1,2,5]、amount=5)。它不再算組合,而是算排列。
正確的 4 種組合:{5}、{1,1,1,1,1}、{1,1,1,2}、{1,2,2}。
交換後多出來的 5 個,是把後兩種的不同順序當成不同答案:{1,1,1,2} 的 2 可以在 4 個位置 → 算 4 種;{1,2,2} 的 1 可以在 3 個位置 → 算 3 種。合計 1 + 1 + 4 + 3 = 9。
這不是 bug 是換題:9 正是 LC 377 組合總和 IV 的正解(那題雖然叫「組合」,要的其實是排列)。同一份 code,外層放誰,決定你在解哪一題。
A2. 機制在於dp[i] 的語意會隨著輪次演進:跑完第 k 種硬幣那一輪,dp[i] 的意思是「只用前 k 種硬幣湊出 i 的組合數」。這其實是二維 dp[k][i] 滾掉第一維的結果。
因為硬幣是依固定索引順序一輪一輪進表的,當你在 coin=2 那輪加上一枚 2,它只能接在「1 已經處理完」的狀態後面 —— 程式沒有任何路徑能產生「2 排在 1 前面」的寫法。於是每個組合被綁定到唯一一種標準順序(硬幣索引由小到大),只會被數到一次。
反之,金額外層時 dp[i] 的語意是「湊出 i 的序列數」——每個 i 都讓每一枚硬幣當一次「最後一枚」,1+2 和 2+1 是兩條不同的路,兩條都算。去重不是靠程式碼,是靠「固定考慮順序」這件事本身。
A3. 會變成 0/1 背包 —— 每種硬幣最多拿一枚。[1,2,5]/5 實測會得到 1(只剩 {5};1+2=3、1+2+5=8,都湊不出 5)。
原因:算 dp[i] 時要讀 dp[i−coin]。遞增時 dp[i−coin] 在同一輪已經更新過,值裡已經含有「用了本輪這枚硬幣」的情況 → 再加一次就是拿第二枚、第三枚 → 完全背包。遞減時 dp[i−coin] 還是上一輪的舊值 —— 乾淨的「沒用過本輪硬幣」狀態 → 每個最多用一次 → 0/1 背包。
這題硬幣是「無限多枚」,所以必須遞增。
A4. 因為 min 對順序免疫,計數(+)不免疫。
322 求「最少幾枚」。同一個組合 {1,2} 不管以 1+2 還是 2+1 的路徑被算到,得到的枚數都是 2 —— 重複計算只是把同一個值再取一次 min,min(2,2)=2,不影響結果。min 是冪等的。
518 求「幾種」。同一個組合被走到兩次,+ 就會老老實實加兩次 → 答案膨脹。加法不是冪等的。
通則:最佳化型(min/max)不怕重複走,計數型(+)怕。所以 322 外層放誰都對、518 只能放硬幣。這也解釋了 139 的 OR(也是冪等)為什麼另有原因不能把字典放外層 —— 那題是「有序切割」,連可行性都會被順序改變,是更強的限制。
A5. 因為題目保證的是「final answer 放得進 int」,沒有保證中間值。
實測反例(完全合法):coins=[2,4,6,…,300]、amount=4999。硬幣全偶、金額為奇 → dp[4999]=0,答案完全符合保證;但過程中 dp[4998] 是天文數字,5000 格裡有 2378 格超過 INT_MAX。用 int 寫,那些 += 就是有號溢位 = UB —— 拿 -fsanitize=signed-integer-overflow 一跑當場報 runtime error: signed integer overflow。unsigned int 的溢位則是定義良好的 mod 2³²,不是 UB。所以這裡的 unsigned 是真的有用。
494 則相反:答案是 ± 組合數,總共才 2ⁿ 種,n ≤ 20 ⇒ 答案和每個中間值都 ≤ 2²⁰ ≈ 10⁶,連 INT_MAX 的 0.1% 都不到 → unsigned 純屬多餘。
方法論(三題三種):377 = 答案本身溢位(unsigned 必要)· 518 = 只有中間值溢位(unsigned 好判斷)· 494 = 都不溢位(unsigned 多餘)。永遠先問兩句:答案的上界?中間值的上界?
App 1. 這是有界背包(bounded knapsack)。
最笨但可行:把每種硬幣複製 3 份當成 3 個不同物品,跑 0/1 背包(內層遞減)。時間 O(3n·amount) —— 能用,但複製份數大時會爛掉。
更好:加一維數量 dp[i][j] = 湊出 i 且本種硬幣用了 j 枚,或直接對每種硬幣做「枚數 0..3」的轉移:dp_new[i] = Σ_{t=0..3} dp_old[i − t·coin]。
最好:當上限 K 很大時用二進位拆分 —— 把 K 枚拆成 1,2,4,… 份物品(O(log K) 個),再跑 0/1 背包,可組出 0..K 任意枚數。時間 O(n·log K·amount)。
App 2. 輸出一種:一維表丟失了歷史,要保留二維 dp[k][i],從 (n, amount) 回溯 —— 在每格判斷這個組合是「不用第 k 種硬幣」(退到 dp[k−1][i])還是「用了一枚」(退到 dp[k][i−coin]),記下決策直到 (0,0)。O(n + amount) 回溯。
輸出全部:那就不能只靠 DP —— 輸出量本身就是 O(組合數 × 平均長度),而組合數可以是指數級。直接 DFS 枚舉即可(用「硬幣索引不回頭」來保證只產生組合、不產生排列 —— 和迴圈外層放硬幣是同一個道理)。計數比列舉容易,這是 DP 的典型價值。
App 3. 背包不能用 —— O(n·amount) 是偽多項式,amount = 10⁹ 表根本開不出來。但 coins.length ≤ 3 是強烈暗示:換成數學。
3 種硬幣 a,b,c 求 ax+by+cz = amount 的非負整數解個數:枚舉 x(0..amount/a)還是太多,所以要用兩層化簡 —— 固定 z 後解二元 ax+by = R,那是線性丟番圖方程,解數有閉式(用擴展歐幾里得找一組特解,再數區間內的解個數)。整體可壓到 O(amount / c) 甚至更好。
再一般化,k 種固定面額的組合數對 amount 是個準多項式(Ehrhart / Popoviciu 定理),理論上有閉式。判斷準則:amount 大而 n 小 → 數論/閉式;amount 小而 n 大 → 背包。和 494 的 meet-in-the-middle 是同一種思路 —— 當偽多項式的那一維爆掉,就要換一個維度下手。
