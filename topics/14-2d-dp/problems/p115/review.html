範例 Trace// "babgbag" × "bag" → 5

// 完整表(實跑輸出) "" b a g "" 1 0 0 0 b 1 1 0 0 a 1 1 1 0 b 1 2 1 0 g 1 2 1 1 b 1 3 1 1 a 1 3 4 1 g 1 3 4 5 return dp[7][3] = 5
// 答案那一步 dp[7][3] s[6]='g' 配 t[2]='g': dp[7][3] = 上 dp[6][3](1) + 左上 dp[6][2](4) = 5 左上 dp[6][2]=4 = 「湊出 ba 有 4 種」(3 個 b 各接倒數第 2 個 a,再加…) 上 dp[6][3]=1 = 「不用最後這個 g,前面已湊出的 bag」= 第一個 g 收尾那條 → 4 + 1 = 5
// 手動列出 5 種 "bag"(用 s 的 index 標記) s = b a b g b a g 0 1 2 3 4 5 6 1) b0 a1 g3 2) b0 a5 g6 3) b2 a5 g6 4) b4 a5 g6 5) b0 a1 g6 5 種不同的 index 組合 = dp 表算出的 5,一個不多一個不少
DP 數的是「不同的 index 挑法」,不是「不同的字串」—— 這 5 種挑出來都是字串 "bag",但用到的位置不同,所以算 5 種。 「distinct subsequences」的 distinct 指的是子序列(位置集合)相異,不是字串相異。

應用題// APPLY

App 1. 若要的不是「幾種」而是「存不存在 t 為 s 的子序列」(LC 392),還需要 DP 嗎?最省的做法是什麼、複雜度多少?
App 2. 若允許 t 裡出現萬用字元 ?(配任意單一字元),轉移要怎麼改?若是 *(配任意長度)呢?這會把題目推向哪一題?
App 3. 若要真的印出其中一種挑法(而非只數數量),二維表夠嗎?能不能只用 O(n) 空間做到?
// ANSWERS
A1. 兩項是對 s[i−1] 的兩種處置:
● dp[i−1][j−1] = 「拿」—— 用 s[i−1] 去配掉 t[j−1],兩邊各退一格,方法數繼承自「更短的 s 配更短的 t」。
● dp[i−1][j] = 「不拿」—— 跳過 s[i−1],t 目標不變,用 s 前面的字元去湊 t 的前 j 字。
為什麼相等了還要算「不拿」?因為即使 s[i−1] 能 配 t[j−1],你不一定要用它 —— 也許改用 s 前面另一個相同字元,那是不同的子序列,得分開數。rabbbit 的三個 b:每多一個 b,「拿新的」和「用舊的」都成立,方法數就 1→2→3 疊上去。漏掉「不拿」= 把不同挑法當成同一種,直接少算。
A2. 根源是兩個字串的地位不對等。
115 把 t 當成固定目標去 s 裡數:t 的每個字都必須被配掉,不能跳過;只有 s 可以跳。所以「跳過某字」只有一種方向 ——「跳 s」,對應 dp[i−1][j](上)。「跳 t」不存在,所以沒有「左」dp[i][j−1]。
LCS 兩個字串對等:找的是共同子序列,兩邊都可以跳過字元。所以不等時要比較「跳 s(上)」和「跳 t(左)」哪個更長 → max(dp[i−1][j], dp[i][j−1])。
一句話:誰能被跳過,決定你有幾個來源。115 只有 s 能跳 → 只有上;LCS 兩邊能跳 → 上和左都要看。這也是為什麼 115 的表不對稱(換 s、t 答案會變),LCS 對稱。
A3. 取 s = 35 個 'a'、t = 30 個 'a' —— m,n ≤ 1000,完全合法。全同字元時 dp[i][j] = C(i,j)。
● 最終答案 dp[35][30] = C(35,30) = C(35,5) = 324,632 → 放得進 int32,題目的保證成立。
● 但中間 dp[35][17] = C(35,17) = 4,537,567,650 → 超過 INT_MAX(2.1×10⁹)。整張表有 11 格爆掉。
原因:中間欄數的是較短的 t 前綴,二項式係數在正中間 j≈m/2 最大,答案落在遞減側反而較小。
用 int,那些 + 是有號溢位 = UB,-fsanitize=signed-integer-overflow 實測當場報 signed integer overflow: 1166803110 + 1037158320。所以 unsigned long long 是對的、必要的 —— 和 518 同一個道理:題目只保證答案,沒保證中間值。
A4. dp[i][0] = 「s 前 i 字裡,有幾個子序列等於空字串」。空字串的湊法只有一種:什麼都不選 —— 不管 s 多長都成立,所以整欄都是 1。只設 dp[0][0]=1 會讓 dp[1][0]…dp[m][0] 停在 0,而它們是每一列轉移的左上來源,一旦是 0,配對格永遠加不進「拿」那條,整張表崩掉。
dp[0][j](j≥1)= 「空的 s 裡有幾個子序列等於非空 t」= 0(空字串生不出非空目標)。這由 vector 初始化的 0 自動成立,不用手設。兩個邊界合起來:空 t 給 1、空 s 給 0,計數的種子就齊了。
A5. 內層 j 要由大到小(右往左)掃。
壓一維時 dp[j] 同時扮演「這一列的新值」和「上一列的舊值」。轉移要用 左上 dp[i−1][j−1],也就是「上一列、左邊一格」。若 j 由小到大,dp[j−1] 在算 dp[j] 之前已經被本列覆蓋,左上就丟了。由大到小掃,算 dp[j] 時 dp[j−1] 還是舊值(左上完好),更新完 dp[j] 才輪到更小的 j。
(或者仿 1143 保留一個 prev 變數存左上,那樣左往右也行。)
和 97 的差別:97 交錯字串只依賴上和左,沒有左上,所以壓一維時無腦左往右就對、不用 prev。要不要小心掃描方向,取決於轉移有沒有碰「左上」。
App 1. 不需要 DP。「存不存在」是可行性,不用數方法,用雙指標貪心:i 掃 s、j 掃 t,s[i]==t[j] 就 j++,最後看 j 是否走完 t。時間 O(m)、空間 O(1)。
貪心正確的原因:配 t 的某個字時,選最早能配的 s 位置永不吃虧(留最多後路給後面的字)—— 這是子序列匹配的經典交換論證。
對比很有啟發:「有沒有」O(m) 貪心就夠,「有幾個」才要 O(mn) DP。計數把問題變難了一個量級 —— 這也是 392 是 Easy、115 是 Hard 的原因。
App 2. ?(配任意單一字元):把「字元相等」的條件放寬成 s[i−1]==t[j−1] || t[j−1]=='?',轉移完全不變。因為 ? 一定「用掉一個 s 字、消一個 t 字」,和普通配對的結構一樣。
*(配任意長度,含空):結構就變了 —— * 可以吃 0 個或多個 s 字。轉移要加「* 不吃(dp[i][j−1])」和「* 多吃一個(dp[i−1][j])」兩條。這正是把題目推向 LC 44 萬用字元匹配 / LC 10 正則匹配 的家族 —— 從「計數」轉成「可行性」,但格子仍是 dp[i][j]、仍看上/左/左上的組合,只是聚合換成 OR、來源隨萬用字元語意增加。整個兩字串 DP 家族共用同一張表,差別只在「每格怎麼合」。
App 3. 印一種挑法要保留完整二維表(O(mn) 空間)—— 一維滾動丟了歷史,無法回溯。
回溯法:從 (m, n) 往回走。在 (i, j):若 s[i−1]==t[j−1] 且 dp[i−1][j−1] > 0,可以「拿」——記下 s 的 index i−1,退到 (i−1, j−1);否則(或你想要另一種)走「不拿」退到 (i−1, j)。走到 j==0 就得到一組完整的 index。O(m) 步。
能不能只用 O(n) 空間還印得出來?可以,但要用分治回溯(Hirschberg 的手法):在中間切一刀,兩半各算一次一維 DP 決定分界點,再遞迴 —— 時間仍 O(mn)、空間降到 O(n)。這是「計數只需 O(n)、但輸出方案通常要 O(mn) 或更巧的分治」的經典取捨,和 LCS 要印出實際子序列時遇到的問題一模一樣。
