class Solution {
public:
    int numDistinct(string s, string t) {
        int m = s.size(), n = t.size();
        // dp[i][j] = s 前 i 字裡,有幾個子序列 = t 前 j 字
        vector<vector<unsigned long long>> dp(m + 1, vector<unsigned long long>(n + 1, 0));

        // base:空的 t 永遠有 1 種湊法(什麼都不選)
        for (int i = 0; i <= m; ++i) {
            dp[i][0] = 1;
        }

        for (int i = 1; i <= m; ++i) {
            for (int j = 1; j <= n; ++j) {
                if (s[i - 1] == t[j - 1]) {
                    // 拿 s[i-1] 配 t[j-1] → dp[i-1][j-1];不拿 → dp[i-1][j]
                    dp[i][j] = dp[i - 1][j] + dp[i - 1][j - 1];
                } else {
                    // 字元不合,只能跳過 s[i-1]
                    dp[i][j] = dp[i - 1][j];
                }
            }
        }
        return dp[m][n];
    }
};

注意轉移只讀上一列(i−1)—— 每格看「正上方」和「左上方」兩格。這是子序列計數的標準形狀: 相等時把兩條路的方法數相加,不等時照抄上方。

為什麼是 unsigned long long// LOAD-BEARING

題目寫「答案保證放得進 32-bit signed int」。那為什麼你還用 unsigned long long? 用對了 —— 而且理由和 518 是同一個:題目只保證最終答案,沒保證中間格子。

實測 · 一個「答案合法、中間卻爆 int」的合法輸入

取 s = 35 個 'a'、t = 30 個 'a'(m,n ≤ 1000,完全合法)。 全同字元時 dp[i][j] = C(i, j)(二項式係數):

　最終答案 dp[35][30] = C(35,30) = C(35,5) = 324,632　← 舒舒服服放進 int32
　中間格子 dp[35][17] = C(35,17) = 4,537,567,650　← 超過 INT_MAX(2.1×10⁹)

為什麼中間會比答案大?因為中間欄 j 數的是「較短的 t 前綴」,而二項式係數在正中間(j≈m/2)最大 —— 答案落在遞減側,反而比中間小。這張表裡有 11 格超過 INT_MAX。

用 int 跑,那幾格的 + 就是有號溢位 = UB。-fsanitize=signed-integer-overflow 當場抓到: runtime error: signed integer overflow: 1166803110 + 1037158320 cannot be represented in type 'int'。 換 unsigned long long 就徹底沒事(而且遠遠用不到它 64 位的上限)。

題目答案溢位?中間值溢位?該用什麼
377會會unsigned(要 mod 2³² 的值)
518不會會unsigned(避 UB)

永遠問兩句:「答案的上界?」(決定回傳型別)+「中間值的上界?」(決定計算會不會 UB)。 115 和 518 都是只有第二句才咬到的那種 —— 也是最容易被「題目說答案放得下」這句話騙過去的那種。
