重點回顧// RECAP

面向要點
狀態dp[i][j] = s1 前 i + s2 前 j 能否交錯成 s3 前 i+j
s3 索引k = i+j−1(第三維被綁死 → 消掉)
拿 s1(上)dp[i−1][j] && s1[i−1]==s3[k]
拿 s2(左)dp[i][j−1] && s2[j−1]==s3[k]
合併兩者 OR;先擋 m+n != s3.size()
壓維只靠上/左 → O(n),不需 prev

範例 Trace// "aa" × "ab" → "aaba"

// 邊界(第一列/第一行 = 只用單一字串) dp[0][0] = T dp[0][1]: s2[0]='a' == s3[0]='a' ✓ → T dp[0][2]: dp[0][1]=T 但 s2[1]='b' ≠ s3[1]='a' → F dp[1][0]: s1[0]='a' == s3[0]='a' ✓ → T dp[2][0]: dp[1][0]=T 且 s1[1]='a' == s3[1]='a' ✓ → T
// 內部格(每格 k = i+j−1) dp[1][1]: k=1, s3[1]='a' 上 dp[0][1]=T 且 s1[0]='a'=='a' ✓ → T dp[1][2]: k=2, s3[2]='b' 上 dp[0][2]=F ✗ ; 左 dp[1][1]=T 且 s2[1]='b'=='b' ✓ → T dp[2][1]: k=2, s3[2]='b' 上 dp[1][1]=T 但 s1[1]='a'≠'b' ✗ 左 dp[2][0]=T 但 s2[0]='a'≠'b' ✗ → F dp[2][2]: k=3, s3[3]='a' 上 dp[1][2]=T 且 s1[1]='a'=='a' ✓ → T
// 完成表 ∅ a b ∅ T T F a T T T a T F T return dp[2][2] = true　　取法(回溯):s2'a' · s1'a' · s2'b' · s1'a' → "aaba"
注意 dp[2][1] = F 卻不影響答案 —— 死路可以存在,只要有一條路通到右下角就是 true。 有重複字元(兩個 a)時「該從 s1 還是 s2 拿」無法貪心決定,OR 讓兩種可能同時活著,這正是要 DP 的理由。

應用題// APPLY

App 1. 若要回報一種實際的交錯方式(哪些字取自 s1、哪些取自 s2),怎麼從 dp 回溯?
App 2. 若把「能否」改成「有幾種交錯方式能組出 s3」,DP 要怎麼改?聚合換成什麼?
App 3. 若改成三個字串交錯成 s4,狀態要幾維?第幾維可以省掉?複雜度變多少?
// ANSWERS
A1. 因為交錯的定義要求每個字都要用掉、且都填進 s3,沒有「跳過不用」這回事。所以「用了 i 個 s1 字 + j 個 s2 字」必然剛好填滿 s3 的前 i+j 位 —— 下一個待配的字元固定是 s3[i+j−1](0-indexed)。第三維是前兩維的函數(k = i+j),沒有額外資訊,可以直接消掉 → 二維就夠。這是 DP 降維的常見手法:發現某維可由其他維算出,就砍掉它。
A2. 因為「字元對上」只保證這一步能接,不保證前面真的能交錯出來。dp[i−1][j] 為 true 才代表「用 s1 前 i−1 字 + s2 前 j 字確實能組出 s3 前 i+j−1 字」;在那個基礎上再接一個對得上的 s1[i−1],才真的成立。只看字元相等,會把「前面根本接不起來」的死路誤判成可行 —— 例如某條路早就在中間卡住了,後面字元湊巧對上也沒用。兩個條件必須 &&。
A3. 「上」dp[i−1][j] = 「s3 的這個字從 s1 拿」(s1 多用一個字、s2 不變);「左」dp[i][j−1] = 「從 s2 拿」。這是兩種互斥但都可能的走法 —— 只要其中一種能成立,dp[i][j] 就是 true,所以用 OR。用 AND 會變成「兩種都要成立」,那是錯的語意(s3 的某個字不可能同時來自 s1 和 s2)。可行性 DP 的分支是 OR,不是 AND。
A4. 因為交錯只依賴「上」dp[i−1][j] 與「左」dp[i][j−1],沒有斜對角 dp[i−1][j−1]。一維滾動裡:dp[j](未更新)天然就是「上」、dp[j−1](已更新)天然就是「左」—— 兩個都能直接取到,不會被覆蓋掉。1143 LCS 相同字元時要用斜對角,而 dp[j−1] 已被本列覆蓋,所以必須用 prev 先存住舊值。要不要 prev,就看轉移有沒有用到斜對角。
A5. 那是在更新第一行(j=0,完全不用 s2、只用 s1 的情形):dp[0] 代表 dp[i][0] = 「s1 前 i 字能否單獨組成 s3 前 i 字」。它只有「上」一個來源(左邊界外),所以是 舊 dp[0](= dp[i−1][0]) && s1[i−1]==s3[i−1]。拿掉的話,dp[0] 會一直停在上一列的值(甚至永遠是 true),讓「只用 s1 其實接不下去」的情形被誤判可行,整列都跟著錯。二維版對應的就是那個獨立的 dp[i][0] 邊界迴圈。
App 1. 從 (m,n) 往回走:在 (i,j) 檢查兩個來源 —— 若 dp[i−1][j] && s1[i−1]==s3[i+j−1] 成立,代表這個字取自 s1,記下並退到 (i−1,j);否則(必然是 dp[i][j−1] && s2[j−1]==s3[i+j−1])取自 s2,退到 (i,j−1)。一路退到 (0,0),把記錄反轉就是一種取法。若兩個來源都成立,任選一個即可(有多種交錯方式)。要回溯必須保留二維表(一維滾動丟了歷史)。
App 2. 把布林換成計數:dp[i][j] = 交錯方式數,轉移從 OR 換成加法:dp[i][j] = (s1[i−1]==s3[k] ? dp[i−1][j] : 0) + (s2[j−1]==s3[k] ? dp[i][j−1] : 0),base dp[0][0]=1。這就是「可行性用 OR、計數用 +」的老規矩(和 62/63 的網格計數同源)。注意方法數可能很大,要留意溢位或取模。
App 3. 三個字串交錯成 s4:狀態 dp[i][j][k] = s1 前 i、s2 前 j、s3 前 k 能否交錯成 s4 前 i+j+k 字 —— 三維。而「s4 的位置」第四維一樣可以省掉(同 A1 的理由:= i+j+k)。轉移是三種來源 OR(從 s1/s2/s3 各拿一字)。複雜度 O(|s1|·|s2|·|s3|),空間可壓成 O(|s2|·|s3|)(滾掉第一維)。規律:交錯 k 個字串 → k 維狀態,「目標字串位置」永遠是那個可省的維度。
