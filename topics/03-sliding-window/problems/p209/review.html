觀念題// CONCEPT Q

Q1

求最短為什麼是「達標就縮」,而求最長(3/424)是「違規才縮」?一句話說出兩者收縮時機為何相反。

Q2

為什麼要先更新 minL 再縮左?順序反過來會漏掉什麼?

Q3

這個滑動視窗解為什麼依賴「全為正數」?若陣列含負數或零,哪一步會失效?該改用什麼?

Q4

時間是 O(n),但裡面有內層 while 收縮,看起來像巢狀。為什麼不是 O(n²)?

Q5

Follow-up 的 O(n log n) 前綴和 + 二分:為什麼「正數」讓前綴和可以二分?對每個右界要找什麼樣的左界?

3

範例 Trace// 全部加起來都不夠 → 0

// target = 11, nums = [1,1,1,1,1,1,1,1](總和 8 < 11) r=0..7: total 一路累加到 8,始終 < 11 → while 從不進入 minL 從未更新,維持 INT_MAX 掃完 → minL == INT_MAX → 回傳 0 0(整段和 8 < target 11,不存在合格子陣列)
這個 case 展示「不存在」的處理:整段總和都不到 target 時,while (total >= target) 一次也沒進、minL 保持 INT_MAX,最後轉回 0。 用 INT_MAX 當「還沒找到」的哨兵、最後判斷轉 0,是求最小值題的標準寫法 —— 別忘了這個收尾判斷。

應用題// APPLY

App 1

LC 76 最小覆蓋子字串:求 s 中包含 t 所有字元的最短子字串。它也是「求最短、達標就縮」,合格條件變成什麼?怎麼 O(1) 判斷「已覆蓋」?

App 2

LC 862 含負數版:允許負數,求和 ≥ k 的最短子陣列。為什麼滑動視窗直接壞掉?單調隊列怎麼救?

App 3

恰好等於 target / 求最長:若改成「和恰好 = target 的最短」或「和 ≤ target 的最長」,滑動視窗還適用嗎?各要怎麼調?

// ANSWERS
A1

因為目標相反,所以收縮的目的相反。求最長:視窗越長越好,只有當它不合法時才被迫縮回合法 —— 縮是「止損」。求最短:視窗一旦合格就已經是候選答案,但我們想更短,所以趁著還合格拚命縮,直到再縮就不合格 —— 縮是「進取」。一句話:最長「違規才縮(恢復合法)」,最短「達標就縮(逼出更短)」。這是可變長視窗兩大類的分水嶺。

A2

因為當前這個合格視窗 [l, r] 本身可能就是最短。若先縮再記,你就把 [l, r] 跳過、只在縮完後才量長度,而縮掉一格後 total 可能已 < target(不再合格),這個更短但不合格的視窗不能算 —— 於是漏掉了 [l, r] 這個真正的候選。正確順序:在還合格時先 minL = min(minL, r-l+1),再 total -= nums[l++]。

A3

滑動視窗依賴一個單調性:「縮左(移出 nums[l])一定讓 total 變小」,這樣「達標就縮、跌破就停」才有意義、每個元素只進出視窗一次。全為正數保證這點。若有負數或零:縮左可能不減反增或不變,「達標→縮→跌破」的邏輯崩潰,而且擴張時 total 也非單調,無法用「窗一達標就縮」。此時要改用前綴和 + 單調隊列(LC 862),維護前綴和的單調遞增隊列來找最短區間。

A4

O(n)。關鍵是攤還分析:l 只會往右移,整個過程最多移動 n 次 —— 內層 while 的總執行次數跨越整個外層加起來也 ≤ n(每個元素最多被 l 移出一次)。所以總指針移動 = r 的 n 次 + l 的 ≤ n 次 = O(n)。它不是「每個 r 都重掃一遍」的巢狀,而是兩個指針各單向走一趟。

A5

因為全為正數 → 前綴和 pre 嚴格遞增(每加一個正數就變大),遞增數列才能二分。子陣列 [l, r-1] 的和 = pre[r] - pre[l]。要它 ≥ target,即 pre[l] ≤ pre[r] - target;要子陣列最短,就要 l 最大(最靠右)。於是對每個右界 r,在遞增的 pre[0..r-1] 裡用 upper_bound 找最大的、滿足 pre[l] ≤ pre[r]-target 的 l,更新最短長度。每個右界 O(log n),共 O(n log n)。

App 1

LC 76 也是「求最短、達標就縮」,只是「合格」= 視窗涵蓋了 t 的所有字元(含重複次數)。用 need[26/128] 記 t 每字元還差幾個,再維護一個計數 missing = 還沒湊齊的字元數;右擴張時若某字元湊到需求就 missing--。missing == 0 即已覆蓋(O(1) 判斷),此時進入「達標就縮」:縮左、每次仍覆蓋就更新最短,直到縮掉某個必要字元 missing 變正才停。骨架和本題完全一樣,只是「達標」從「和 ≥ target」換成「覆蓋 t」。

App 2

LC 862:含負數時,擴張不再讓和單調增、縮左不再讓和單調減,「達標就縮」失去依據。改用前綴和 pre + 單調遞增雙端隊列:對每個右界 r,(1) 從隊頭彈出所有 pre[r] - pre[deque.front()] ≥ k 的左界並更新最短(它們之後不會有更近的右界更優);(2) 從隊尾彈出 pre[deque.back()] ≥ pre[r] 的(它們比 pre[r] 大又更左,永遠不如 r),再把 r 推入。維持隊列裡前綴和遞增,總 O(n)。這是滑動視窗在「有負數」時的正確推廣。

App 3

和 ≤ target 的最長(正數):可以!這是「求最長、違規才縮」——擴張,當 total > target(違規)就縮左到 ≤ target,取 max。和恰好 = target 的最短/計數:正數時仍可滑動視窗(達標就縮,但要精確等於較 tricky);更通用是前綴和 + 雜湊記 pre 出現位置,找 pre[r] - pre[l] == target(這招對含負數也行,如 560)。重點:「≥/≤ + 正數」→ 滑動視窗;「恰好 = / 含負數」→ 前綴和 + 雜湊/單調隊列。
