題目簡介// THE QUESTION

QUESTION

給正整數陣列 nums 和正整數 target,求「和 ≥ target」的最短連續子陣列長度;不存在回傳 0。 例:target=7, nums=[2,3,1,2,4,3] → 2(子陣列 [4,3]);target=8, nums=[1,4,4] → 2。

這是可變長視窗的「求最短」面貌,和 3、424 的「求最長」正好收縮時機相反: 求最長是「違規才縮」;求最短是「達標就縮」—— 右指針擴張累加,一旦 sum ≥ target(已合格),就拚命縮左看能不能更短,每次仍合格就更新最短。全部元素為正是這招成立的關鍵前提。

2

為什麼「達標就縮」的可變長視窗// WHY

× REJECT · O(n²)
枚舉所有子陣列

試每個起點往後累加到 ≥ target。n 到 10⁵ → 平方級,超時。

△ O(n log n)
前綴和 + 二分

前綴和遞增(正數),對每個右界二分找最短左界。O(n log n),是follow-up 解;但滑動視窗更快。

SHRINK-WHILE-VALID · ADOPT ↓
● ADOPT · O(n) / O(1)
達標就縮左、取最短

右指針 r 擴張、total += nums[r];當 total ≥ target,進入 while:更新最短 minL、total -= nums[l++],一直縮到 total < target。 因為全是正數,縮左一定讓和變小,收縮方向單調可控 → 兩指針各走一趟 O(n)、O(1) 空間。

3

核心策略 · 三步驟// STRATEGY

  1. 擴張累加 —— r 前進,total += nums[r]。
  2. 達標就縮 —— while (total ≥ target):更新 minL = min(minL, r-l+1),再 total -= nums[l++]。
  3. 回傳 —— 掃完 minL 若沒被更新過(仍 INT_MAX)回 0,否則回 minL。

動畫演示// SHRINK WHILE VALID

nums=[1,4,4], target=8:擴張到 sum=9 ≥ 8 後拚命縮左 —— 移出 1 後 sum=8 仍達標(更短!minL: 3→2),再移出就不達標,停。

求最長 vs 求最短 · 收縮時機相反// THE MIRROR

最長:違規才縮 · 最短:達標就縮

可變長視窗兩大類,差別在收縮觸發的時機:

‧ 求最長(3、424):維持視窗合法,不合法時才縮,答案取 max(r-l+1)。目標是「在合法前提下盡量長」。
‧ 求最短(本題、76):一旦視窗合格(達標),就拚命縮,每次仍合格就更新 min(r-l+1)。目標是「在合格前提下盡量短」。

一眼判斷:題目問「最長」→ 違規才縮(縮是為了恢復合法);問「最短」→ 達標就縮(縮是為了逼出更短的合格解)。 本題「達標」= sum ≥ target,而正數保證縮左必使和變小、遲早跌破 target —— 這是收縮有意義的前提。若有負數,縮左不一定變小,滑動視窗就失效(得改用前綴和 + 單調隊列,見 862)。

複雜度// COMPLEXITY

T = O(n)　　S = O(1)

r 前進 n 次;l 全程最多也前進 n 次(內層 while 跨整個外層加起來 ≤ n 次,攤還 O(1)/步)。只用幾個變數 → O(1)。follow-up 的前綴和 + 二分是 O(n log n),較慢但示範另一種思路。

一句洞察// INSIGHT

可變長視窗的求最短:右擴張累加,達標(sum≥target)就拚命縮左、每次合格更新最短。和 3/424 的「違規才縮」正好相反。 正數保證縮左單調變小,是這招的前提;含負數要改前綴和 + 單調隊列(862)。O(n)。
