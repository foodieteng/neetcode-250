範例 Trace// t 有重複字元

// s = "aa", t = "aa"(need a:2, required=1 種) r=0 a: freq[a] 2→1 (仍缺) → formed=0 窗"a" [0,0] 未覆蓋 r=1 a: freq[a] 1→0 湊滿 → formed=1==required → 覆蓋! 記最短 "aa"(2); 縮左: ++freq[a] 0→1 (>0 不足) formed=0, l=1 → 停 回傳 "aa"(需要兩個 a,少一個都不算覆蓋)
t="aa" 展示重複字元:required 是 1 種(只有字母 a),但 need[a]=2 —— 要湊到兩個 a 才算這一種「滿足」。 所以 r=0 只有一個 a 時 formed 還是 0(沒滿),r=1 湊到第二個才 formed=1。「種類數」與「每種的需求次數」是兩回事:required 數種類,freq 管次數,缺一不可。

// ANSWERS
A1

required 要數的是「有幾種字元需要被滿足」,也就是 t 的相異字元數 = freq.size()。formed 每滿足一種字元(不管它需求是 1 個還是 5 個)就 +1,滿足全部種類時 formed == required。若用 t.size()(總長度),當 t 有重複(如 "aab",size 3、種類 2)時 required 偏大,formed 最多只能到「種類數」永遠達不到,永不覆蓋 → 回空。所以必用相異字元數。

A2

因為 formed 記「已剛好滿足的種類數」,只在狀態跨過「剛好滿足」界線時才該變。擴張 --freq:從 1→0 是「從缺到剛好」,這一刻該 formed++;若用 <= 0,則每次減(包括從 0→-1 的 surplus)都會再 ++,重複計數。縮左 ++freq:從 0→1 是「從剛好到不足」,該 formed--;若用 >= 0,則移出 surplus(-1→0)也會誤 --。只認「正→0」和「0→正」兩個跨零瞬間,才不多不少。

A3

freq[c] > 0 = 視窗還缺 c(需求未滿);== 0 = 剛好滿足;< 0 = 有多餘的 c(surplus)。因為初值設成「需求(正)」,一個 freq 就能表達「缺多少 / 多多少」—— 缺是正的絕對值、多是負的絕對值。用兩張表(need 固定、window 累加)寫更直白,但要對每個字元比 window[c] >= need[c];而符號版把「差值」直接存在一張表裡,少一次比較、少一張表。兩者等價,符號版更精巧。

A4

共用「求最短、達標就縮」模板:右擴張;一旦達標就進 while 拚命縮左、每次仍達標更新最短;縮到不達標停。差別只在「達標」的定義:209 是 視窗和 ≥ target(用 total 判斷);76 是 視窗覆蓋 t(用 formed == required 判斷)。一句話:換「達標條件」與「怎麼 O(1) 維護它」,骨架不變。76 難在後者 —— 覆蓋比求和難判斷,所以引入 formed 計數器。

A5

O(m+n)。若每步都重新檢查「視窗每種字元數 ≥ 需求」,是 O(字元集)/步 → O(m·字元集)。formed 計數器把「是否覆蓋」壓成一次整數比較 formed == required(O(1)),它靠 freq 的跨零瞬間增量維護,不必每步重掃。加上 l、r 各只掃 s 一次(攤還),總時間 O(m+n)。這正是「用計數器把集合/覆蓋判斷降成 O(1)」帶來的加速,和 567 的 matches、424 的 maxF 同源。

App 1

可變長視窗總覽(換「合法/達標條件」即換題):
‧ 3 最長無重複 — 求最長,合法=無重複字元,違規(重複)才縮。
‧ 424 替換 — 求最長,合法=長−眾數≤k,違規才縮。
‧ 209 最短子陣列和 — 求最短,達標=和≥target,達標就縮。
‧ 76 最小覆蓋 — 求最短,達標=覆蓋 t,達標就縮。
一眼套:問最長→違規才縮、取 max;問最短→達標就縮、取 min。剩下只需想「合法/達標怎麼 O(1) 判斷」。

App 2

本題答案唯一,若要所有最小覆蓋窗:先跑一遍求出最小長度 bestLen;再第二遍(或同一遍記錄)收集所有「覆蓋且長度 == bestLen」的視窗起點。實作上,在 while (formed==required) 內,當 r-l+1 == bestLen 時把 l 存進結果清單(需先知道 bestLen,故兩遍較簡單)。仍是 O(m+n)。核心不變,只是把「記一個最短」改成「記所有等於最短的」。

App 3

本題字元是大小寫英文(52 種),vector<int>(128)(ASCII)完全夠用且比雜湊快。若是任意 Unicode,改用 unordered_map<char32_t,int>;required 仍是「t 相異字元數」= map 大小,formed 邏輯不變。唯一要注意的是字元集大時陣列不適用(太大或稀疏),換雜湊即可,時間仍 O(m+n)(雜湊操作平均 O(1))。骨架與計數器邏輯完全不動。
