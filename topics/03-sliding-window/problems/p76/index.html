題目簡介// THE QUESTION

QUESTION

給 s、t,求 s 中包含 t 所有字元(含重複次數)的最短子字串;不存在回傳 ""。答案保證唯一。 例:s="ADOBECODEBANC", t="ABC" → "BANC"。

這是滑動視窗的hard 集大成,也是 209 的近親:同樣「求最短、達標就縮」,只是「達標」的定義從「和 ≥ target」換成「視窗覆蓋了 t」。 難點在如何 O(1) 判斷「已覆蓋 t」 —— 用一個 formed / required 計數器:required = t 的相異字元數,formed = 目前已湊滿需求次數的字元種類數;formed == required 就是覆蓋。

2

為什麼「達標就縮 + formed 計數」// WHY

× REJECT · O(m²·n)
枚舉所有子字串 + 檢查覆蓋

試每個子字串再數字元看是否覆蓋 t。m 到 10⁵,平方以上,超時。

△ 每步全比一次
每步比對整張 need 表

可變長視窗但每步掃 t 全部字元判覆蓋,O(m·字元集)。能過,但可用 formed 計數做到每步 O(1)。

SHRINK-WHILE-COVER · ADOPT ↓
● ADOPT · O(m+n) / O(字元集)
formed==required 就拚命縮左

need[c] 記 t 每字元的需求。右擴張時某字元湊滿需求 → formed++;formed == required 即覆蓋,進入縮左:更新最短、移出左端,若使某字元不足 → formed-- 停。 每字元進出視窗各一次、formed O(1) 更新 → O(m+n)。

3

核心策略 · 四步驟// STRATEGY

  1. 建需求 —— freq[c] 記 t 每字元次數;required = 相異字元數。
  2. 擴張 + 湊滿 —— r 前進,--freq[s[r]];若剛好變 0(湊滿該字元)→ formed++。
  3. 覆蓋就縮 —— while (formed == required):更新最短視窗;++freq[s[l]],若 > 0(移出後不足)→ formed--;l++。
  4. 回傳 —— 最短視窗;從沒覆蓋過回 ""。

formed / required · O(1) 判斷覆蓋// THE COVERAGE COUNTER

用一個計數器把「有沒有覆蓋 t」變成 O(1)

若每步都重新比對「視窗每個字元的數量是否都 ≥ t 的需求」,是 O(字元集)。技巧是把 freq[c] 初始化成 t 的需求(正數),之後：

‧ 擴張 --freq[s[r]]:對 t 需要的字元,計數從正數往下減。剛好減到 0 代表這個字元的需求恰好被湊滿 → formed++。多的會變負(surplus),不再觸發。
‧ 縮左 ++freq[s[l]]:把移出的字元計數加回。加到 > 0 代表移出後這個字元從剛好變成不足 → formed--,視窗不再覆蓋。

於是「是否覆蓋 t」= formed == required —— 一個整數比較,O(1)。關鍵在只在「跨過剛好滿足的界線」時動 formed(減到 0、或加到 1),surplus/不足的其他變化都不影響。這和 567 的 matches 計數器是同一種「用計數器把集合比對降成 O(1)」的手法。

複雜度// COMPLEXITY

T = O(m + n)　　S = O(字元集)

建 t 需求 O(n);主體是滑動視窗,l、r 各掃過 s 一次(每字元進出視窗各一次,攤還 O(1)/步),formed 也 O(1) 維護 → O(m)。計數表大小 = 字元集(英文字母 52)→ O(1) 視為常數。

一句洞察// INSIGHT

「求最短、達標就縮」的可變長視窗(同 209),只是「達標」= 覆蓋 t。用 formed / required 計數器把覆蓋判斷降成 O(1):湊滿一種 formed++、移出致不足 formed--。 這是滑動視窗的 hard 模板,O(m+n)。
