觀念題// CONCEPT Q

Q1

為什麼「s2 含 s1 的排列」可以化約成「有一段長度 |s1| 且字元計數 == s1 的子字串」?排列的定義為什麼讓「順序」不重要?

Q2

為什麼相鄰兩個視窗的計數只差兩個字母(一進一出),不必每次重數整個視窗?

Q3

本題視窗是「恰好定長」,和 219(上限 ≤ k+1)、3(可變長) 在左右指針的移動上各有什麼不同?

Q4

matches 計數器版本:加入一個字元後,為什麼判斷 win[in]==need[in](+1)和 win[in]==need[in]+1(-1)就能維護「相等字母數」?

Q5

為什麼建完首窗要先比一次 s1Map==cmp 再進滑動迴圈?不比會漏掉什麼?

範例 Trace// 首窗就命中

// s1="adc", s2="dcda" → 首窗 "dcd" 不是,但要注意首窗先比 s1Map: a:1 d:1 c:1 首窗 [0,2]="dcd": cmp d:2 c:1 ≠ 目標 → 不命中(先比過) r=3 '+a −d': [1,3]="cda" cmp c:1 d:1 a:1 == 目標 → return true true(視窗 "cda" 是 "adc" 的排列)
// s1="ab", s2="ab" → 首窗即答案(展示「先比首窗」的必要) 首窗 [0,1]="ab": cmp a:1 b:1 == 目標 → return true(不先比就會漏!) true;若跳過首窗比對、直接進 r 迴圈,這個 case 會漏答
第二個 case s1==s2=="ab" 說明「先比首窗」不可省:答案就在最開頭時,滑動迴圈根本還沒開始,若不先比一次就直接漏掉。 這是「定長視窗」常見的邊界 —— 首窗要獨立處理一次,之後才進入滑動。

App 1

App 2

App 3

回傳實際子字串:除了 true/false,還要輸出第一個是排列的子字串。掃描時多記什麼?這和「定長」有什麼便利?

// ANSWERS
A1

排列就是把 s1 的字元重新排順序 —— 字母的種類和各自數量完全不變,只有順序可能不同。所以「某子字串是 s1 的排列」⟺「它長度等於 |s1|,且每個字母出現次數都和 s1 一樣」。順序不重要正是因為排列允許任意重排;我們只需比對「字母計數這個和順序無關的指紋」。這把「找排列」變成「找計數相等的定長視窗」。

A2

因為視窗恰好定長、每步只平移一格:右邊進一個新字元 s2[r]、左邊出一個舊字元 s2[l],其餘字元原封不動留在視窗裡。既然只有兩個位置變動,計數就只有「進的那個 +1、出的那個 -1」兩處要改,其餘 24 個字母不變。這就是增量更新:相鄰視窗O(1) 轉移,不必 O(m) 重數,是滑動視窗打敗「每窗重算」的核心。

A3

恰好定長(本題):l 和 r 同步前進,間距恆為 |s1|-1,每步一進一出,沒有 while。上限定長(219):r 每步進,l 只在 r-l>k 時才追上來縮,窗長可小於上限。可變長(3):r 每步進,l 在視窗違規時用 while 縮到合法,窗長浮動。一句話:本題 l、r 鎖步同行,另兩種 l 按條件追趕。

A4

matches = 「目前計數已相等的字母數」。加入字元 in 後 win[in]++,只有 in 這個字母的相等狀態可能改變:若 win[in] 剛好變成 need[in] → 從不等變相等,matches++;若剛好變成 need[in]+1 → 從相等變超過(不等),matches--;其他情形相等狀態沒變。移出字元對稱處理(用 need-1)。因為每次只動一個字母、且只在「跨過相等界線」時才變 matches,所以 O(1) 就能維護。matches==26 代表全部字母都相等。

A5

因為答案可能就在 s2 最開頭那 |s1| 個字元。原版的滑動迴圈從 r = |s1| 開始,處理的是第二個以後的視窗;首窗 [0, |s1|-1] 是在迴圈之前建好的,若不在那裡比一次,像 s1="ab", s2="ab" 這種「首窗即答案」的情況就會直接漏掉、回傳 false。所以「建首窗 → 先比一次 → 再進滑動」三段缺一不可。

App 1

LC 438 幾乎一模一樣,差別只在:本題找到一個就回 true,438 要收集所有命中的起始索引。所以把 if (cmp==s1Map) return true; 改成 if (cmp==s1Map) ans.push_back(l);(l 是當前視窗左端),掃完回傳 ans。骨架(定長視窗 + 增量計數 + 相等判定)完全共用 —— 567 是「存在性」、438 是「全部位置」。

App 2

vector<int>(26) 只適用小寫字母。字元集大(Unicode)時改用 unordered_map<char,int> 或更大的陣列;matches 版的目標從 == 26 改成 == 目標中「不同字元」的種類數(即 need 裡非零的鍵數,win 只需追蹤這些鍵)。核心邏輯不變,只是「全部相等」的門檻從固定 26 改成實際字元種類數。

App 3

因為視窗定長,只要記下命中時的左端 l,子字串就是 s2.substr(l, |s1|) —— 長度固定不必另存。找第一個時 return s2.substr(l, s1.size()) 即可;要全部就把每個命中的 l 存起來最後統一 substr。定長的便利正在於「知道起點就知道整段」,不像可變長還要記長度。
