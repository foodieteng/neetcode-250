重點回顧// RECAP

關鍵一句話
結構單調遞減雙端隊列(存索引)
front恆為當前視窗最大值
彈 backnums[i] ≥ nums[back](被蓋過)
彈 fronti-k ≥ front(滑出視窗)
為何存索引要判斷 front 是否過期
攤還每元素 push 一次、pop ≤ 一次 → O(n)
複雜度O(n) 時間 / O(k) 空間

觀念題// CONCEPT Q

Q1

為什麼加入 nums[i] 時,可以放心彈掉隊尾所有 ≤ nums[i] 的元素?證明它們永遠不可能再當某個視窗的最大值。

Q2

為什麼隊列存索引而不是值?只存值會缺什麼資訊?

Q3

裡面有內層 while(彈 back)看起來像巢狀,為什麼總時間仍是 O(n) 而非 O(n·k)?用「每個元素進出隊列幾次」說明。

Q4

彈 back 用 >=(含等號)還是 >(嚴格)?兩者都對嗎?差在哪?

Q5

為什麼「過期」只需檢查 front(至多彈一次),不必掃整個隊列找過期元素?

範例 Trace// 遞增序列 → 每步彈 front

// nums = [1,2,3,4], k = 2(遞增 → back 從不彈,front 常過期) i=0 (1): dq=[0] i=1 (2): 2≥1 彈索引0; dq=[1] → nums[1]=2 i=2 (3): 3≥2 彈索引1; dq=[2] → nums[2]=3 i=3 (4): 4≥3 彈索引2; dq=[3] → nums[3]=4 回傳 [2,3,4];嚴格遞增時新的最大一來就把舊的全彈掉,隊列恆長 1
// nums = [4,3,2,1], k = 2(遞減 → back 從不彈,front 每步過期) i=0 (4): dq=[0] i=1 (3): 3<4 不彈; dq=[0,1] → nums[0]=4 i=2 (2): front 0 過期; 彈; 2<3 不彈; dq=[1,2] → nums[1]=3 i=3 (1): front 1 過期; 彈; dq=[2,3] → nums[2]=2 回傳 [4,3,2];遞減時全靠「彈過期 front」讓最大值遞補
兩個極端把單調隊列的兩個動作各自放大:遞增序列每步用「彈 back」清空舊的(新的更大);遞減序列每步用「彈過期 front」讓次大遞補。 一般序列是兩者交織 —— 但無論如何,每個元素只被 push 一次、pop 一次,所以永遠 O(n)。

App 3

單調隊列總覽:回顧第 03 章的三種視窗大小控制(定長 / 上限 / 可變)與這題的「定長 + 求極值」,單調隊列補上了哪一塊拼圖?

// ANSWERS
A1

設隊尾是 j,新來的是 i(j < i)且 nums[j] ≤ nums[i]。考慮今後任何包含 j 的視窗:因為視窗是連續的且 i 在 j 右邊,只要視窗還含 j(j 尚未從左滑出),它一定也含 i(i 更靠右、更晚滑出)。而 nums[i] ≥ nums[j] —— 所以那個視窗的最大值至少是 nums[i],j 當不了最大。既然 j 在它「還活著」的每個視窗裡都被 i 壓過,它永遠沒有出頭之日,彈掉不後悔。

A2

存索引是為了判斷 front 是否已滑出視窗(過期) —— 這需要知道它的位置:i - k >= dq.front()。若只存值,你知道「最大值是多少」,卻不知道它在哪一格、何時滑出,無法判過期。存索引後,取值只要 nums[dq.front()]。一句話:值回答「最大是多少」,索引回答「它還在不在視窗裡」,後者才是滑動的關鍵。

A3

O(n),靠攤還分析。關鍵事實:每個索引一生只被 push_back 一次,也只可能被 pop(從 back 或 front)一次 —— 被彈掉後就永遠離開隊列。所以整個過程所有 pop 的總次數 ≤ n,那個內層 while 跨越整個外層加起來也只跑 ≤ n 次。總操作 = n 次 push + ≤ n 次 pop = O(n)。這不是「每個 i 都彈 k 次」的巢狀,而是「總彈掉次數受總元素數上限」的攤還。

A4

兩者都對,慣用 >=。遇到相等值時:>= 會彈掉舊的、只留索引較大(較晚)的那個 —— 它更晚過期,能讓這個最大值在隊列裡撐更久,隊列也更短。用嚴格 > 則把相等的都留著(隊列較長、常數略大),但因為它們值相同、front 仍是正確的最大值,答案不變。正確性上等價,>= 效率略優。

A5

因為隊列裡的索引從 front 到 back 嚴格遞增(我們總是 push 當前 i 到 back,i 一直變大)。「過期」= 索引 ≤ i-k(滑出左界),而索引最小的就是 front。所以若有過期元素,一定是 front;彈掉 front 後,下一個(次小索引)若也過期會繼續彈,但實務上每步至多一個新元素滑出視窗,故至多彈一次。單調的索引順序,讓「找過期」退化成「只看 front」。

App 1

滑動視窗最小值:把單調性反過來 —— 維持遞增隊列。彈 back 條件改成 nums[i] <= nums[dq.back()](新的更小就把大的彈掉),front 仍是最小值。過期判斷、攤還分析完全一樣。求 max 用遞減隊列、求 min 用遞增隊列,front 永遠是你要的極值 —— 同一結構的鏡像。

App 2

LC 862(含負數的最短子陣列和 ≥ k):先算前綴和 pre,子陣列 [l,r] 的和 = pre[r]-pre[l]。對每個右界 r,要找最靠右的 l 使 pre[r]-pre[l] ≥ k。含負數時 pre 不遞增,不能二分,改用單調遞增雙端隊列存 pre 的索引:從 front 彈出所有已滿足的(更新答案)、從 back 彈出 pre 不小於當前的(它們更左又更大,永遠不如當前優)。異:239 維護「視窗極值」用遞減、862 維護「前綴和候選」用遞增;同:都靠「後來者若更優就淘汰前面沒用的」把每元素進出各一次做到 O(n)。

App 3

第 03 章的滑動視窗有三種大小控制:恰好定長、上限定長、可變長(最長/最短)。它們回答「視窗長度該多少」。而本題是定長 + 求視窗內極值 —— 前面幾題視窗內只需計數/求和(增量 O(1) 好維護),但「極值」在元素滑出時無法簡單回退(最大值走了,次大是誰?)。單調隊列正是補上這塊:它在 O(n) 內維護「滑動視窗的極值」,是計數/求和之外,滑動視窗的第三種「視窗摘要」工具。
