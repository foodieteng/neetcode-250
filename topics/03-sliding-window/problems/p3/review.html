重點回顧// RECAP

關鍵一句話
類型可變長滑動視窗(求最長)
擴張r 每步納入 s[r]
收縮freq[s[r]]>1 時縮左到合法
合法視窗內無重複字元
答案maxL=max(maxL, r-l+1)
跳版l=max(l, last[s[r]]+1)
複雜度O(n) 時間 / O(k) 空間

觀念題// CONCEPT Q

Q1

收縮為什麼只需盯 freq[s[r]] > 1?為什麼只有剛加入的 s[r] 可能造成重複,不用檢查別的字元?

Q2

收縮用 while 而不是 if。舉一個輸入,說明用 if(只縮一格)會漏掉什麼、答案怎麼錯。

Q3

本題(可變長)和 219(定長) 都是「右擴左縮」,唯一的差別在哪一步?一句話概括兩者收縮依據的不同。

Q4

「記上次位置一步跳」版本裡,為什麼 l 要寫 max(l, last[s[r]]+1) 而不能直接 l = last[s[r]]+1?

Q5

時間為什麼是 O(n) 而不是 O(n²)?while 收縮迴圈看起來像巢狀,為什麼不會讓複雜度變平方?

範例 Trace// abba 的 l 不倒退

// s = "abba"(跳版:last[] 記上次索引,l 不可倒退) r=0 'a': last[a]=0 l=0 窗"a" maxL=1 r=1 'b': last[b]=1 l=0 窗"ab" maxL=2 r=2 'b': last[b]=1>=l → l=2; last[b]=2 窗"b" maxL=2 r=3 'a': last[a]=0, 但 0 < l=2 → 不跳!l 保持 2 last[a]=3 窗"ba" maxL=2 回傳 2;若在 r=3 把 l 設成 last[a]+1=1 會「倒退」→ 錯
"abba" 是跳版的經典陷阱:r=3 的 a 上次在 index 0,但那已在目前視窗 [2,3] 外。若無腦 l=last[a]+1=1,l 會從 2 倒退到 1,把已移出的 b 又納回來 → 視窗重複、答案錯。 l = max(l, last[s[r]]+1) 或判斷 last[s[r]] >= l 才跳,保證 l 只增不減。

應用題// APPLY

App 1

LC 424 最長重複字元替換:最多可換 k 個字元,求最長「同一字元」子字串。合法條件變成什麼?(提示:視窗長 - 眾數次數 ≤ k)

App 2

LC 209 最小長度子陣列和:求和 ≥ target 的最短連續子陣列。這也是可變長視窗,但收縮邏輯和「求最長」有什麼相反之處?

App 3

回傳子字串本身:除了長度,還要輸出那個最長無重複子字串。掃描時要多記什麼?

// ANSWERS
A1

因為在加入 s[r] 之前,視窗 [l, r-1] 已保證無重複(這是迴圈不變式)。新加入的 s[r] 是唯一可能製造重複的字元 —— 如果它和窗內某字元相同,重複的必然是「s[r] 這個值」出現了兩次。所以只需檢查 freq[s[r]] > 1;其他字元的計數不可能因為這一步變成 >1。收縮就是把左端一路移出,直到 s[r] 這個值在窗內只剩一份。

A2

要 while。反例 "pwwkew" 的 r=2:窗是 "pw",加入第二個 w 後 freq[w]=2。光移出一格 p(if 只做一次)後窗變 "ww",freq[w] 仍是 2 —— 重複沒清掉!必須繼續移出第一個 w,窗才變 "w" 恢復合法。用 if 會留下重複、後續 maxL 被高估。收縮要「縮到合法為止」,量不固定,所以是 while。

A3

差別只在「何時收縮」那一步。219 定長:while (r - l > k) —— 看大小,超過固定上限就縮,和內容無關。本題可變長:while (freq[s[r]] > 1) —— 看內容合不合法,出現重複才縮,縮到合法為止。一句話:定長看「幾個」,可變長看「合不合法」。其餘(右擴張、更新答案)兩者完全一樣。

A4

因為 last[s[r]] 記的是該字元整段字串裡的上次位置,可能早已在目前視窗左邊界之外。若直接 l = last[s[r]]+1,當那個舊位置 < l 時會把 l 往回拉,等於把已移出的字元重新納入視窗 → 破壞「無重複」。用 max(l, last[s[r]]+1)(或先判 last[s[r]] >= l)保證 l 只增不減。"abba" 的 r=3 就是這個陷阱(見 Trace)。

A5

O(n)。雖然有內層 while,但 l 整個過程只會往右移,最多 n 次 —— 它不會回頭重數。所以「r 前進 n 次」+「l 前進至多 n 次」= 總共 O(n) 次指針移動,這是攤還分析:內層 while 的總執行次數跨整個外層加起來也只有 n。不是每個 r 都重掃一遍,所以不是 O(n²)。這正是滑動視窗打敗暴力枚舉的關鍵。

App 1

LC 424:視窗合法條件變成 視窗長度 - 視窗內出現最多的字元次數 ≤ k(要替換的字元數 = 非眾數的字元數,不能超過 k)。維護 freq[26] 與目前眾數次數 maxCount;當 (r-l+1) - maxCount > k 就縮左。答案是最長的合法視窗。骨架與本題一模一樣,只是把「無重複」換成「可替換 ≤ k 個」這個合法條件。

App 2

LC 209 求最短,收縮方向相反。本題(求最長)是「違規才縮、答案取 max」;209 是「一旦達標(sum ≥ target)就拚命縮、每次達標更新 min」。也就是:求最長 → 視窗不合法時收縮;求最短 → 視窗已合格時收縮(想在保持合格下盡量短)。認出目標是 max 還是 min,就知道收縮該放在「違規時」還是「達標時」。

App 3

多記最佳視窗的起點。每次更新 maxL 時,如果 r-l+1 比舊的 maxL 大,就記下 bestStart = l(和 bestLen = r-l+1)。掃完用 s.substr(bestStart, bestLen) 取出子字串。仍是 O(n),只是把「長度」升級成「長度 + 起點」。這是所有「求最長/最短區間」題想輸出區間本身時的通用做法。
