合法樹判定

重點回顧// RECAP

面向要點
樹的條件n−1 條邊 且 連通(無環),缺一不可
並查集逐邊 union;同根 = 環 → 非樹
parent 陣列負值=根、−值=size;非負=父節點
路徑壓縮parent[x]=Find(parent[x])
按 size 合併小樹掛大樹,避免長高
判定成功合併 add == n−1
複雜度時間 O(E·α(n))≈O(E),空間 O(n)

觀念題// QUIZ

Q1. 為什麼「合法樹」必須同時滿足「n−1 條邊」和「連通」?各舉一個只滿足其中一個卻不是樹的例子。
Q2. 並查集裡「一條邊的兩端 find 同根」為什麼就代表成環?
Q3. parent 一個陣列怎麼同時表示「父指標」和「size」?負值和非負值各代表什麼?
Q4. 路徑壓縮和按 size 合併各自解決什麼問題?只做其中一個夠快嗎?
Q5. 若已經有 edges.size() == n−1 的前置檢查,最後的 add == n−1 還有必要嗎?兩者關係是什麼?

範例 Trace// valid vs cycle

// ✓ 合法樹 n=5, edges=[[0,1],[0,2],[0,3],[1,4]] edges=4 == n-1 ✓ union(0,1)✓ union(0,2)✓ union(0,3)✓ union(1,4)✓ add=4 == n-1 → true 最終一棵樹:0 為根,1/2/3/4 皆掛在 0 下
// ✗ 有環 n=4, edges=[[0,1],[1,2],[2,0]] edges=3 == n-1=3 ✓(邊數過關,但…) union(0,1)✓ union(1,2)✓ union(2,0):find(2)=find(0) 同根 → false(環!) add=2 < n-1=3 → false (3 是孤點,{0,1,2} 成環又漏了 3 → 非樹)
同樣 3 條邊,合法樹會「每條都接上新節點」;有環的那組會在某條邊上撞見同根,union 回 false。 「成功合併次數」正好在數:我究竟把幾個獨立塊接成了一塊。接滿 n−1 次才是一棵覆蓋全部的樹。

應用題// APPLY

App 1. 若把問題改成求連通分量數量(323),同一套並查集怎麼改?最後怎麼數?
App 2. 若要動態地一邊加邊、一邊隨時回答「現在是不是森林(無環)」,並查集為什麼特別適合?
App 3. 用 DFS/BFS 而非並查集判合法樹要怎麼寫?判環在有向 vs 無向圖有什麼不同?
// ANSWERS
A1. 樹是「連通且無環」的圖,而「n 點的連通無環圖恰好有 n−1 條邊」。只連通不夠:三角形 [[0,1],[1,2],[2,0]] 連通但有 3 條邊(> n−1)→ 有環,不是樹。只有 n−1 條邊也不夠:n=4, [[0,1],[1,0]?]… 更清楚的例子 n=4, [[0,1],[2,3],[?]] —— 若兩條邊各自成一組、第三條在其中一組內成環,就是「3 條邊(=n−1)但一邊成環、另一邊斷開」→ 不連通、非樹。兩條件合起來才等價於樹。
A2. 並查集裡「同根」代表 u 和 v 早就在同一個連通塊裡 —— 意思是它們之間已經存在一條路徑相連。這時再加一條直接邊 (u,v),等於在既有路徑之外多開一條路,兩條路徑構成一個環。所以 find(u)==find(v) 的邊必定製造環,union 回 false、不計入 add。
A3. 約定:parent[x] < 0 表示 x 是根,而且 −parent[x] 就是這棵樹的 size(所以初始 −1 = 根 + size 1);parent[x] >= 0 表示 x 不是根,值就是它的父節點編號。合併時 parent[大根] += parent[小根] 是兩個負數相加,size 正確累加;parent[小根] = 大根 把小樹掛上去。一個陣列省掉額外的 size[],是常見的省空間寫法。
A4. 路徑壓縮解決「查詢路徑太長」:Find 時把沿途節點全部直接指向根,樹被壓扁,下次幾乎一步到根。按 size(或 rank)合併解決「合併把樹接歪、越長越高」:永遠讓小樹掛到大樹下,樹高受控。兩者合用才有攤還近乎 O(1)(反阿克曼 α(n))的保證;只做其中一個能到 O(log n),已很快但非最優。實務兩個都寫、程式碼也沒長多少。
A5. 兩者其實等價於同一件事,可視為互相印證。前置 edges.size()==n−1 先擋掉「邊數根本不對」的情況、提早回 false(剪枝)。在已知邊數 = n−1 的前提下,「add == n−1」⟺「沒有任何一條邊成環」⟺「n−1 條邊全部有效地把圖接成一塊」= 樹。所以嚴格說留一個判斷也能對(例如只保留 add==n−1 並額外確認邊數),但兩個一起寫語意最清楚、也最好講:先數量、後結構。
App 1. 幾乎不用改:一樣逐邊 Union(不需要回傳 bool),跑完後數有幾個根即為連通分量數 —— 掃 parent,parent[i] < 0 的就是一個根,累加即可。初始 n 個孤立點就是 n 個分量,每成功合併一次分量少一個,所以也可用 cnt = n − 成功合併次數。這正是 323 的並查集寫法。
App 2. 因為並查集是增量(online)結構:每加一條邊只做一次近乎 O(1) 的 union,並能當場判斷這條邊是否連接兩個不同塊。「加邊後仍是森林(無環)」⟺「每次 union 都成功(沒撞同根)」。相比之下 DFS/BFS 每次加邊都要重跑整張圖判環,成本高。所以「動態連通性 / 動態判環」是並查集的主場。
App 3. DFS 版:先檢查 edges==n−1,建無向鄰接表,從 0 開始 DFS 數能訪到幾個點,訪到 n 個就連通(配合邊數即為樹)。無向圖判環:DFS 時記錄「來時的父節點」,若遇到已訪且非父的鄰居 → 有環(注意要用父節點/邊避免把回頭誤判)。這和有向圖判環不同:有向圖要用三態(0/1/2),走回「進行中(1)」才是環(見 207)。無向只需「已訪且非父」即可。
PREV程式碼與陷阱 NEXT→ 連通分量數 323
// MASTERY CHECK
能說出「樹=n−1 邊+連通、同根邊=環、parent 兼職父+size、路徑壓縮+按 size、add==n−1」,這題就通了。
