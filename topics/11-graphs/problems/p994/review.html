觀念題// QUIZ

Q1. 為什麼要把所有腐爛橘子一起入隊(多源),而不是從某一個腐爛橘子開始 BFS?
Q2. int sz = q.size() 的「一次清完一整波」有什麼用?如果不分批、直接逐格處理,分鐘數會怎麼算錯?
Q3. fresh 這個計數器有哪兩個用途?少了它,程式在什麼輸入上會出錯?
Q4. if (--fresh == 0) return time + 1; 這裡為什麼是 time + 1 而不是 time?
Q5. 這題和 286 牆與門是同一個引擎。為什麼 994 需要 sz 波次計數、而 286 不需要?

範例 Trace// [[2,1,1],[1,1,0],[0,1,1]] → 4

// 初始(2=腐爛 1=新鮮 0=空) 2 1 1 1 1 0 0 1 1 源:(0,0) 入隊;fresh = 6(6 顆新鮮)
// 一波一分鐘擴散 分鐘1:(1,0)(0,1) 腐爛 fresh 6→4 分鐘2:(1,1)(0,2) 腐爛 fresh 4→2 分鐘3:(2,1) 腐爛 fresh 2→1 (空格 (2,0) 擋住一邊) 分鐘4:(2,2) 腐爛 → --fresh==0 → return time+1 = 3+1 = 4
// RESULT 所有橘子在第 4 分鐘全部腐爛 // OUTPUT: 4
對照無解:[[2,1,1],[0,1,1],[1,0,1]] 裡左下角 (2,0) 被空格圍住、腐爛擴不到它。 佇列清空時 fresh 仍 > 0 → 回 −1。這就是為什麼一定要有 fresh 計數。

應用題// APPLY

App 1. 若感染改成八方向(含斜角),程式要改哪裡?分鐘數與複雜度會變嗎?
App 2. 若題目改成「每個空房到最近腐爛的距離」(填每格、不問總分鐘),你會怎麼改?這時還需要 sz 嗎?
App 3. 若有兩種腐爛源、感染速度不同(A 每分鐘擴 1 格、B 每 2 分鐘擴 1 格),還能用單純的多源 BFS 嗎?會遇到什麼困難?
// ANSWERS
A1. 因為腐爛是所有腐爛橘子同時發生的。把它們全放進第 0 波,等於一個「超級源」一起往外擴,BFS 的第 k 波就是「第 k 分鐘」新腐爛的一圈。若只從一個腐爛橘子出發,會忽略其他源的同時擴散,分鐘數會算多(某些橘子其實被別的源更早感染)。
A2. BFS 佇列裡會同時混著「這一波(這一分鐘)」和「這一波剛感染、下一分鐘才算」的格子。先記 sz = q.size() 只處理這 sz 個,就把「這一分鐘」和「下一分鐘」切開,清完這波才 time++。若不分批、把新入隊的也當同一輪,等於把好幾分鐘壓成一分鐘,分鐘數會算少。
A3. 兩個用途:① 早退 —— 開頭 fresh==0 代表本來就沒有新鮮橘子,直接回 0;② 判 −1 —— BFS 結束(佇列空)時若 fresh>0,代表有橘子被牆/空格隔開、腐爛擴不到,回 -1。少了 fresh,在「有孤立新鮮橘子」的輸入上就無法區分「全爛」與「爛不完」,會誤回一個分鐘數。
A4. time 是「已經完整過去的分鐘數」。當前程式正把某些鄰居從新鮮變腐爛,這個動作屬於下一分鐘 = time + 1。所以當最後一顆新鮮在此刻被感染,答案就是 time + 1。這是「早退」寫法;等價於「把這一波清完、time++、再回傳 time」,只是提前結束省一步。
A5. 差別在要回傳什麼。286 要「每格到最近門的距離值」,把距離直接寫回格子(rooms[r][c]+1)即可,每格自帶答案、不必知道現在第幾層,所以可省 sz。994 要「全部腐爛的總分鐘數」= BFS 一共擴了幾層,層數這個資訊只能靠 sz 一波一波數出來。要距離值不數波;要層數/分鐘才要 sz 分批。
App 1. 只要把 dirs 從 4 方向擴成 8 方向(加四個斜向),主體不動。斜向讓腐爛擴得更快,分鐘數可能變小;但複雜度量級不變,鄰居數 4→8 只是常數,仍 O(mn)。
App 2. 那就是 286 牆與門:把距離寫回格子 grid[nr][nc] = grid[r][c] + 1,不需要 sz 分批、也不需要 time —— 每格的答案就是它被填入的值。fresh 也可省(不必判總完成)。可見兩題只差「輸出距離值 vs 輸出總層數」這一點。
App 3. 單純多源 BFS不夠了。BFS 隱含「每條邊權重都是 1(每步一分鐘)」;當不同源的擴散速度不同,等於邊權不一致,會退化成加權最短路問題,要改用 Dijkstra(以「感染時間」為距離、用 min-heap 取最早)。BFS 的「一波一分鐘」前提被打破,先進佇列不再等於最早到達。
