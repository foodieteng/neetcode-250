面向要點
問題本質有向圖中入度 n−1、出度 0 的唯一節點
建模邊 a→b = a 信任 b → out[a]++, in[b]++
判定掃 1..n:out[i]==0 && in[i]==n−1

觀念題// QUIZ

Q1. 用「入度 / 出度」的語言,法官的兩個條件分別對應什麼?為什麼這兩個就完整刻畫了法官?
Q2. 判定為什麼是 in[i]==n−1 而不是 in[i]==n?差一在哪?
Q3. 為什麼最多只會有一個法官?用度數論證。
Q4. 這題明明分在 Graphs,為什麼不需要建鄰接表、也不需要 BFS/DFS 走訪?
Q5. 當 n==1 且 trust 為空時,答案是多少?程式怎麼算出來的?

範例 Trace// n=3 · trust=[[1,3],[2,3]] → 3

// 累加度數(掃 trust) 初始:in=[_,0,0,0] out=[_,0,0,0] (index 0 不用) 邊 [1,3]:out[1]++, in[3]++ → out=[_,1,0,0] in=[_,0,0,1] 邊 [2,3]:out[2]++, in[3]++ → out=[_,1,1,0] in=[_,0,0,2]
// 掃 1..n 找法官(n−1 = 2) i=1: out[1]=1 ≠ 0 → 不是 i=2: out[2]=1 ≠ 0 → 不是 i=3: out[3]=0 且 in[3]=2==n−1 → 符合 → return 3
// 對照 · 無法官 [[1,3],[2,3],[3,1]] → −1 多一條 [3,1]:out[3]++ → out[3]=1 i=3 時 out[3]=1 ≠ 0 → P3 也信任了人,不是法官 掃完 1..3 都不符合 // OUTPUT: -1
法官那格的特徵極鮮明:所有箭頭都指向它、它不射出任何箭頭(in=n−1、out=0)。 只要有一條它射出去的邊(out>0),它就出局。

應用題// APPLY

App 1. 如何只用一個陣列 score 解?累加規則與最後判定條件各是什麼?
App 2. 若 trust 可能有重複邊(同一對 [a,b] 出現多次),這份程式會不會誤判?為什麼?
App 3. 這題和「找名人(Celebrity)」有何異同?名人問題若只給一個 knows(a,b) 查詢函式,能做到 O(n) 嗎?
// ANSWERS
A1. out[i]==0 對應「法官誰都不信任」——它沒有射出任何信任邊;in[i]==n−1 對應「其他每個人都信任法官」——除了自己以外的 n−1 人都指向它。題目對法官的定義就是這兩條(加上唯一性),所以這兩個度數條件充分且必要地刻畫了法官。
A2. 因為法官不會信任自己,所以不會有一條 judge→judge 的邊,in[judge] 最多累到其他 n−1 個人的信任。要求 in==n 等於要求連自己都投自己一票,永遠達不到,程式就永遠回 −1。差的那個 1 正是法官自己。
A3. 反證:假設有兩個法官 x, y。x 是法官 → 其他所有人都信任 x,包含 y → y 至少信任了 x 這一人 → out[y]≥1。但 y 也是法官需要 out[y]==0,矛盾。所以最多一個法官,掃描時第一個符合的就是唯一答案。
A4. 因為答案只由每個節點的入度與出度決定,和「誰連到誰的具體路徑、圖是否連通」無關。BFS/DFS 是用來探索可達性 / 路徑 / 連通分量的;這裡不需要那些資訊,單純掃邊累加兩個計數器就足夠,O(n+m)。建鄰接表反而多花空間與時間。
A5. 答案是 1。只有一人時,trust 必為空 → in[1]=out[1]=0;而 n−1=0。判定 out[1]==0 且 in[1]==0==n−1 成立 → 回傳 1。直覺上「鎮上只有一個人,他自己就是法官」,而公式 in==n−1 在 n=1 時退化成 in==0,剛好吻合。
App 1. 用 score[i] += 1(當 i 被信任,b)、score[i] -= 1(當 i 信任別人,a)。法官會被 n−1 人信任(+ (n−1))且不信任任何人(−0),所以唯一判定條件是 score[i]==n−1。一個陣列、一次掃描,把 in 與 out 合成淨值。
App 2. 會受影響。若 [a,b] 重複兩次,in[b] 會被加兩次,可能讓某個非法官的 in 虛高到 n−1,或讓真法官的計數超過 n−1 而錯過。題目保證 trust 內不重複,所以原程式正確;若不保證,需先對 trust 去重(例如放進 set),或改用「每個人記錄信任集合」。
App 3. 兩者本質相同:名人 = 被所有人認識(in=n−1)、自己不認識任何人(out=0),就是法官。差別在輸入形式:997 直接給你所有邊,數度數 O(n+m);Celebrity 只給 knows(a,b) 查詢,可用擂台淘汰法:兩兩比較,每次淘汰一人(若 a 認識 b 則 a 出局,否則 b 出局),O(n) 找到候選,再 O(n) 驗證,總計 O(n) 次查詢。
