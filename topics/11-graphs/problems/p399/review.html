面向要點
問題本質加權圖上求兩點間路徑邊權連乘
建圖A/B=v → A→B:v、B→A:1/v

// 建圖 graph[a][b]=2 graph[b][a]=0.5 graph[b][c]=3 graph[c][b]=0.333…
// 幾個查詢 a/c:a →(×2) b →(×3) c = 6.0 b/a:b →(×0.5) a = 0.5 a/a:start==end = 1.0 a/e:e 不在圖裡 = -1.0 x/x:x 不在圖裡 = -1.0
注意 x/x = -1 而 a/a = 1 的差別:關鍵不是「分子分母相等」,而是這個變數有沒有被定義過(出現在等式裡)。 沒定義的變數,連 1 都不能回 —— 它根本不在我們的知識範圍內。start==end 的特判擺在「兩者都在圖裡」之後才安全。

應用題// APPLY

App 1. 改用 DFS 求同樣的路徑乘積怎麼寫?和 BFS 比,哪個在「找任一條路徑」時更省心?
App 2. 若查詢非常多(遠大於變數數),怎麼預處理讓每次查詢降到 O(1)?(提示:Floyd-Warshall 傳遞閉包 / 帶權並查集)
App 3. 若題目不保證無矛盾(可能給 a/b=2 又 a/b=3,或環上乘起來 ≠ 1),要怎麼偵測矛盾?
// ANSWERS
A1. 一條路徑 C → x → y → D,每條邊權重都是「起點除以終點」:C/x、x/y、y/D。相乘:(C/x)·(x/y)·(y/D)。把它寫成分數連乘,分子是 C·x·y、分母是 x·y·D,中間的 x、y 各出現一次分子、一次分母,約掉後只剩 C/D。所以不論中間經過哪些變數,乘積永遠是頭尾之比。
A2. 因為除法雙向可推:知道 A/B=v 就同時知道 B/A=1/v。查詢可能從任一端出發,若只建 A→B,想算 B/… 就走不出去。反向權重是 1/v 正因為 B/A = 1 ÷ (A/B) = 1/v。建雙向邊讓圖在兩個方向都連得通,BFS 才能自由走。
A3. 回 -1.0。判準是「變數有沒有被定義」——即有沒有出現在任何等式裡。x 從未出現 → 它不在圖裡 → 我們對它一無所知,連 x/x 都不能宣稱是 1。程式先檢查 !graph.count(start) || !graph.count(end) 回 -1,再檢查 start==end 回 1。順序不能反,否則 x/x 會被誤判成 1。
A4. 「無矛盾」保證任兩點間所有路徑的乘積都相同,所以我們不必比較多條路徑、也不必檢查一致性 —— BFS 找到任一條路徑就能直接回答。若可能矛盾,就得改成:把每個連通塊的所有點都算出「相對某個基準點的比值」,建表時若發現同一對點被賦予不同比值(超過浮點誤差)就是矛盾;或用帶權並查集,在 union 時檢查兩點既有的相對比值是否和新等式衝突。
A5. 因為答案是連乘,乘法的單位元是 1。起點到自己「什麼都還沒乘」時,比值是 start/start = 1。之後每經過一條邊乘上邊權,累積值才慢慢變成 start/當前點。若起始給 0,無論後面乘什麼都是 0,全錯。(對比:如果是加權求和類的題,初始才給 0。)
App 1. DFS 版:dfs(cur, target, acc, visited),若 cur==target 回 acc;否則標訪、對每個鄰居 dfs(nb, target, acc*w, …),任一支回非 -1 就往上傳。找「任一條路徑」時 BFS、DFS 都行、複雜度一樣;DFS 遞迴寫起來更短,BFS 用佇列更好控制、且天然是最短(邊數最少)的路徑(這題不影響答案,但少乘幾次浮點更穩)。
App 2. Floyd-Warshall:把「可達 + 比值」當傳遞閉包,d[i][j] = d[i][k]·d[k][j] 三重迴圈預算所有配對,O(V³) 建表、每次查 O(1)。或帶權並查集:每個點存「到其 root 的比值」,find 時做路徑壓縮並累乘比值,同組兩點 x/y = ratio[x]/ratio[y],近乎 O(1) 查。查詢遠多於變數時,這兩種比「每查一次 BFS」划算。
App 3. 用帶權並查集最自然:每個點記 ratio[x](= x/root)。處理 a/b=v 時,若 a、b 已在同一組,就檢查既有的 ratio[a]/ratio[b] 是否等於 v(容許浮點誤差);不等 → 矛盾。若不同組才 union,並據 v 調整其中一棵樹的 ratio 使兩邊自洽。這樣建圖過程順便就把矛盾抓出來了。
