範例 Trace// two cases

// 三角形 edges=[[1,2],[1,3],[2,3]] union(1,2) ✓ union(1,3) ✓ union(2,3):find(2)=find(3)=1 → 同根 → false return [2,3]
// 較大 edges=[[1,2],[2,3],[3,4],[1,4],[1,5]] union(1,2)✓ union(2,3)✓ union(3,4)✓ union(1,4):find(1)=find(4)(1-2-3-4 已連通)→ 同根 → false return [1,4] (注意:[1,5] 根本沒被處理到 —— 遇到答案就提早 return)
環上的邊 1-2-3-4-1 裡,前三條 (1,2)(2,3)(3,4) 都在把新節點接進來(成功); 第四條 (1,4) 的兩端早已被 1→2→3→4 這條路連通 —— 它是閉合環的那條,也正是環上輸入位置最後的邊。

應用題// APPLY

App 1. 若是 685 Redundant Connection II(有向圖,樹加一條邊變成環或雙父),為什麼不能直接照本題做?要多考慮什麼?
App 2. 若要回傳的是「多餘邊」而題目改成多解取字典序最小,程式要怎麼調整?
App 3. 不用並查集,改用 DFS 也能做:對每條邊,先看兩端是否已連通(DFS)再決定加不加。這樣複雜度變多少?為什麼並查集更好?
// ANSWERS
A1. union(u,v) 失敗代表 find(u)==find(v) —— u、v 早已在同一連通塊,也就是它們之間已經存在一條路徑。此時再加邊 (u,v) 等於在既有路徑外多開一條路,兩條路徑合起來就是一個環。而圖只有這一個環,所以這條「把環閉合」的邊就是能刪掉、讓圖變回樹的多餘邊。
A2. 唯一的環由若干條邊組成,它們在輸入陣列裡有先後次序。並查集照輸入順序逐邊 union 時,環上較早出現的邊都還在扮演「把兩個不同塊接起來」的有效樹邊(成功),只有掃到環上最後那條時,兩端才已被前面接通、union 失敗。所以「第一條失敗的邊」恰好就是環上輸入位置最靠後的那條 —— 正是題目要的「多解取最後出現」。
A3. 因為題目節點是從 1 編到 n(1-indexed),最大索引是 n。若 parent 只開 n,合法索引是 0..n−1,存取 parent[n] 就越界。開 n+1(索引 0..n)才放得下節點 1..n,index 0 空著不用。
A4. 那行 Union(u,v); 只有在第一次 Union 成功(這條邊不成環、往下沒 return)時才會執行。但這時 u、v 已經被第一次 Union 併進同一組,所以第二次 Union 進去 find(u)==find(v)、直接回 false、什麼都不改。因此它無害但完全多餘,刪掉程式更乾淨。「一條邊 union 一次」就對了。
A5. 三題共用同一套並查集(Find 路徑壓縮 + Union 按 size)。差別只在問法/回傳:261 問「是不是樹」→ 檢查 n−1 條邊且全部 union 成功(回 bool);323 問「幾個連通塊」→ 全部 union 完數根;684 問「哪條邊多餘」→ 回傳第一條 union 失敗的邊。骨架一樣,把握「union 成功/失敗」與「根的數量」兩個訊號就能通吃。
App 1. 685(有向)複雜在:多加一條有向邊可能造成兩種問題 —— ① 某個點有兩個父親(入度 2);② 形成有向環;或兩者兼具。不能只做無向並查集。標準做法:先找有沒有「入度 2」的點、記下它的兩條候選入邊;再分情況——若無雙父就跟本題一樣用並查集找環邊;若有雙父,則試著移除後一條候選看還有沒有環,據此決定回傳哪條。方向讓判斷細很多。
App 2. 若要「字典序最小」而非「輸入最後」,就不能只回第一條失敗邊。做法:先跑一遍找出環上的所有邊(例如找到環後回溯),再從其中挑字典序最小的回傳。或把邊排序後再判,但要小心排序會破壞「輸入順序」的語意 —— 通常題目二選一,看要哪種 tie-break。本題要的是「最後出現」,順序掃 + 第一條失敗最省。
App 3. DFS 版:對每條邊 (u,v),先在目前已加入的邊構成的圖上 DFS 看 u 能否到 v;能到就代表這條會成環 → 是答案;否則把邊加入圖繼續。每條邊一次 DFS 是 O(V+E),總共 O(V·E)=O(n²)。並查集把「兩端是否已連通」壓到近乎 O(1),總複雜度 O(n·α(n))≈O(n),明顯更快、程式也更短 —— 這正是並查集在「動態連通性」上的招牌優勢。
