島嶼周長

1

題目簡介// THE QUESTION

QUESTION

m×n 網格,1 是陸地、0 是水,恰好一座島(陸地四連通、無湖)。 求這座島的周長 —— 也就是島的外圍一圈有幾條單位邊。

關鍵翻譯:一格陸地有 4 條邊,一條邊只要朝向水或朝向網格外就是周長的一部分;朝向另一塊陸地的邊則是內部、不算。 所以周長 = 每個陸地格「朝外的邊」數量的總和。用 DFS 走遍島、每次「踩出島」就 +1,加起來就是答案。

2

為什麼「檢查要放進遞迴」// WHY · MY BUG NOTE

A 把邊界判斷寫在 dfs 呼叫「前」 我的錯
在鄰居迴圈裡先檢查 grid[nr][nc] 合不合法、是水就 continue 不遞迴。這樣根本不會踩進水/出界那格,那條周長邊就永遠沒被 +1 → 周長偏小。
× REJECT · 漏算周長
B 直接數:每格 4 − 2×(相鄰陸地對) 也可
雙迴圈掃每個陸地格,每格 +4,每一對「右鄰/下鄰也是陸地」就 −2。正確、O(mn),但這裡想練 grid DFS。
△ ALSO OK · 非 DFS
ADOPT
C DFS:邊界判斷當 base case O(mn)
遞迴先照樣往鄰居走,把「出界 / 踩到水」的判斷放進 dfs 的開頭(base case):一旦踩出島就 res += 1; return;。 踩到已訪陸地(-1)就 return;踩到新陸地就標記 -1 並繼續 flood 四方向。這樣每一條朝外的邊都會被真正「踩到」一次。
● ADOPT · 踩出島 = +1
錯誤筆記:這題的巧妙點就是「踩進水/出界」這件事本身要被執行到才數得到。把檢查寫在呼叫前(pre-guard)會把這一步攔掉,邊就漏算了。 判斷必須在 dfs 內當 base case,不能當呼叫前的守衛。
3

核心策略// STRATEGY

  1. 找起點:雙迴圈掃到第一個 grid[i][j] == 1,對它呼叫 dfs(島連通,一次就走遍全島)。
  2. dfs base case(先寫):if 出界 或 grid[r][c] == 0 → res += 1; return;(踩出島,記一條周長邊)。
  3. 去重:if (grid[r][c] == -1) return;(已訪,不重複)。否則把 grid[r][c] = -1 標記,再對上下左右各遞迴一次。

為什麼「踩出島 = 一條邊」// WHY IT COUNTS

LEMMA · 一次遞迴呼叫 = 穿過一條邊

從陸地格 (r,c) 往某方向 dfs(nr,nc),就是「試著穿過 (r,c) 與該方向之間的那一條邊」。

· 若對面是水或網格外 → 這條邊是島的邊界 → base case res += 1。
· 若對面是陸地 → 這條邊是內部 → 不加,繼續往裡 flood。
· 若對面是已訪陸地(-1)→ 走過了 → 直接 return,避免無限遞迴。

每個陸地格恰被標記一次(-1),但它的四個方向都各被「踩」一次 —— 所以每條朝外的邊都恰好被 +1 一次,不多不少。

6

複雜度// COMPLEXITY

T = O(m·n)　　S = O(m·n)

每格最多被標記一次、被四個鄰居各「踩」一次,總工作量與格數成正比。空間是遞迴堆疊深度,最壞(整格是島、蛇形)可達 O(mn)。

一句洞察// INSIGHT

周長 = 島「朝外的邊」總數。DFS 讓「踩出島」這件事真的被執行到,才數得到那條邊 —— 所以邊界判斷要當 base case,而不是呼叫前的守衛。
