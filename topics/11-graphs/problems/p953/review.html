範例 Trace// ["word","world"] · order「worldabc…」→ false

// 建 rank(只列相關字母) order = "worldabcefghijkmnpqstuvxyz" rank[w]=0 rank[o]=1 rank[r]=2 rank[l]=3 rank[d]=4 注意:正常字典序裡 d(3rd) < l(12th),但這裡 rank[d]=4 > rank[l]=3 → 外星序 d > l
// 比較相鄰對 (word, world) minL = min(4, 5) = 4 j=0: 'w'=='w' → continue j=1: 'o'=='o' → continue j=2: 'r'=='r' → continue j=3: 'd' ≠ 'l' → 查 rank:rank[d]=4, rank[l]=3 4 > 3 → 前者較大 → return false
// RESULT "word" 在外星序其實應排在 "world" 之後,卻放在前面 → 沒排好 // OUTPUT: false
// 前綴規則對照 · ["apple","app"](正常 order) minL = 3;j=0,1,2:'a','p','p' 全相同 → 內層跑完沒 break,j=3=minL w1="apple"(5) > w2="app"(3) → return false(短的該排前面)
兩個 false 各示範一條規則:word/world 是第一個不同字元由 rank 定勝負;apple/app 是前綴規則(短的較小)。 合起來就涵蓋兩字比較的所有情況。

應用題// APPLY

App 1. 若把字元換成 Unicode 字串、順序由一個 unordered_map<char,int> 給定,程式要怎麼調整?複雜度會變嗎?
App 2. 若不只要 true/false,還要回報第一個違規在哪(第幾對、第幾欄或前綴違規),怎麼改?
App 3. 若要把一批 words 依這個外星序實際排序(不是驗證),而不是 O(n²),該怎麼做?
// ANSWERS
A1. 因為「排序」在任何全序下都有遞移性:若 w0 ≤ w1 且 w1 ≤ w2,則 w0 ≤ w2 自動成立。所以整個 list 有序 ⟺ 每一對相鄰都滿足 words[i] ≤ words[i+1]。只要有一對相鄰逆序,整體就沒排好;反之全部相鄰皆遞增,整體必定有序。因此線性掃 n−1 對就夠,不必比 O(n²) 對。
A2. rank[c] 存字母 c 在外星 order 裡的位置(第幾個)。外星字典序把字母大小重新定義了,不能用 ASCII;直接 w1[j] > w2[j] 比的是正常字母序。把兩個字元各查一次 rank 再比,才是外星序下的大小。建表只花 O(26),之後每次比較 O(1)。
A3. break 用在「這一欄字元不同、但合法(前者 rank 較小)」時:兩字的大小已由這個字元拍板,後面的字元不該再看,直接結束這對、去比下一對。若漏掉 break,迴圈會繼續往後比,可能把後面某欄的逆序誤判成違規。以正常字母序的 ["ay","bx"] 為例:第 0 欄 a<b 已經定序、答案該是 true。但若不 break,迴圈跑到第 1 欄比 y vs x,y>x → 竟 return false,錯了。break 保證「大小一旦拍板就停」。
A4. 前綴規則處理「一方是另一方前綴」的情況:兩字前 minLen 個字元完全相同時,短的字典序較小。所以 ["app","apple"](短的在前)是 true;["apple","app"](長的在前)是 false。程式用「內層跑到 j==minL 都沒 break、且 w1 較長」來偵測這個違規。
A5. 269 給你一組已排序的 words,要你反推字母的順序 —— 從相鄰對的第一個不同字元建立「字母 A 在 B 之前」的邊,再拓撲排序得出 order(還要偵測矛盾/環)。953 剛好相反:order 已經給你,只要驗證 words 是否符合。已知順序 → 不用建圖、不用拓撲,單純線性比較即可。
App 1. 把 rank 從 int[26] 換成 unordered_map<char,int>(或字元集大的話用陣列),查表從 rank[c-'a'] 改成 rank[c]。比較邏輯完全不變。複雜度仍是 O(Σ|words|),只是每次查表變成雜湊 O(1) 均攤;空間變成 O(字元集大小)。
App 2. 把兩個 return false 改成記錄並回傳位置:第一個 false 回 (i, j)(第 i 對、第 j 欄字元逆序);前綴那個 false 回 (i, "prefix")。其餘掃描邏輯不動,第一次遇到違規就回報並結束。
App 3. 用同一張 rank 當比較器丟給 sort:sort(words.begin(), words.end(), [&](a,b){ 逐欄比 rank,第一個不同定序,前綴短者小 })。這樣是 O(n log n · L) 的真正排序,而非驗證。驗證(本題)只需 O(Σ|words|),比排序更省 —— 這也是為什麼「驗證有序」不該用排序來做。
