範例 Trace// merge via shared email

// 輸入 accounts = [ ["John", a, b], // a=johnsmith, b=john_ny ["John", a, c], // a 重複! c=john00 ["Mary", d], // d=mary(單獨) ]
// ① 映射 email→id a=0 b=1 c=2 d=3 owner = [John, John, John, Mary] parent = [-1,-1,-1,-1]
// ② 星形 union 帳號0:union(a,b) = union(0,1) → {0,1} 帳號1:union(a,c) = union(0,2) → a 已和 b 同組 → {0,1,2} 帳號2:只有 d,不 union → {3} parent 大致 [-3,0,0,-1]
// ③ 分組輸出 buf[0] = {a,b,c} → sort → [a,b,c] → 掛 John buf[3] = {d} → [d] → 掛 Mary ans = [[John,a,b,c], [Mary,d]]
合併的發動點是共用的 email a:帳號0、帳號1 各自 union 到 a 的同一個 id,兩帳號就被串在一起。 名字、排序都是收尾;真正的核心只有一句「同 email 同 id → union 到它就合併」。

應用題// APPLY

App 1. 不用並查集,改用 DFS/BFS 也能做:怎麼建圖?節點與邊各是什麼?複雜度一樣嗎?
App 2. 若不同人可能剛好同名(名字不能當合併依據),本題的做法要不要改?為什麼仍然正確?
App 3. 這題和 200 島嶼數量、323 連通分量 本質上都是「數/併連通塊」。三者最大的差別在哪一步?
// ANSWERS
A1. 因為「合併」的判準是共用 email,而 email 是天然的連結鍵。把 email 當節點、每個不同 email 對到唯一 id,那麼「兩帳號共用 email」就直接體現在「它們都 union 到那個 id」——不必額外計算。若把帳號當節點,得先掃一遍建「email → 有哪些帳號用它」的反查表,再對共用同一 email 的帳號兩兩連邊,多一層前處理。email 當節點把「共用」編碼進 id 映射,最直接。
A2. 並查集只在乎「這些元素最終在同一組」,不在乎它們內部怎麼連。要讓 acc[1..k] 全同組,只要每個都和同一個代表(acc[1])union 一次 —— 星形,共 k−1 次、O(k)。傳遞性(a~b、a~c ⇒ b~c)自動保證 b、c 也同組。兩兩相連要 C(k,2)=O(k²) 次 union,多做且無必要。
A3. 映射要涵蓋帳號裡的每一個 email(acc[1] 也要有 id),所以從 i=1。union 是「acc[1] 當星心去連其餘」,星心自己不用連自己,所以從 i=2。若映射也從 i=2,acc[1] 這個 email 就沒被註冊 id,之後 id[acc[1]] 取不到、甚至漏掉「只有一個 email」的帳號(它的唯一 email 正是 acc[1])。差這一格很關鍵。
A4. 可以隨便取,因為同一組的所有 email 一定屬於同一個人。合併只會發生在「共用 email」時,而題目保證共用 email 的帳號名字相同;所以一個並查集組裡的每個 email,其原始帳號名字都一致。取 root 的 owner 只是「挑其中一個」,不會錯。(反過來也說明:名字只是附帶資料,不是合併依據。)
A5. union / find 有路徑壓縮 + 按 size,每次近乎 O(1)(反阿克曼),總 O(N·α(N)) ≈ O(N)。但輸出時每組 email 要字典序排序,所有組加起來共 N 個 email,排序是 O(N log N),壓過了近線性的並查集。所以整體瓶頸在排序,總複雜度 O(N log N)。
App 1. 建圖:節點是 email;對每個帳號,把 acc[1] 和其餘 email 連無向邊(星形,和 union 對應)。然後對每個沒訪過的 email 起 DFS/BFS,收集整個連通塊的 email、記下起點帳號的名字,排序後輸出。複雜度同樣是 O(N log N)(排序主導)。並查集版通常程式更短、常數更小,但 DFS 版更直觀,兩者都是標準解。
App 2. 不用改,仍然正確。本題的合併完全靠共用 email驅動,名字從頭到尾沒有參與合併判斷(union 只看 email 的 id)。名字只在最後輸出時,從該組 root 的 owner 取一個貼上。就算兩個「John」是不同人,只要他們沒有共用任何 email,就會落在不同的並查集組、各自輸出,不會被錯併。名字同不同名根本不影響結果。
App 3. 三者核心都是「連通塊」,差別在節點/邊怎麼定義與要輸出什麼。200 島嶼:節點是網格格子、邊是上下左右相鄰的陸地(隱式圖),輸出塊數。323:節點/邊直接給定,輸出塊數。721:節點是 email、邊靠「同帳號」與「共用 email」建立,輸出每一塊的完整內容(排序 + 名字)。最大差別是 721 需要把每組的成員蒐集並格式化輸出,而非只數個數 —— 所以多了「映射」和「分組排序」兩步。
