面向要點
問題本質網格的連通分量個數(上下左右相連)
核心掃到未訪 '1' → count++ + DFS 沉掉整座島

觀念題// QUIZ

Q1. 為什麼「外層迴圈啟動 DFS 的次數」正好等於島的數量?
Q2.(注意)為什麼一進 dfs 就要立刻把當前格標成 '0'?如果先遞迴鄰居、後才標記會怎樣?
Q3.(注意)grid 的元素是 char。若把條件寫成 grid[r][c] == 0(沒有引號)會發生什麼?
Q4.(注意)這個沉島寫法會改動輸入 grid。若面試官要求不可修改輸入,你會怎麼調整?
Q5.(注意)什麼情況下遞迴 DFS 有堆疊溢位風險?怎麼避免?

範例 Trace// 3×4 grid → 3

// GRID · '1'=陸地 '0'=水 1 1 0 1 0 0 0 1 1 0 0 0 掃描順序:row-major(由左至右、由上到下)
// 掃描 + 沉島 (0,0)='1' → count=1,DFS 沉 (0,0)(0,1) → 藍島 繼續掃:(0,1)(0,2) 已是 '0'/水,跳過 (0,3)='1' → count=2,DFS 沉 (0,3)(1,3) → 棕島 (1,*) 全 '0',跳過 (2,0)='1' → count=3,DFS 沉 (2,0) → 綠島 其餘皆 '0'
// RESULT 啟動了 3 次 DFS,對應 3 個連通塊 // OUTPUT: 3
沉島後,同一座島的其他格都變 '0',外層掃過去自然跳過 —— 所以一座島只被 count 一次。 count 的每次 ++ 都精準對應「一個新連通塊的第一次觸碰」。

應用題// APPLY

App 1. 把 DFS 換成 BFS(queue) 要怎麼寫?為什麼超大網格時 BFS 比遞迴 DFS 安全?
App 2. 若「島」的定義改成八方向(含斜角)相連,程式要改哪裡?複雜度變嗎?
App 3. 若要回傳最大島的面積(LeetCode 695)而非島數,dfs 與主流程各怎麼改?
// ANSWERS
A1. 外層只在碰到還沒被沉掉的陸地時才 count++ 並啟動 DFS,而那次 DFS 會把整座相連的島全部沉成 '0'。於是同一座島的其他格,之後被掃到時都是 '0' 而被跳過,不會再觸發 count++。因此每個連通塊恰好貢獻一次 count,啟動 DFS 的次數就等於島數。
A2. 因為要防止相鄰兩格互相無限遞迴。若先遞迴鄰居、當前格還是 '1':A 遞迴到相鄰的 B,B 又看到 A 仍是 '1' 而遞迴回 A,A 再遞迴 B…… 來回不止、堆疊爆掉。正解是一進 dfs 就 grid[r][c]='0',這樣鄰居踩回來時看到 '0',在 base case 直接 return。
A3. grid[r][c] 是 char,其值是字元 '0'(ASCII 48)/ '1'(ASCII 49),不是整數 0/1。寫 == 0 會拿字元和整數 0(空字元 '\0')比較,永遠不相等 → base case 的「踩到水」判斷失效,島永遠沉不掉、邏輯全錯。必須帶單引號:== '0'、= '0'。
A4. 改用一個獨立的 vector<vector<bool>> visited:base case 判斷改成「出界 或 grid[r][c]=='0' 或 visited[r][c]」,進入時設 visited[r][c]=true 取代覆寫 grid。代價是多 O(mn) 空間,但輸入保持不變。
A5. 當網格幾乎全是陸地(例如 300×300 一整塊,或一條蛇形長島),遞迴深度可達 O(mn) ≈ 90000,可能堆疊溢位。避免方式:改用 BFS + queue(把深度改成佇列的堆積記憶體,不吃呼叫堆疊),或用顯式 stack 的迭代式 DFS。
App 1. 用 queue<pair<int,int>>:碰到 '1' 時 count++,把該格入隊並立刻標 '0';之後不斷出隊、把四個方向未沉的陸地標 '0' 並入隊,直到隊空。BFS 用堆積上的佇列存待處理格,深度不進呼叫堆疊,所以超大網格不會 stack overflow —— 這是它比遞迴 DFS 安全的原因。
App 2. 把 dirs 從 4 個方向擴成 8 個(再加四個斜向 {-1,-1},{-1,1},{1,-1},{1,1})。主體邏輯不變。每格鄰居數 4→8 只是常數,複雜度量級不變,仍是 O(mn)。
App 3. 讓 dfs 回傳面積:base case 回 0;否則沉掉自己後回 1 + Σ dfs(四鄰)。主流程對每個未訪 '1' 呼叫一次,用 maxArea = max(maxArea, dfs(i,j)) 取最大。從「數幾座」變成「量每座多大、取最大」,骨架相同。
