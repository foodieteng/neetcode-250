完整實作 · 從後往前// O(m+n)/O(1)

class Solution {
public:
    void merge(vector<int>& nums1, int m, vector<int>& nums2, int n) {
        int l = m - 1, r = n - 1, i = m + n - 1;  // 三指針從後起

        while (l >= 0 && r >= 0) {            // 兩邊都還有 → 取較大放尾
            if (nums1[l] > nums2[r])
                nums1[i--] = nums1[l--];
            else
                nums1[i--] = nums2[r--];
        }

        while (r >= 0) {                       // nums2 有剩 → 補上
            nums1[i--] = nums2[r--];
        }
        // nums1 有剩則不用管:它們本來就在 nums1 前段、已排序
    }
};
為什麼只補 nums2、不補 nums1:主迴圈結束時,若是 nums1 先耗盡(l<0),nums2 剩的還沒放,要搬過來;若是 nums2 先耗盡(r<0),nums1 剩下的元素本來就躺在 nums1 前段、且已排序,寫入位 i 也剛好停在它們後面 —— 不用動一根汗毛。

對照 · 從前往後為什麼要備份// FROM FRONT

// 從前往後填會覆蓋未讀資料 → 得先複製一份 nums1(多 O(m) 空間) void merge(vector<int>& nums1, int m, vector<int>& nums2, int n) { vector<int> a(nums1.begin(), nums1.begin() + m); // 備份前 m 個 int l = 0, r = 0, i = 0; while (l < m && r < n) nums1[i++] = (a[l] <= nums2[r]) ? a[l++] : nums2[r++]; // 取較小放前 while (l < m) nums1[i++] = a[l++]; while (r < n) nums1[i++] = nums2[r++]; } O(m+n) 時間但多 O(m) 空間;從後往前免備份、O(1),所以更優

從前往後「取較小放前」很直覺,但寫入位 i 會追上並覆蓋還沒讀的 nums1 元素,所以要先把前 m 個備份到 a。 從後往前的巧妙之處,就是利用尾端空位讓寫入永遠落在未讀資料後面,省掉這份備份。方向對了,空間就從 O(m) 降到 O(1)。

邊界與陷阱// PITFALLS

陷阱症狀正解
從前往後原地填 覆蓋未讀的 nums1 元素 → 錯 從後往前(或先備份)
漏補 nums2 尾巴 nums1 先耗盡時,nums2 剩的沒放 → 漏數 第二個 while 補 r≥0
多寫「補 nums1 尾巴」 其實不需要;寫了也無害但多餘 nums1 剩的本就在原位
i 初值錯 i=m+n 越界 i = m+n−1
m=0 或 n=0 擔心空陣列 l/r<0 天然跳過,安全

// nums1=[1,2,3,_,_,_] m=3, nums2=[2,5,6] n=3 l=2 r=2 i=5 : 3 vs 6 → 放 6 nums1=[1,2,3,_,_,6] r=1 i=4 l=2 r=1 i=4 : 3 vs 5 → 放 5 nums1=[1,2,3,_,5,6] r=0 i=3 l=2 r=0 i=3 : 3 vs 2 → 放 3 nums1=[1,2,3,3,5,6] l=1 i=2 l=1 r=0 i=2 : 2 vs 2 → 放 2(nums2) nums1=[1,2,2,3,5,6] r=-1 i=1 主迴圈:r=-1 停;補 nums2 尾巴:r<0 無 nums1 剩 [1,2] 本來就在 index 0..1、已排序 → 不動 [1, 2, 2, 3, 5, 6]
注意第三步:取自 nums1[2]=3 放到 i=3,l 退到 1。此時 index 2 的舊 3 還在,但下一步就被 nums2 的 2 覆蓋 —— 而那個 3 早已被複製到 index 3,不會遺失。 因為 i(=2) ≥ l(=1),覆蓋的永遠是「已讀或空」的位置,絕不碰未讀的 nums1[0..1]。
