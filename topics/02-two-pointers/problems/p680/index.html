題目簡介// THE QUESTION

QUESTION

給字串 s(僅小寫字母),最多刪除一個字元後若能成為回文,回傳 true。 例:"abca" → true(刪 'c' 或 'b');"abc" → false。

這是 125 驗證回文的升級:多了「可以刪一個」的容錯。 關鍵洞察:一路對撞比較,直到第一個不相符 —— 那一刻就得動用唯一的刪除機會,而且要刪的一定是那兩個衝突字元之一(s[l] 或 s[r])。各試一次,任一成立即可。

2

為什麼「雙指針 + 一次分叉」// WHY

× REJECT · O(n²)
枚舉刪掉每一個字元

對每個 index 刪掉後檢查是否回文。n 個 index × 每次 O(n) 檢查 = O(n²),太慢。

△ 想太多
在任意位置嘗試刪除

以為刪除點可能在中間某處。其實只要一路都相符,問題就出在第一個不符的那對,刪別處沒意義。

FORK ONCE · ADOPT ↓
● ADOPT · O(n)
對撞到第一個不符,分兩路各驗一次

l/r 對撞:相符就內收。碰到 s[l] != s[r] 時,刪除必是這兩個之一 → 檢查 isPali(l+1, r)(刪左)或 isPali(l, r-1)(刪右),任一是回文即 true。 這個分叉只發生一次(只能刪一次),所以總時間仍是 O(n)。

3

核心策略 · 三步驟// STRATEGY

  1. 對撞比較 —— l/r 從兩端往中間,s[l] == s[r] 就 l++、r--。
  2. 不符就分叉 —— 第一次遇到 s[l] != s[r]:回傳 isPali(l+1, r) || isPali(l, r-1)(試刪左端 或 試刪右端)。
  3. 撐到底 —— 全程都相符(不需刪) → true。isPali 是普通的兩指針回文檢查(無容錯)。

兩個關鍵論證// WHY IT WORKS

① 為什麼刪的一定是 s[l] 或 s[r]

走到第一個不符 s[l] != s[r] 時,外面 [0, l−1] 和 [r+1, n−1] 已經兩兩對稱(一路相符才走到這)。 要讓整體變回文,s[l] 和 s[r] 這一對的衝突必須解決,而唯一的刪除只能花在這裡:刪 s[l](讓 s[l+1] 去對 s[r])或刪 s[r](讓 s[l] 去對 s[r−1])。刪其他位置對這對衝突毫無幫助。所以只有兩種試法。

② 為什麼「分叉」還是 O(n)

主迴圈對撞到不符,是 O(n)。分叉後呼叫兩個 isPali,各自最多 O(n) —— 但分叉只發生一次(容錯用掉就沒了,isPali 內部是不再容錯的普通檢查,不會再分叉)。所以總成本 = O(n)(主掃) + O(n) + O(n)(兩次子檢查)= O(n)。「只能刪一次」正是把它壓在線性的原因 —— 若能刪 k 次,才會變成更貴的遞迴。

6

複雜度// COMPLEXITY

T = O(n)　　S = O(1)

主對撞 O(n),分叉後至多兩次 O(n) 子檢查、只發生一次。只用 l、r 幾個索引(isPali 用引用不複製字串)→ O(1) 空間。比「枚舉刪每個」的 O(n²) 快一個數量級。

一句洞察// INSIGHT

對撞比較到第一個不符,刪除只能是那兩個衝突字元之一 → isPali(l+1, r) || isPali(l, r-1)。 分叉只一次(容錯用完),所以還是 O(n)。「允許刪 1 個」= 普通回文檢查 + 一次二選一的補救。
