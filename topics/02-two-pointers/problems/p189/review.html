範例 Trace// k > n 的取模

// nums = [1,2], k = 3 → k %= 2 = 1(右轉 1 格) k %= 2 → 1 ① reverse 全部: [1,2] → [2,1] ② reverse 前 1: [2,1] → [2,1](單元素,不變) ③ reverse 其餘[1,2):[2,1] → [2,1](單元素,不變) [2,1] = 右轉 1 格;k=3 取模成 1,沒有白轉整圈
k=3、n=2:轉 2 格就回到原狀,所以只有 3 mod 2 = 1 格是「真的有效」的。 若不取模,begin()+3 會越過 end() 造成未定義行為。k %= n 同時解決「白轉整圈」與「越界」兩個問題。

// ANSWERS
A1

設前段 A(前 n-k)、尾段 B(尾 k),原陣列 AB,右轉 k 要得 BA。① 全反:(AB)ᴿ = BᴿAᴿ —— B 塊已移到前面、A 塊移到後面(位置對了),但兩塊各自順序被反。② 反前 k:前段現在是 Bᴿ,反一次 (Bᴿ)ᴿ=B 恢復。③ 反其餘:後段 Aᴿ 反成 A。得 BA。所以全反負責「換位置」,後兩次負責「修各塊順序」,靠的是「反兩次抵消」(Xᴿ)ᴿ=X。

A2

不能省。k 可到 10⁵,而 n 可能更小。例 n=2, k=3:begin()+3 已越過 end(),reverse 傳入非法迭代器 → 未定義行為(可能崩潰)。就算不越界,轉 n 的整數倍格等於沒轉,是純浪費。k %= n 把 k 化到 [0, n),同時避免越界和白轉。

A3

得到的是左轉 k。因為「先反前 k、再反其餘、最後全反」= ((AᴿBᴿ))ᴿ... 具體看:把陣列切成前 k(P)和其餘(Q),原 PQ。先反 P、Q 得 PᴿQᴿ,再全反得 (PᴿQᴿ)ᴿ = QP —— 把前 k 個搬到最後,正是左轉 k。所以「三次反轉」的順序決定左轉或右轉,反轉的是哪一段是關鍵。

A4

另開陣列:O(n) 時間、O(n) 空間,最好寫(res[(i+k)%n]=nums[i]),但違反原地。環狀替換:O(n) 時間、O(1) 空間,但要處理 gcd(n,k) 個獨立環、用計數控制,最難寫對。三次反轉:O(n) 時間、O(1) 空間,三行、最好記、最不易錯。面試要「原地 O(1)」又要穩,三次反轉是甜蜜點。

A5

reverse 的實作就是:左指針 l 從頭、右指針 r 從尾,swap(a[l++], a[r--]) 往中間夾,直到 l ≥ r —— 這正是對撞雙指針。共用的核心動作是「兩端往中間收」:167 是收斂找兩數、344 是收斂交換字元、189 用它當反轉原語。同一個「一頭一尾、往中間走」的骨架,套不同的每步操作,就解不同的題。

App 1

左轉 k = 右轉 n-k。最省事:把 k 換成 k = (n - k % n) % n,再照右轉三次反轉即可 —— 只改一個數。或直接調整反轉順序(見 A3)。兩者等價;實務上「左轉轉成右轉」最不易出錯。

App 2

環狀替換:從 start=0 開始,cur=start、prev=nums[start],反覆 next=(cur+k)%n、把 prev 放到 nums[next]、更新 prev=舊 nums[next]、cur=next,直到回到 start;用 count 累計搬過的元素,搬滿 n 個才結束。為什麼要計數:當 gcd(n,k) > 1 時,單一起點的環不會覆蓋所有位置(有 gcd(n,k) 個獨立環),一圈回到起點後要換 start=1,2,… 繼續,直到 count==n。這是它難寫對的原因。

App 3

LC 796:s 是 goal 的某輪轉 ⟺ s.size()==goal.size() 且 s 是 goal+goal 的子字串。因為 goal 的所有輪轉都會出現在 goal+goal 裡(把兩份接起來,任何起點長度 n 的視窗都是一個輪轉)。一行:return s.size()==goal.size() && (goal+goal).find(s)!=npos;。這是「輪轉 = 在雙倍串裡找視窗」的漂亮技巧,和本題「輪轉的結構」互相呼應。
