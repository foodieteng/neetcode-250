重點回顧// RECAP

關鍵一句話
限制每船最多 2 人、合重 ≤ limit
做法排序 + 對撞:最輕配最重
配得下兩人同船 l++, r--
配不下最重單獨 r--
正確性交換論證:最輕填最重的空位不吃虧
l≤r含等號 → 最後一人也給一船
複雜度O(n log n) 時間 / O(1) 空間
2

觀念題// CONCEPT Q

Q1

用交換論證證明:當最重的 H 能載下最輕的 L 時,「讓 H 帶 L」至少和任何最優解一樣好。

Q2

為什麼對最重的 r,只需比一次(跟最輕的 l)就能決定?若 l 都超重,為什麼不用試 r 配別人?

Q3

迴圈條件是 l <= r 而非 l < r。l == r 時發生什麼?寫成 < 會漏算什麼?

Q4

為什麼「把重的人湊在一起(最重配次重)」是錯的貪心?舉例說明它為何比「最重配最輕」差。

Q5

複雜度 O(n log n) 的瓶頸在哪?若 people 已排序,能到多快?

3

範例 Trace// 全部都能兩兩配

// people = [3,5,3,4], limit = 5 → 排序 [3,3,4,5] l=0,r=3: boats=1; 3+5=8 > 5 → r-- (5 單獨) 船{5} l=0,r=2: boats=2; 3+4=7 > 5 → r-- (4 單獨) 船{4} l=0,r=1: boats=3; 3+3=6 > 5 → r-- (3 單獨) 船{3} l=0,r=0: boats=4; 3+3=6 > 5 → r-- (剩 3 單獨) 船{3} l=0,r=-1: 結束 回傳 4;limit 太小 → 幾乎人人單獨
limit=5 對這組人偏小:每組配對都超重,結果四人四船。這展示貪心的另一面 —— 當沒有任何一對能同船,最少船數就是人數,對撞會自然地一路 r-- 得出。 反之 limit 夠大時,盡量兩兩配對、船數趨近 ⌈n/2⌉。limit 大小決定了能配對的程度。
4

應用題// APPLY

App 1

每船最多 3 人:若一艘船能載 3 人(合重 ≤ limit),還能用「最輕配最重」的簡單對撞嗎?為什麼變難了?

App 2

對撞貪心家族:本題和 11 盛水容器、167 兩數之和 II 都是「排序/有序 + 對撞」。三者的「移動哪一端」決策各是什麼?共同點是什麼?

App 3

能不能用計數排序?people[i] ≤ limit ≤ 3×10⁴,值域不大。怎麼用計數排序把總時間壓到 O(n + limit)?

// ANSWERS
A1

設某最優解裡最重的 H 沒帶最輕的 L —— 要嘛 H 單獨、要嘛 H 帶了別人 X;而 L 在別的船上(和某人 Y 或單獨)。做一次交換:把 L 移到 H 的船。因為 L 是最輕的,H + L ≤ H + X ≤ limit(若原本 H 帶 X)或 H + L ≤ limit(前提成立),H 這船仍合法;而 L 原本那船拿掉 L 只會更輕,也仍合法(甚至可能空出來省一船)。所以交換後船數不增 —— 「H 帶 L」的解至少和最優一樣好,貪心最優。

A2

因為 people[l] 是目前還在的最輕者。若 people[l] + people[r] > limit,那 r 配剩下任何人 p(都 ≥ people[l])必有 p + people[r] ≥ people[l] + people[r] > limit —— 沒有人能陪 r,他必單獨,一次比較就定案。若 people[l] + people[r] ≤ limit,交換論證告訴我們帶最輕的最優。所以每個 r 只跟 l 比一次,不必逐一試其他人 —— 這正是能線性掃描(排序後)的原因。

A3

l == r 代表只剩一個人。while (l <= r) 讓迴圈再跑一次:boats++ 給他一艘船,然後 sum = 2·people[l],不論過不過 limit(過→r--、不過→l++, r--)都讓 l > r 結束 —— 反正就這一人一船。若寫 l < r,當人數為奇數、最後剩一人時,迴圈提早結束,漏算那艘船,答案少 1。所以含等號不可省。

A4

「最重配次重」幾乎總是兩個重的加起來超 limit → 兩人都得單獨,浪費了配對機會。例 [1,1,3,3], limit=4:正解「最重配最輕」= {3,1}、{3,1} 共 2 船;而「最重配次重」= {3,3} 超重 → {3}、{3}、{1,1} 共 3 船,更差。把重的人湊一起只會互相排擠;讓重的人配最輕的,才能塞進最多對。貪心方向錯了,結果就差。

A5

瓶頸是排序 O(n log n);對撞本身只 O(n)。若 people 已排序,直接對撞 → O(n)。此外本題值域小(people[i] ≤ limit ≤ 3×10⁴),可用計數排序把排序降到 O(n + limit),整體 O(n + limit)(見 App 3)。所以「排序 + 對撞」的下限取決於能否更快地排序。

App 1

每船 3 人時,簡單對撞不再夠。2 人船的美好在於「最重者的伴只有一個名額,配最輕的最優」——決策唯一。3 人船時,最重者要配兩個人,「配哪兩個」的組合暴增,且「兩個輕的 vs 一輕一中」哪個省下的船更多並不顯然,簡單的最輕配最重會失去最優性保證。這類「裝箱(bin packing)」問題一般是 NP-hard;k=2 是恰好能用貪心對撞解的特例。這也解釋為什麼本題把每船限制在 2 人。

App 2

三者都是「排序/有序 + 一頭一尾對撞」,差別在移動決策:
‧ 167 兩數之和 II:sum 太小移左、太大移右 —— 單調消去。
‧ 11 盛水容器:移較矮牆 —— 解開瓶頸的貪心。
‧ 881 救生艇:最重必上船,配得下就帶最輕(l++, r--)、配不下單獨(r--)—— 配對貪心。
共同點:有序性讓「兩端」承載了最有用的資訊(最小/最大),於是每步只需 O(1) 決策就能安全地淘汰一端,把 O(n²) 降到 O(n)(排序後)。

App 3

計數排序:因為 people[i] ∈ [1, limit]、limit ≤ 3×10⁴,開一個 cnt[limit+1] 桶陣列,O(n) 統計每個體重的人數,再 O(limit) 依序展開成有序序列(或直接在桶上用兩個「桶指針」對撞:一個從最小體重、一個從最大體重,按人數消耗)。這樣把排序的 O(n log n) 降成 O(n + limit),總時間 O(n + limit)。當 limit 不大時,這比比較排序更快 —— 是「值域小 → 計數排序」的典型應用。
