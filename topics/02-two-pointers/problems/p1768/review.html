範例 Trace// 兩尾互斥

// 短 + 長:word1="a", word2="xyz" l=0 r=0 : 'a'+'x' → "ax" l=1 r=1 主迴圈:l=1=m 停 l<m? 否;r<n? 1<3 是 → res += "yz" → "axyz" axyz(word2 有尾)
// 長 + 短:word1="xyz", word2="a" l=0 r=0 : 'x'+'a' → "xa" l=1 r=1 主迴圈:r=1=n 停 l<m? 1<3 是 → res += "yz" → "xayz" xayz(word1 有尾)
兩個對稱的例子:一個是 word2 有尾、一個是 word1 有尾,但永遠只有一方有尾 —— 主迴圈在先用完的那方停,剩下的必是另一方。 這就是為什麼兩個接尾 if 天生互斥、不會重複追加。

App 2

App 3

合併 k 個序列交替:若不是兩個而是 k 個字串輪流各取一個,怎麼推廣?用什麼結構管理 k 個指針?

// ANSWERS
A1

對撞型:兩指針在同一序列的頭尾,往中間收,靠「兩端的對稱/配對關係」做事(反轉、回文、找和)。平行型(本題):兩指針各走一個序列,同向前進,把兩個序列合併或比對。判斷用哪種:「一個序列的兩端關係」→ 對撞;「兩個序列的對應/合併」→ 平行。同樣叫雙指針,結構完全不同。

A2

不可能兩個都有剩。主迴圈條件是 l < m && r < n,只要其中一方走到底就退出 —— 退出當下,走到底的那方已無剩,另一方可能有剩。所以「有尾巴」的至多一方。若兩方等長,退出時兩方都剛好用完,兩個尾巴都空,結果就是純交替。

A3

因為單迴圈把「交替」和「接尾」統一成同一套條件。i 從 0 跑到 max(m,n)−1:前段兩者都沒越界 → 交替各取(等同主迴圈);到了後段,較短的那串 i < 短長 已為假、不再取,只有較長的 i < 長長 仍為真、持續追加 —— 這自然就是「接尾巴」。用越界條件過濾,長短不齊免特判,少了兩段 substr。

A4

res = res + a + b 每次可能產生臨時字串並重新配置整個 res。若編譯器/實作沒優化,重複拼接 n 次、每次複製長度到 n,最壞是 O(n²)。reserve(m+n) 先把容量開好,之後 push_back / += 攤還 O(1),全程 O(n)、無重配置。本題 n ≤ 100 看不出差,但養成「預留容量」的習慣在長字串/大量拼接時很重要。

A5

像:都是平行雙指針,各走一序列、把兩者合成一個。不同:本題是硬性交替(不看內容,你一個我一個),LC 88 / merge sort 是依大小挑(比 a[i] 與 b[j],取小的那個,只推進被取的那個指針)。所以本題兩指針同步前進,合併有序陣列則各自按需前進。兩者都以「一方耗盡就把另一方剩的接上」收尾 —— 收尾邏輯完全一樣。

App 1

LC 88:比 nums1[i] 與 nums2[j],取較大者放到結果尾端,只推進被取的指針。之所以從後往前填:因為要原地合併進 nums1(它尾部有空位),從後往前寫不會覆蓋還沒讀的 nums1 前段元素。若從前往後會蓋掉未處理的資料。收尾:某方用完後,把另一方剩下的複製過去。這是「依大小」的平行雙指針經典。

App 2

LC 392:兩指針 i 走 s、j 走 t。每步只有 j 一定前進;當 s[i] == t[j] 時 i 也前進(這個字元配對成功)。若 i 走完 s(全配上了)→ 是子序列。這是「不對稱的平行雙指針」:一個指針掃主串、另一個只在匹配時前進。O(m+n)。

App 3

k 個序列交替:維護 k 個指針(一個陣列 pos[k])。一輪掃過所有還沒耗盡的序列,各取一個字元;哪個序列用完就把它移出(或用一個「還活著的序列」清單/佇列)。重複到所有序列都空。每個字元仍只取一次 → O(總長度)。核心從「兩個游標」升級成「一組游標 + 誰還活著」的管理,平行雙指針的自然推廣。
