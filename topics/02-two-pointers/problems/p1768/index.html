題目簡介// THE QUESTION

QUESTION

給兩個字串 word1、word2,從 word1 開始交替各取一個字元合併;若一方較長,把多出的字元直接接到尾端。回傳合併字串。 例:"abc" + "pqr" → "apbqcr";"ab" + "pqrs" → "apbqrs"。

這題示範雙指針的另一種面貌:不是像 344/125 那樣兩端對撞,而是兩個指針各走一個字串、平行前進(same-direction / parallel)。 這正是合併排序的 merge 步驟的雛形 —— 兩個序列各一個游標,交替取用。

2

為什麼「平行雙指針」// WHY

△ OK · 單迴圈也行
跑到 max(m,n),各自越界就跳過

一個 i 從 0 到 max(m,n),i<m 取 word1[i]、i<n 取 word2[i]。也對、更短,但邏輯上仍是「兩個游標」。

× 別過度
先算長度、切片、拼接各種特判

分「誰長誰短」寫一堆分支容易出錯。其實一個 while 交替、剩下接尾,就涵蓋所有情況。

PARALLEL · ADOPT ↓
● ADOPT · two cursors
l 走 word1、r 走 word2,交替取、剩下接尾

while (l < m && r < n):先取 word1[l] 再取 word2[r],兩指針各 +1。 迴圈結束時至多一方還有剩,把那一段整段接上。O(m+n) 時間,每個字元剛好放一次。乾淨、無需分「誰長誰短」。

3

核心策略 · 三步驟// STRATEGY

  1. 交替取 —— while (l < m && r < n):接上 word1[l] 再 word2[r],l++、r++。
  2. 接 word1 尾巴 —— 若 l < m,把 word1 剩下的整段接上。
  3. 接 word2 尾巴 —— 若 r < n,把 word2 剩下的整段接上。(兩個尾巴至多一個非空)

動畫演示// PARALLEL MERGE

"abc" + "pqrs":交替取 a,p / b,q / c,r → "apbqcr";word1 用完,把 word2 剩的 "s" 接尾 → "apbqcrs"。 結果的藍/綠交替顯示每個字元來自哪一串。

雙指針的兩大類// CONVERGING vs PARALLEL

對撞(往中間收) vs 平行(各走一序列)

對撞型:兩指針在同一個序列的兩端,往中間收。用在「兩端有對稱/配對關係」—— 344 反轉、125 回文、680 刪一字、已排序 Two Sum、裝水。

平行型(本題):兩指針各走一個序列,同向前進。用在「把兩個序列合起來/比對」—— 交替合併(本題)、合併兩個已排序陣列(LC 88)、合併排序的 merge 步驟、判斷子序列(LC 392)。 看到「一個序列 → 對撞」、「兩個序列 → 平行」,雙指針該用哪種就有方向了。

6

複雜度// COMPLEXITY

T = O(m + n)　　S = O(m + n)(輸出)

每個字元剛好被放進結果一次。空間是回傳字串本身 O(m+n)(不算的話,額外只有兩個索引)。 效能小提醒:res = res + a + b 這種寫法每次可能重新配置字串,大量拼接時偏慢;res.reserve(m+n) 預留、或用 push_back 逐字元追加更穩(見程式頁)。本題 m,n ≤ 100 無所謂。

面向成本說明
交替 + 接尾O(m+n)每字元放一次
輸出空間O(m+n)結果字串
額外空間O(1)兩個索引
拼接效率建議 reserve避免反覆重配置
7

一句洞察// INSIGHT

合併兩個序列 → 平行雙指針:各走一串、交替取,一方用完就把另一方剩下的整段接上。 這是雙指針的「兩個序列」面貌(對撞是「一個序列」),也是合併排序 merge 的雛形。O(m+n)。
