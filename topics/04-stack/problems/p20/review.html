範例 Trace// 三種失敗

// 三種不合法各觸發一個檢查 "(]" → '(' push [(]; ']' top='(' ≠ '[' → false (①類型錯) "]" → ']' 堆疊空 → false (②右括號多) "(" → '(' push [(]; 掃完 st 非空 → false (③左括號多) "()[]{}" → 每對即時配掉,掃完空 → true (合法) 三個檢查各擋一種錯,合法時堆疊剛好清空
三種失敗恰好對應三個檢查點:"(]" 靠比頂端抓類型錯、"]" 靠判空抓右括號多、"(" 靠最後判空抓左括號多。 合法的 "()[]{}" 則是「每個右括號一來就把對應左括號配掉」,堆疊在結尾自然清空。把「何時 false」想清楚,程式就寫對了。

App 1

App 3

只有一種括號:若字串只有 (),能不能不用堆疊、只用一個計數器判斷?為什麼多種括號就不行?

// ANSWERS
A1

合法括號是正確巢狀的:像 ([]),外層 ( 先開、內層 [ 後開,但內層 ] 要先閉、外層 ) 才閉。也就是「最後開的、最先閉」—— 後進先出(LIFO)。堆疊正是 LIFO:每遇左括號 push(記下「有一個等著被閉合」),每遇右括號就看堆疊頂端(最近、最該先閉的那個)是不是它的對應。這個「最近的先處理」語意,只有堆疊天然提供,所以括號匹配 = 堆疊的招牌題。

A2

① 類型/順序錯("(]"、"([)]"):右括號來時 st.top() != 對應左括號 → 立刻 false。② 右括號多("]"、"())"):右括號來時 st.empty()(沒有左括號可配)→ false。③ 左括號多("("、"([]"):掃完後 st 還有殘留 → 最後 return st.empty() 為 false。三個檢查各守一種,合起來覆蓋所有不合法情形,沒有遺漏。

A3

因為「數量相等」只保證不多不少,不保證順序/巢狀正確。反例 "([)]":(、)、[、] 各一個,左右數量都對,但它是交叉的(( 還沒閉就冒出 ]),不合法。要抓這種錯,必須知道「當前最該閉合的是哪個」—— 那需要記住開括號的順序,正是堆疊做的事。計數丟失了順序資訊,所以不夠。

A4

因為 C++ 的 || 是短路求值:st.empty() || st.top() != x 會先算 st.empty(),若為 true 就不會去算 st.top()。若寫反成 st.top() != x || st.empty(),當堆疊空時會先呼叫 st.top() —— 對空堆疊呼叫 top() 是未定義行為(UB),可能崩潰。所以「判空」必須放在前面,用短路保護後面的 top()。

A5

因為迴圈裡的檢查只在遇到右括號時觸發;如果字串只有左括號(如 "(("、"([{"),或右括號不夠把左括號配完("([]"),迴圈跑完不會回 false,但堆疊裡還留著沒閉合的左括號。return st.empty() 就是最後這道關卡:堆疊空才代表「所有左括號都被閉合了」。不檢查會把「左括號多」的輸入誤判成合法。

App 1

LC 150 逆波蘭(後綴):堆疊存運算元(數字)。遇數字 push;遇運算子就 pop 兩個運算元 b、a,算 a (op) b,把結果push 回去;掃完堆疊剩一個數就是答案。和括號匹配都是「用堆疊處理有結構的序列」:括號題堆疊存待閉合的括號、RPN 堆疊存待運算的數。這也呼應本章 682 的「+ 取最近兩筆」—— 堆疊擅長「跟最近幾筆互動」。

App 2

LC 1249:兩遍或一遍堆疊。用堆疊存左括號的索引;遇 ) 時若堆疊有左括號索引就配對 pop,沒有則這個 ) 是「多餘的」,記下它的索引要刪。掃完後,堆疊裡剩下的左括號索引也是「沒閉合、要刪」的。把所有標記要刪的索引集合起來,重建字串時跳過它們即可。核心仍是「堆疊配對、配不到的就是該刪的」,只是記的是索引而非字元。

App 3

只有一種括號時可以用計數器:一個變數 bal,遇 ( 加一、遇 ) 減一;過程中 bal 不可為負(負了代表右括號多),結尾 bal == 0(左右相等)即合法。因為只有一種括號,「頂端是不是對應類型」永遠成立,不需要記類型 —— 計數就等於堆疊高度。多種括號時,計數丟失了「頂端是哪一種」的資訊,無法分辨 "([)]" 這種類型交叉,所以必須用堆疊記住每個開括號的種類與順序。
