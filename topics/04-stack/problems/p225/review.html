重點回顧// RECAP

關鍵一句話
矛盾佇列 FIFO,堆疊要 LIFO
訣竅push 後把新元素旋轉到 front
front= 最後 push 的 = 堆疊 top
旋轉次數舊 size 次(先記 sz 再 push x)

觀念題// CONCEPT Q

Q1

為什麼用佇列(FIFO)實作堆疊(LIFO),某個操作必然是 O(n)?這個 O(n) 的本質是什麼?

Q2

push 裡為什麼要先記 sz = q.size() 再 q.push(x)?順序反過來會旋轉幾次、錯在哪?

Q3

旋轉「舊 size 次」把新元素轉到 front。為什麼恰好是 size 次(不是 size-1 或 size+1)?

Q4

本題把 O(n) 放在 push(push-heavy)。若改成 pop-heavy(push O(1)、pop O(n)),程式怎麼變?什麼情況該選哪種?

Q5

反過來:用兩個堆疊實作佇列(232) 怎麼做到攤還 O(1)?為什麼它比本題更漂亮(沒有 O(n) 的單次操作瓶頸)?

範例 Trace// push 交錯 pop

// push1, push2, pop, push3, top(front 在左) push(1): [1] top=1 push(2): [1,2]→旋轉1→[2,1] top=2 pop() : 取 front 2 → [1] 回傳 2,top=1 push(3): [1,3]→旋轉1→[3,1] top=3 top() : 3 堆疊語意正確:2 後進先出、3 最後進當 top
交錯 push/pop 也照常運作:pop 只是取 front(O(1)),不影響其餘元素的相對順序;下一個 push 一樣旋轉當前 size 個把新元素頂到前面。 不變式:任何時刻,佇列從 front 到 back 都是「插入時間由新到舊」 —— 這個性質被每次 push 的旋轉維持,所以 front 恆為 top。

應用題// APPLY

App 1

pop-heavy 版:寫出 push O(1)、pop/top O(n) 的單佇列版。pop 時怎麼取到「最後 push 的」?

App 2

攤還分析:本題 push 是 O(n)。若一連串操作大多是 push、少數 pop,總成本是多少?push-heavy 在這種情況划算嗎?

App 3

用 A 實作 B 的通則:「用佇列做堆疊」「用堆疊做佇列」「用陣列做堆疊/佇列」…… 這類題的共同思考框架是什麼?

// ANSWERS
A1

因為佇列只能前端出、尾端進,而堆疊要的是最後進的先出 —— 兩種存取順序相反。要讓佇列表現得像堆疊,必須在某處把「插入順序」翻轉一次,而翻轉 n 個元素本質上就是 O(n) 的搬移,無法用 O(1) 完成(佇列沒有「直接拿尾端」的操作)。這個 O(n) 是兩種結構順序落差的代價,躲不掉,只能選擇放在 push 還是 pop。

A2

因為要旋轉的是「x 加入之前的舊元素」。先記 sz = q.size()(舊元素數),再 q.push(x),然後旋轉 sz 次,把那 sz 個舊的繞到 x 後面。若先 push(x) 再取 size,size 會是 sz+1(含 x),旋轉 sz+1 次會把 x 也繞走一圈 → x 又回到後面,front 變成舊元素,語意全錯。所以順序不能顛倒。

A3

push x 後佇列是 [舊₁, 舊₂, …, 舊_sz, x](x 在尾)。要把 x 頂到最前,得把它前面那 sz 個舊元素逐一從 front 搬到 back。搬第一個(舊₁)後 [舊₂,…,x,舊₁]、…… 搬完第 sz 個後,所有舊元素都排到 x 後面,佇列變 [x, 舊₁, …, 舊_sz],x 到 front。所以恰好 sz 次 —— 每個舊元素被繞一次。sz-1 會少繞一個(x 不在最前)、sz+1 會把 x 也繞走。

A4

pop-heavy:push(x) 直接 q.push(x)(O(1),新元素在尾);pop()/top() 時,把前面 size-1 個從 front 搬到 back(旋轉),讓最後 push 的那個轉到 front,再取它。pop 是 O(n)。怎麼選:看操作分布 —— push 多、pop 少 → pop-heavy(讓便宜的操作頻繁);pop/top 多(尤其常 top 只讀)→ push-heavy(讓讀取 O(1))。本題選 push-heavy,top() 快。

A5

232 用兩個堆疊:in 收 push、out 供 pop/peek。push 就 in.push(O(1));pop/peek 時若 out 空,把 in 整個倒進 out(順序反轉一次),再從 out 頂取 —— 倒過去後 out 的頂端正是佇列 front。妙在:每個元素一生只被「倒」一次(從 in 到 out),所以雖然某次 pop 可能花 O(n) 倒,但攤還到每個操作是 O(1)。它比本題漂亮,因為「翻轉」的成本被攤平、沒有反覆重做(本題每次 push 都重新旋轉整個佇列,是真 O(n)/次)。兩堆疊的「懶惰搬移」是關鍵。

// REVIEW TAKEAWAY
FIFO↔LIFO 順序相反 → 某操作必 O(n)(翻轉)。push 先記 sz 再 push x、旋轉 sz 次→front=top。push-heavy vs pop-heavy 看操作分布。對偶 232 兩堆疊懶搬移 → 攤還 O(1),更漂亮。
